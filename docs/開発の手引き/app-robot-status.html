このIssueは何？

ゴールは、ロボの今の状態（待機中・移動中・到着・帰還中）をアプリの画面にリアルタイムで映すこと。利用者は本を呼び出したあと「ロボが今どこで何をしているか」を知りたい。これがあると安心感が段違いに上がる、地味だけど効く機能。

状態の出どころは robot_status テーブル。このテーブルは 常に1行だけ（id=1 固定）で、state 列に idle/moving/arrived/returning のどれかが入っている。アプリはこの1行を読むだけ。書き換えるのはロボ側の役目（このIssueでは書き換えない）。

設計の肝（疎結合 / アプリ⇄ロボはDB経由）

アプリとロボは直接つながらず、Supabaseのテーブルを介して間接的につながる。アプリは robot_status を購読して読むだけ、ロボ側が robot_status.state を更新する。だから今回作るのは「読む・映す」側だけでよい。

置き場所（最初の一歩）: components/RobotStatus.tsx という新規クライアントコンポーネントを作り、トップ画面 app/page.tsx（S-1・本一覧/ホーム）で import して表示する。トップに小さなステータス表示ウィジェットを置くイメージ。

状態を更新するのはロボ側Issue（別Issue・未実装でもOK）。このIssュー単体では Supabase の手入力で代用して検証してよい。最後の項目だけは「他Issue/ロボがそろってからの統合確認」なので、できなくても自分の実装が悪いわけではない。

進め方ステップ

  1. 置き場所を決めて空のコンポーネントを作る
    components/RobotStatus.tsx を新規作成し、ファイル先頭に use client を付けたクライアントコンポーネントとして作る（後で購読hookを使うので最初からクライアント側にする。理由は手順3とdocsで確認）。まずは固定文字だけ返して、app/page.tsx で import → 画面に出ることを確認する。
  2. 1行を読むだけのコードを足す
    lib/supabaseClient.ts の supabase を import し、robot_status から1行 select して、その DBの state 値を画面に出す。まだリアルタイムでなくてよい（開いた瞬間の状態が出ればOK）。1行取得の方法は research の『1行取得』を参照。
  3. なぜ use client が要るのかを docs で理解する
    useState / useEffect / 購読はブラウザ側でしか動かないので use client が必須。App Router の既定はサーバー側なので、これを付けないと state が使えない。理由は node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と .../03-api-reference/01-directives/use-client.md で確認する。
  4. DBの state 値を React の状態に入れる
    手順2で読んだ DBの state 文字列を、useState で作った React の状態（useState）に入れて表示するように直す。ここで「DBの列 state」と「Reactの状態(useState)」は別物だと意識する（用語ミニ辞典参照）。
  5. Realtime購読を足してリアルタイムにする
    useEffect の中で robot_status の変更を購読し、変更が来たら DBの新しい state 値を React の状態に入れ直す。購読の張り方は research の『Realtime購読』を参照。これで Supabase 側で値が変わると画面が自動更新される。
  6. 後片付け（unsubscribe）を入れる
    useEffect はクリーンアップ関数を返すことで、コンポーネントが消えるときに購読を解除できる。supabase.removeChannel(channel) 等で解除する。やらないと購読が二重・三重に増える。
  7. stateを日本語ラベルに変換して表示する
    idle→「待機中」のような対応表（オブジェクト等）を作り、DBの英語の state を日本語に変換して見せる。用語ミニ辞典の対応をそのまま使ってよい。
  8. Supabase側で robot_status の Realtime を有効化する
    対象テーブル robot_status の Realtime を有効にしないと、購読しても何も届かない。SupabaseのUIはバージョンで場所が変わるため、ダッシュボード上部の検索で「Realtime」「Publications」を検索するか、Table Editor で robot_status を開いて Enable Realtime のトグルを探す。手順は research の『Realtime有効化』で公式ガイドを確認。

📚 進めながら調べる・学ぶこと

  • robot_status の1行取得（select / single と maybeSingle）
    開いた瞬間の状態を出すには、まず現在の1行を読む必要がある。robot_status は id=1 固定で常に1行ある想定だが、single() は0行や複数行だと例外を投げる。schema未投入や空テーブルで試すと初期表示が落ちることがある。
    Supabase JS select: https://supabase.com/docs/reference/javascript/select 。single() と、行が無くても落ちにくい maybeSingle() の違いを確認する。
  • Supabase Realtime の購読（channel / postgres_changes）
    値が変わったら自動で画面を更新するため。supabase.channel(...) で購読を作り、postgres_changes で robot_status テーブルの変更を受け取る考え方を理解する。
    Realtimeガイド: https://supabase.com/docs/guides/realtime と Postgres Changes: https://supabase.com/docs/guides/realtime/postgres-changes 。channel・on('postgres_changes', ...)・subscribe() の流れを見る。
  • RLS有効テーブルで postgres_changes が届く条件（最大のハマりどころ）
    robot_status は schema.sql で RLSが有効。postgres_changes は RLSで select できる行・テーブルだけ配信される。今回は sel_stat ポリシー（全員 select OK）があるので publishable(anon) キーでも届く。ただしダッシュボードでテーブルをRealtime対象に追加し忘れると一切来ない。「有効化したのに来ない」で長時間詰まる典型。
    https://supabase.com/docs/guides/realtime/postgres-changes の Authorization（認可）節。『RLS有効テーブルのRealtime』の条件を確認する。schema.sql の sel_stat（robot_status の select=true）が届く根拠。
  • Supabaseダッシュボードでの Realtime 有効化の現行UI
    テーブルをRealtime配信対象に入れないと購読しても無音。Database > Replication という古いメニュー名は現行UIでは無い/改称されていることがあり、固定の導線名を覚えると迷子になる。
    https://supabase.com/docs/guides/realtime/postgres-changes で現在の手順を確認。現行は Table Editor のテーブル設定の Realtime トグル、または Database > Publications の supabase_realtime に robot_status を追加。UIが変わるので探し方ベース（上部検索で Realtime/Publications を検索）で当たる。
  • リアルタイム更新を React の状態(useState)に反映する考え方
    購読で受け取った新しいDBの state 値を React の状態(useState)に入れ直すと、画面が再描画されて表示が変わる。DBの値とReactの状態をつなぐ橋渡しの理解が核心。
    react.dev/reference/react/useState 。状態を更新すると再描画される基本を確認する。
  • useEffect と後片付け（クリーンアップ関数）
    購読は useEffect の中で1回だけ張り、コンポーネントが消えるときに解除する。解除しないと購読が増え続け、更新が多重に走る。
    react.dev/reference/react/useEffect の『クリーンアップ関数を返す』節。返り値の関数で removeChannel/unsubscribe を呼ぶ形を確認。
  • Client Component（use client）が必要な理由
    useState/useEffect/購読はブラウザでしか動かない。App Routerの既定はサーバー側なので、付け忘れると state が使えずエラー/手戻りになる。
    node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と .../03-api-reference/01-directives/use-client.md。

つまずきポイントと対処

症状原因・対処
購読しても画面が変わらない対象テーブルの Realtime が未有効が最多。robot_status をRealtime配信対象に追加する（Table Editor の Realtime トグル、または Database > Publications の supabase_realtime に追加）。UIは時期で変わるので上部検索で「Realtime/Publications」を探す。
Realtimeを有効にしたのに何も来ないRLS有効テーブルは select できる行だけ配信される。robot_status は sel_stat（全員select OK）があるので本来届く。届かないならテーブルがRealtime publication に入っているか、購読対象の table 名（robot_status）が正しいか確認。
初回ロードでエラーになる（落ちる）single() は0行・複数行で例外。空テーブルで試していないか確認。schema.sql の insert into robot_status ... (1,'idle') が実行済みか、または maybeSingle() を使う。
use client を付けずにエラーuseState/useEffectを使うファイルには先頭に use client が必須。手順3のdocsで理由を確認。
購読が二重・三重に走るuseEffect のクリーンアップ（返り値の関数）で removeChannel していない。後片付けを入れる。
DBのstateとReactのstateを取り違えるrobot_status.state（DBの列）と React の状態(useState)は別物。手順4で『DBの値をReactの状態に入れる』と主語を意識する。

動作確認のしかた

  1. npm run dev でアプリを起動し、トップ（S-1）に状態ウィジェットが出ることを確認する。
  2. 初回ロードで現在の state（初期は『待機中(idle)』のはず）が正しく日本語で出ること。ブラウザのConsoleに取得エラーが出ていないこと。
  3. Supabase ダッシュボードの Table Editor で robot_status の state を moving に手で書き換える → アプリ画面がリロードなしで『移動中』に変わればOK。
  4. 続けて arrived/returning/idle に変えて、それぞれ正しい日本語に切り替わることを確認する。
  5. 別画面に移動して戻る、を繰り返しても更新が多重にならない（後片付けが効いている）こと。
  6. 【任意】ロボ側が動く環境なら、本を呼び出して moving→arrived と自然に変わるのを見る。

困ったら・参照

  • DB列名の正は schema.sql（supabase/schema.sql）。robot_status: id(=1固定) / state / current_call_id / updated_at。select は sel_stat で全員可、update は upd_stat、insert ポリシー無し＝アプリから行追加は不可（常に1行のまま）。
  • 疎結合の設計（アプリ⇄ロボはDB経由）の詳細は基本設計のロボ連携の項を参照（資料では『疎結合(アプリ⇄ロボはDB経由)』として統一）。
  • 関連Issue: 「本を呼び出す（robot_calls に insert）」（状態が変わるきっかけを作る側）、ロボ側の「robot_status を更新する」Issue（state を書き換える側・未実装でも本Issューは単体で進む）。
  • 環境変数キーは NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY が正。.env.local は各自手作り。詳しくは同フォルダ 環境構築ガイド.html。
  • Next.js 16系は学習データと挙動が違う可能性。必ず node_modules/next/dist/docs/ の該当ガイドで現行の書き方を確認する。
