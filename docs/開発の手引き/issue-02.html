このIssueは何？

アプリが本やロボの情報を読み書きするには、その入れ物であるデータベースのテーブルが必要です。このIssueでは、設計図ファイル supabase/schema.sql を Supabase の SQL Editor（ブラウザ上でSQLを実行できる画面）に貼り付けて実行し、books / loans / users / robot_calls / robot_status / stop_points の6テーブルと、そのRLS（行レベルセキュリティ）を一気に作ります。

このIssueはもう完了しています

DB作成はPM（水谷）が実行済みです。あなたが今から実行し直す必要は基本ありません。この資料は『何が作られたのか』『自分でどう確認するか』『万一作り直す時はどうするか』を理解するための読み物です。チェックリストは“あなたが作る”ではなく“現状こうなっているはず”という確認用として読んでください。

なぜ大事か：ここで決めた列名・型・ルールが、これから作る全画面（本一覧・本登録・借りる/返す・ロボ呼出）の前提になります。スキーマを読めると、後のIssueで『どの列に何を入れるか』が自分で分かるようになります。

完了の定義（Done）

下のチェックは『正しく作られていればこうなっている』という実態です。実際の schema.sql を1行ずつ確認した内容で、テーブルごとにポリシーの有無が違う点に注意してください（ここを誤解すると後で必ず詰まります）。

  • Table Editor に6つのテーブルが並んでいる：books / loans / users / robot_calls / robot_status / stop_points
  • robot_status テーブルに1行だけデータがある（id=1, state='idle'）
  • 6テーブルすべてで RLS が有効（ON）になっている
  • 閲覧(select)ポリシーがある＝ books / loans / robot_calls / robot_status / stop_points の5つ（users には select すら無い）
  • 追加(insert)ポリシーがある＝ books / loans / robot_calls の3つだけ（users / stop_points / robot_status には無い）
  • 更新(update)ポリシーがある＝ books / loans / robot_calls / robot_status の4つ（users / stop_points には無い）
  • 削除(delete)ポリシーは『どのテーブルにも1つも無い』＝アプリからは誰も削除できない（全削除事故の防止＝仕様）
  • users はポリシーが1つも無く、RLSがONなので一般キー（publishable）からは読み書きとも実質できない（auth と紐づける設計のため。これも仕様）
『全テーブルに select/insert/update が揃っている』は誤解

RLSの世界では『ポリシーが無い操作は全部禁止』です。だから users（ポリシー0個）や stop_points（selectのみ）は、Table Editor で見ると“RLSはONなのにデータをいじれない”状態になります。これは壊れているのではなく設計どおり。後述の確認SQLで自分の目で数えられます。

進め方ステップ

  1. 設計図ファイルを開いて全体を眺める
    #4（リポジトリ雛形）でローカルに clone したリポジトリのフォルダ内にある supabase/schema.sql を VS Code などのエディタで開きます（GitHub 上のリポジトリでもブラウザで閲覧できます）。ファイルの並び順は ①stop_points ②users ③books ④loans ⑤robot_calls ⑥robot_status の6つの create table → 後半に RLS の設定、という構成です（①=books ではない点に注意）。
  2. create table の塊を1つずつ読む
    各 create table が『1つのテーブル＝表』の定義です。id の primary key（主キー）、references ... の外部キー、check (...) の値制限、text[] や timestamptz といった型を、用語ミニ辞典を見ながら指でなぞって読みます。答えを丸暗記するのではなく『この列は何のためか』を1つずつ言葉にしてみるのが学びになります。
  3. RLS（ファイル後半）の policy を数える
    alter table ... enable row level security が6つ、その下に create policy ... for select/insert/update が並びます。select/insert/update がテーブルごとに数が違うこと、delete が1つも無いことを、自分の目で数えて Done のチェックと突き合わせます。
  4. （参照）SQL Editor での実行方法を知る
    実行はPMが完了済みです。仕組みを知るために：Supabase ダッシュボード左メニューの『SQL Editor』を開き、新規クエリに schema.sql の中身を全部貼って『Run』を押す、という流れです。実行すると6テーブルとRLSが一気にできます。
  5. （参照）作り直したくなったら、まずPMに相談
    すでに作成済みのため、勝手に作り直すと全員の作業が止まります。作り直しは原則 PM（水谷）に任せてください。どうしても自分の練習用環境で作り直す場合は、外部キーの依存があるため素朴な drop table books; は順序エラーになります。drop table ... cascade; を使う必要がある、という入り口だけ覚えておきましょう（詳しくは『つまずきポイント』参照）。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心です。各テーマを『何を / なぜ / どこで調べるか』で示します。リンクが切れていたら添えてある検索キーワードで調べ直してください。

  • リレーショナルDBの基本（テーブル / 列 / 行 / 主キー / 外部キー）
    schema.sql は『表の設計図』。主キー（その行を一意に決める列）と外部キー（別テーブルの行を指す列。例: loans.book_id が books.id を指す）が分かると、なぜ本を消せない設計になっているか等が腑に落ちる。
    PostgreSQL 日本語ドキュメント https://www.postgresql.jp/document/ の『データ定義』章。リンク切れ時は《primary key foreign key とは PostgreSQL》で検索。
  • create table の読み方（型・check制約・default）
    shelf_level int check (shelf_level between 1 and 5) や status text check (status in (...)) のように、DB側で不正な値を弾くルールが書かれている。これを読めると後のIssueで『どんな値を入れてよいか』を自分で判断できる。
    PostgreSQL 日本語ドキュメント『CREATE TABLE』 https://www.postgresql.jp/document/ 内で CREATE TABLE を検索。リンク切れ時は《CREATE TABLE check 制約 PostgreSQL》で検索。
  • RLS（行レベルセキュリティ）と『ポリシーが無い操作は全部禁止』という核心
    Supabaseは安全のためテーブルごとに『誰がselect/insert/update/deleteしてよいか』をpolicyで明示する。ポリシーが書かれていない操作はすべて拒否される。これを知らないと『users が読めない』『delete できない』を“バグ”と誤解する。実際 users はポリシー0個なので、現状アプリ（publishableキー）からは読み書きできない（auth と紐づく設計のため）。
    Supabase RLSガイド https://supabase.com/docs/guides/database/postgres/row-level-security 。リンク切れ時は《Supabase row level security policy》で検索。
  • create policy の読み方（for select / insert / update、using と with check）
    schema.sql 後半の create policy sel_books on books for select using (true) 等を読めると、『どのテーブルにどの操作が許されているか』を自力で一覧化できる。delete の policy が1つも無い＝誰も削除できない、という意図が読み取れる。
    Supabase『CREATE POLICY / Policies』 https://supabase.com/docs/guides/database/postgres/row-level-security と PostgreSQL の CREATE POLICY ドキュメント。リンク切れ時は《CREATE POLICY using with check PostgreSQL》で検索。
  • Supabase SQL Editor の使い方と Table Editor / Policies 画面
    実行・確認の入口。SQL Editor は管理者権限で動くので、アプリ経由ではできない操作（insert ポリシーが無いテーブルへの追加など）もここからは実行できる、という違いを知っておくと混乱しない。
    Supabase ダッシュボード操作ガイド https://supabase.com/docs/guides/database/overview と画面内の『SQL Editor』『Table Editor』『Authentication > Policies』。リンク切れ時は《Supabase SQL Editor 使い方》で検索。
  • PostgreSQL の型 timestamptz と text[]（配列型）
    schema.sql に出てくる created_at timestamptz や tags text[] の意味を知らないと列の役割が読めない。timestamptz＝タイムゾーン付き日時、text[]＝文字列の配列（1列に複数のタグを持てる）。
    PostgreSQL 日本語ドキュメント『データ型』章 https://www.postgresql.jp/document/ 。リンク切れ時は《PostgreSQL timestamptz 配列 text[] とは》で検索。

つまずきポイントと対処

初心者が『壊れている？』と誤解しやすい、RLSの“非対称”ポイントを中心にまとめます。

  • already exists エラー：schema.sql を全部コピペして Run すると『table ... already exists』と出ることがある。これは“もう作成済み”という意味で、たいてい正常。新たに作り直す必要は無い。
  • users が読めない / 操作できない：users は RLS がON かつ policy が1つも無いので、アプリ（publishableキー）からは select も insert もできない。これは仕様（Supabase の auth と紐づける設計のため）。バグではない。
  • stop_points / robot_status に追加できない：stop_points は select のみ、robot_status は select と update のみ（insert ポリシーが無い）。アプリから直接 insert できないのは仕様。座標や状態の投入は #3（サンプル投入）や管理側／SQL Editor で行う。
  • RLSがONなのにデータがいじれて当然と思い込む：『ポリシーが無い操作は全部禁止』が原則。テーブルごとにできる操作が違うのが正しい状態。
  • robot_status の1行が消えた／無い：このテーブルは id=1 の1行固定が前提。万一行が無い場合は schema.sql の該当1行（insert into robot_status ...）を“自分で探して”SQL Editor で実行すると復活する。SQL Editor は管理者権限なので insert ポリシーが無くても実行できる（アプリ経由では insert ポリシーが無いため不可、という違いに注意）。
作り直し（再実行）は安易にやらない

テーブルには外部キーの依存（books←loans/robot_calls、stop_points←robot_calls、robot_calls←robot_status）があるため、素朴な drop table books; は『他から参照されている』と順序エラーになります。作り直しは原則 PM（水谷）に依頼してください。どうしても自分の練習環境で消すときだけ、SQL Editor で drop table ... cascade;（依存ごと消す）を使う、という方針です。リポジトリに reset 用 SQL は同梱されていないため、再実行は慎重に。

動作確認のしかた

このIssueはアプリ画面ではなく『DBの状態』を確認します。Supabase ダッシュボードと、SQL Editor で打つ確認用SQLの2通りで見られます。

  1. Table Editor を開き、左に books / loans / users / robot_calls / robot_status / stop_points の6つが並ぶことを目で確認。
  2. robot_status テーブルを開き、行が1つ（id=1, state='idle'）あることを確認。
  3. Authentication > Policies を開き、どのテーブルにどのポリシーがあるかを見る（または下の確認SQLで一覧）。

確認用SQL（SQL Editor に貼って Run。これらは“読み取り中心”なので答えのコードではなく確認手段です）：

-- ① テーブルが6つできているか
select tablename from pg_tables where schemaname = 'public' order by tablename;

-- ② 各テーブルで RLS が ON か（relrowsecurity が t ならON）
select relname, relrowsecurity
from pg_class
where relname in ('books','loans','users','robot_calls','robot_status','stop_points');

-- ③ どのテーブルに select/insert/update/delete のどの policy があるか一覧
select schemaname, tablename, policyname, cmd
from pg_policies
where schemaname = 'public'
order by tablename, cmd;

-- ④ robot_status に1行あるか
select * from robot_status;
③のSQLが一番のおすすめ

③を実行すると『books には select/insert/update があり、users には何も無い、delete はどこにも無い』が一目で分かります。Done のチェックリストと突き合わせれば、自分の理解が正しいか確かめられます。

用語ミニ辞典

言葉かんたんな意味
テーブルデータを入れる『表』。列（項目）と行（1件分のデータ）でできている
主キー (primary key)その行を1つに特定する列。例: books.id。重複・空が許されない
外部キー (foreign key)別テーブルの行を指す列。例: loans.book_id は books.id を指す。だから参照中の本は消せない
RLS (行レベルセキュリティ)テーブルごとに『誰がどの操作をしてよいか』を制御する仕組み。ポリシーが無い操作は全部禁止される
policy (ポリシー)RLSの許可ルール。for select/insert/update/delete 単位で作る。無い操作は禁止＝delete は誰も作っていないので削除不可
check 制約列に入れてよい値の制限。例: shelf_level は 1〜5、status は available か on_loan のみ
timestamptzタイムゾーン付きの日時型。created_at などで使う
text[]文字列の配列型。tags のように複数の値を1列に持てる
cascadedrop table ... cascade のように、依存している物ごと一緒に処理する指定
