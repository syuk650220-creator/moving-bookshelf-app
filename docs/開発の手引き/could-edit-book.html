このIssueは何？

ゴール: すでに登録されている本（books テーブルの1行）を、フォームで直して保存（update）できるようにする。例: タイトルの誤字を直す、棚の段（shelf_level）を変える、著者を入れ直す。要件 C-03（本情報の編集）。

なぜ必要か: 登録（insert）で入れた本の情報は、後から間違いに気づいたり、棚を移したりで直したくなる。books には削除ポリシーが無い（誰も消せない仕様）ので、間違えても消してやり直すことはできない。だから『直す（update）』手段があると運用が楽になる。優先度は★☆☆＝余力でOKだが、Supabase の update と『誤って全件更新しない』という大事な型を学べるIssue。

編集の起点と置き場所（実ファイル名）

編集ボタンの起点になるのは S-1 本一覧（app/page.tsx ＝ ルート /）か S-2 本詳細（app/books/[id]/page.tsx ＝ 既にスタブが存在）。編集フォームの置き場所は2案: (A) S-2 本詳細ページ（app/books/[id]/page.tsx）の中に編集UIを足す、(B) 新規に編集専用ページ app/books/[id]/edit/page.tsx を作る。どちらでもOK。app/books/[id]/page.tsx という自然な置き場所が既にあることを起点に考えると迷いにくい。

実ファイルパス注意

本一覧は app/page.tsx（ルート /）です。app/books/page.tsx は存在しません（app/books/ の下は [id]/ だけ）。本詳細は app/books/[id]/page.tsx です。

完了の定義（Done）

  • 本一覧（app/page.tsx）か本詳細（app/books/[id]/page.tsx）から、編集したい本の編集画面に入れる
  • 編集画面を開くと、その本の現在の値がフォームに最初から入っている（空ではない）
  • title / author / isbn / shelf_level を直して保存できる
  • 保存すると books の該当行だけが update され、画面の表示も新しい値に変わる
  • 編集していない別の本は一切変わっていない（最重要: eq で1件に絞れている）
  • shelf_level は 1〜5 の範囲だけ受け付ける（範囲外は保存させない）など、最低限の入力チェックがある
  • tags の扱いを決めてある（このフォームでは触らない＝更新対象から外す、を既定とする。下記注意参照）
  • status はこのフォームでは触らない（貸出・返却の処理側で自動的に変わる値のため）
tags と status は原則スコープ外

books には tags text[]（not null default '{}'）と status（available/on_loan）があります。tags の配列入力UIは難度が高く、status は貸出・返却処理側で変わる値なので、このIssueでは原則 title/author/isbn/shelf_level の4項目だけを編集対象にします。C-03 の確定スコープ（tags を含めるか）は PM に確認してください（下記『困ったら・参照』）。

進め方ステップ

起点を先に決める

編集ボタンの起点は S-1 本一覧（app/page.tsx）か S-2 本詳細（app/books/[id]/page.tsx）。フォームは S-2 詳細ページ内 or /books/[id]/edit に置く案があります。まず『どの本を』編集するかの1件をURL（[id]）で受け取る形にすると、対象が1件に固定できて安全です。

  1. テスト用の本を最低2冊登録する
    Supabase の SQL Editor で books に2冊入れる（下の『困ったら・参照』のシードSQL参照）。1冊だけだと『片方だけ変わるか（eq の効果）』を確認できないので必ず2冊以上。
  2. 編集の起点（ボタン/リンク）を決める
    本一覧（app/page.tsx）の各行か、本詳細（app/books/[id]/page.tsx）に『編集』リンクを置く。フォームを詳細ページ内に置くか、/books/[id]/edit を新設するかをここで決める。
  3. どの本を編集するか1件を特定する（動的ルート）
    URL の [id] から本のidを受け取る。Next.js 16系では params の受け取り方が以前と違う（下の研究項目で必ず確認）。古い『params をそのまま分割代入』はエラーになるので注意。
  4. 対象の本を1件だけ取得する（select + eq）
    supabase（lib/supabaseClient.ts の supabase）を import し、books から id が一致する1件を select。.eq("id", 受け取ったid) で1件に絞る。
  5. フォームに今の値を初期表示する
    取得は非同期なので初回レンダーでは値がまだ無い。useState の初期値は初回しか効かないため、取得が終わってから値をフォームに入れる仕組みが要る（研究項目で確認）。『データは取れているのに欄が空』はこれが原因のことが多い。
  6. 入力チェックを足す
    shelf_level は 1〜5 の整数だけ。title が空なら保存させない、など最低限のバリデーション。範囲外や空のまま update しない。
  7. 保存（update + eq）する
    supabase.from('books').update({ 直した値 }).eq('id', 対象id) の形。eq を必ず付ける（付け忘れ＝全件更新の事故）。書き方は研究項目で確認。
  8. 更新後の表示を反映する
    保存に成功したら画面の表示を新しい値にする（再取得する／詳細ページに戻る等）。反映の考え方は研究項目で確認。
  9. エラー時の表示を足す
    update の戻り値の error を見て、失敗したら画面にメッセージを出す（保存できたのか分からない状態を避ける）。

📚 進めながら調べる・学ぶこと

このNext.jsは16系

学習データの書き方と違う可能性があります。コードを書く前に、必ず node_modules/next/dist/docs/ の該当ガイドを開いて現行の書き方を確認してください。AIには答えを書かせず、調べ方を聞くこと。

  • 編集対象の1件を特定する（動的ルートと params の受け取り）
    16系では params が Promise になり、await（サーバー）または use（'use client'）で取り出す必要がある。古い『params をそのまま分割代入』はエラーになる。ここが訓練データと最も差が出る具体ポイントで、知らないと確実にハマる。
    node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/dynamic-routes.md の冒頭サンプルと、特に『In Client Components』の節（use を使う例）を必ず見る。const { id } = await params（サーバー）／const { id } = use(params)（'use client'）の“ここが変わった点”だけ確認する（答えのコードは書かない）。
  • どこで select / update を呼ぶか（サーバーかクライアントか）
    取得・更新をどちら側でやるかでコードの形が変わり、最初に決めないと手が止まる。このプロジェクトの supabase はブラウザ向けクライアント（NEXT_PUBLIC キー）なので、クライアントから直接呼ぶのが素直。
    node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と 03-api-reference/01-directives/use-client.md。方針: このプロジェクトは lib/supabaseClient.ts のクライアントをブラウザから直接呼ぶ構成なので、まずはクライアント側（'use client'）で select/update する素直な形でよい。データ更新ガイド 07-mutating-data.md は Server Action 中心なので、そこは“反映の考え方”を参考にする読み方をする。
  • Supabase の update で1件だけ更新する（eq で絞る）
    Done の最重要。update に eq を付け忘れると全件が同じ値に書き換わる重大事故になる。eq の置き場所と書き方を正確に知る必要がある。
    https://supabase.com/docs/reference/javascript/update （update 専用ページ）。.update({...}).eq('id', 対象id) の形と、戻り値の data / error の見方を確認。update は必ず eq とセットで使う。
  • フォームの初期値に既存値を入れる（非同期取得の反映タイミング）
    取得は非同期なので初回レンダーでは値が無い。useState の初期値は初回しか効かないため、後から来た値が初期値に反映されず欄が空のままになる典型ハマりがある。
    node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md と react.dev『You Might Not Need an Effect / Synchronizing with Effects』。取得完了後に値を入れる仕組み（useEffect で set する等）が要る点を確認（答えは書かず観点だけ）。
  • 更新後の表示反映（再取得 / 画面遷移）
    保存できても画面が古い値のままだと『直したのに変わらない』に見える。クライアント取得なら state 更新や再取得、戻る遷移などで反映する。
    node_modules/next/dist/docs/01-app/01-getting-started/07-mutating-data.md（Server Action 中心だが“更新後どう反映するか”の考え方を読む）と 04-linking-and-navigating.md（保存後に詳細へ戻す導線）。クライアント直更新の場合は『保存成功→再 select する or 詳細に戻る』が素直。
  • 入力バリデーションと誤更新の防止
    shelf_level は 1〜5 の制約があり、空タイトルや範囲外を送るとDBエラーや不正データになる。保存前に弾くと事故が減る。
    react.dev『制御コンポーネント（Controlled inputs）』と、DBスキーマ supabase/schema.sql の books 定義（shelf_level int 1..5、title 等）。送る前に値が範囲内・空でないかを確認する考え方を学ぶ。

つまずきポイントと対処

症状原因と対処
保存したら全部の本が同じ内容になった（重大事故）update に .eq('id', 対象id) を付け忘れて全件更新した。update は必ず eq とセット。テストは2冊以上で行い、片方だけ変わるか確認する。
params でエラー / id が取れない16系では params が Promise。古い直接分割代入はエラー。await params（サーバー）か use(params)（'use client'）で取り出す。dynamic-routes.md の該当節を確認。
フォームに既存値が出てこない（欄が空）取得が非同期で初回レンダーでは値が無い。useState の初期値は初回しか効かないため、取得完了後に set する仕組み（useEffect 等）が要る。
tags を空にしたら not null エラーtags は text[] not null default '{}' なので null 不可。空にするなら null でなく空配列 {} を送る必要がある。このフォームでは tags を触らない方針なら、そもそも update の対象から外す（送らない）。
status を勝手な文字列にして保存できないstatus は available / on_loan のDB制約があり範囲外は弾かれる。そもそも status は貸出・返却の処理側で自動的に変わる値なので、本情報編集フォームでは触らない。
shelf_level でエラー1〜5 の整数制約。範囲外や数値以外を送ると弾かれる。保存前にバリデーションで弾く。
保存したのに画面が古いままupdate 成功後に表示を更新していない。再 select する／詳細ページに戻る等で反映する。
eq 付け忘れ＝全件更新

update における最大のリスク。eq が無い update はテーブル全行を書き換えます。books は削除ポリシーが無く消して直すこともできないので、被害が大きい。必ず eq で1件に絞り、テストは2冊以上で『片方だけ変わるか』を確認すること。

動作確認のしかた

  • テスト用の本を最低2冊登録しておく（1冊だと eq の効果を体感できない）。
  • 編集画面を開くと、その本の現在の値がフォームに最初から入っている → 空なら『非同期取得の反映タイミング』を確認。
  • title などを直して保存 → 画面の表示が新しい値に変わる → 変わらないなら更新後の反映処理を確認。
  • Supabase の Table Editor で books を開き、編集した1冊が新しい値になっている。
  • 同じ Table Editor で、編集していないもう1冊が無傷（元のまま）であることを必ず確認 → ここが eq が効いている証拠。変わっていたら eq 付け忘れ。
  • shelf_level に 0 や 6、空タイトルを入れて保存しようとすると弾かれる（バリデーション動作）。
  • 保存に失敗させた時（例: ネット切断）にエラーメッセージが出る。

用語ミニ辞典

言葉かんたんな意味
update既存の行の値を書き換える操作。Supabase では .update({...}).eq(...) の形。必ず eq とセットで使う。
eq（イコール）『この列がこの値の行だけ』に絞る条件。.eq('id', 対象id) で1件に固定する。これが無いと全件が対象になる。
動的ルート [id]URL の一部を変数として受け取る仕組み。app/books/[id]/ の [id] がその本のidになる。
params（16系）URL から渡る値の入れ物。16系では Promise なので await params か use(params) で取り出す。
初期値 / 制御コンポーネントフォーム欄の値を state で管理する作り。編集では既存値を初期値として入れる。useState の初期値は初回しか効かない点に注意。
shelf_level棚の段。1〜5 の整数だけ許可（DB制約）。
tags（text[]）本のタグ。配列型で not null・既定は空配列 {}。null は入れられない。今回は触らない方針。
status本の状態（available / on_loan）。貸出・返却の処理側で自動的に変わる値。編集フォームでは触らない。
