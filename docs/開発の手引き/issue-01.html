このIssueは何？

アプリの入口になるホーム画面(S-1)を作ります。Supabaseの books テーブルを読み込んで、本を一覧表示する画面です。タイトル・著者・段（shelf_level）・状態（status）が見え、status を色分けし、/register（本登録）と /history（履歴）への導線を置きます。

なぜ最初におすすめ？

「DBから読んで画面に出す」という、このアプリで何度も使う一番の基本を一通り体験できるからです。ここで掴んだ流れ（import → select → map → 表示）は他の画面でもそのまま使えます。

現状 app/page.tsx は中身が仮の文章だけのスタブ（仮置きファイル）です。これを本物の一覧画面に作り替えます。編集するのは基本 app/page.tsx の1枚。DB接続の窓口 lib/supabaseClient.ts は import するだけで、中身は触りません。

このNext.jsは最新版(16系)

あなたの知っている書き方と違う可能性があります。答えのコードを覚えるのではなく、必ず node_modules/next/dist/docs/ の該当ガイドを開いて「今の正しい書き方」を確認しながら進めてください。下の『調べる・学ぶこと』が道案内です。

完了の定義（Done）

  • books の一覧が表示され、各行に タイトル / 著者 / 段(shelf_level) / 状態(status) が見える
  • status を色分けしている（available=在庫あり、on_loan=貸出中 が見た目で区別できる）
  • 画面上部に /register と /history へのリンクがある（next/link の Link を使う）
  • 読み込み中の表示が出る（loading）
  • 本が0件のとき「本がありません」等のメッセージが出る（empty）
  • 取得に失敗したときのエラー表示が出る ※どう発火させるか（自分で throw するか、state方式で出し分けるか）は下の研究セクションとdocsで確認
  • PRを出して Closes #1 を本文に書く

進め方ステップ

  1. ステップ0：初回だけの環境準備（ここが生命線）
    まだこのプロジェクトを動かしたことがなければ先に：(1)リポジトリを clone、(2)npm install で依存をインストール、(3).env.local を自分で手作りする。.env.local の正しいキー名・値は同フォルダの「環境構築ガイド.html」の通りに書く（正しいキー名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY。古い資料の …ANON_KEY は誤り）。これが終わっていないと npm run dev すら起動しません。
  2. ステップ1：ブランチを切る
    main へ直接書かない。feature/book-list のような自分のブランチを作ってから作業する。コマンドは下の『動作確認のしかた』のcodeブロックにまとめてあります。
  3. ステップ2：開発サーバーを起動して現状を見る
    npm run dev を実行し、ブラウザで http://localhost:3000 を開く。今は仮の文章だけが出るはず。ここを作り替えていく。
  4. ステップ3：サーバー/クライアントどちらで作るか決める
    一覧を「表示するだけ」ならサーバーコンポーネントがシンプルで素直。理由と違いは下の研究『サーバーとクライアントの違い』で確認する。迷ったらまずサーバー方式で。
  5. ステップ4：supabase を import して books を select する
    import { supabase } from '@/lib/supabaseClient' のように読み込み、books を全件 select する。⚠重要：現状の app/page.tsx は export default function Home() で async ではない。サーバー側で await して読むなら、この関数を export default async function Home() に変える必要がある（ここに気づかず詰まりやすい）。なお lib/supabaseClient.ts の supabase は 公開鍵(PUBLISHABLE_KEY)で作られた読み取り用クライアントで、サーバー/ブラウザどちらから import して使ってもよく、どちらも RLS で保護されているので安全。
  6. ステップ5：取れたデータを map で1行ずつ表示する
    select の結果（配列）を .map() で回し、各本のタイトル/著者/段/状態を1行として描く。React で配列を出すときの key の付け方も研究セクションで確認。
  7. ステップ6：status を色分けする
    status が available か on_loan かで色を変える。Tailwind のクラス（例: 文字色や背景色のユーティリティ）を条件で切り替える考え方を研究セクションで確認。
  8. ステップ7：loading / 0件 / error を出す
    読み込み中・0件・失敗の3状態を用意する。サーバー方式なら『読み込み中＝loading.tsx』『エラー＝error.tsx』というファイル規約が使える。⚠️ error.tsx は例外的に先頭に 'use client' が必須（公式docsに『Error boundaries must be Client Components』と明記）。また Supabase の select はエラーを throw せず { data, error } の error で返すことが多いので、error.tsx を発火させたいなら自分で throw する、またはサーバーコンポーネント内で error を見て自前のエラー表示を出す（state方式/分岐）かを選ぶ。どちらにするかは研究セクションで確認。
  9. ステップ8：上部に /register と /history のリンクを置く
    画面上部に2つの導線を置く。next/link の Link コンポーネントを使う（<a> タグだとページ全体が再読込される）。共通ナビは layout.tsx にまだ無いので、このIssueでは page.tsx 内に置けばOK。
  10. ステップ9：動作確認して PR を出す
    下の『動作確認のしかた』で各状態を確認 → コミット → push → GitHubで PR。本文に『何をしたか / Closes #1 / 動作確認』を書く。マージは PM(水谷) 承認後に Squash and merge。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心です。各ステップで詰まりやすい概念を「何を / なぜ / どこで調べるか」でまとめました。答えのコードはあえて載せません。下のdocsを自分で開いて、今のNext.js16の書き方を確認してください。

  • サーバーコンポーネントとクライアントコンポーネントの違い（use client）
    どちら側でDBを読むか・どこに 'use client' を書くかでファイルの作り方が変わる。一覧表示だけならサーバー側がシンプルだが、なぜそうなのかを理解しておくと後で迷わない。
    node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-client.md
  • サーバーコンポーネントの『最小の骨組み』：関数を async にして await でDBを読む
    現状の page.tsx は function Home()（async でない）。サーバーで await して読むには async function Home() に変える必要がある。この“形”を最初に掴まないとステップ4で必ず詰まる。
    node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md のサーバーコンポーネント冒頭（async関数にして await でデータを取る形）
  • データ取得をサーバー側でやる書き方（考え方）
    docs で『async関数の中で await してDBを読む』という骨組みを掴むため。
    node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md の『With an ORM or database』の節。※注意：この節の例は別のDBライブラリ(ORM, Drizzle風 db.select().from(...))なので構文は Supabase と違う。docs では“async関数にして await でDBを読む”という形だけを掴み、実際の select の文法は次のSupabase公式で確認すること。
  • Supabase の select の基本（実際の構文）
    実際に books を読むときの文法（.from('books').select(...)）と、戻り値が { data, error } であることを知るため。
    https://supabase.com/docs/reference/javascript/select 。⚠重要：error は throw されず戻り値の error フィールドで返ることが多い。error.tsx に出したいなら自分で throw する、state方式なら自分で出し分ける——どちらにするかをここで判断する。列名は title / author / shelf_level / status / id（DBスキーマ通り）。
  • 配列を map で表示する／React の key
    select で返るのは配列。各行を描くには map で回す。React は各要素に一意な key（本なら id が最適）を求める。
    Reactの基本（公式 https://react.dev/learn/rendering-lists ）。本プロジェクトでは id を key にするのが素直。
  • Tailwind で status を色分けする
    available / on_loan を見た目で区別する。条件によって付けるクラスを変える考え方が要る。
    Tailwind 公式 https://tailwindcss.com/docs （文字色・背景色のユーティリティ）。条件分岐でクラス文字列を切り替えるのは普通のJS/TSの三項演算子等でOK。
  • loading.tsx でのローディング表示
    サーバー方式では、そのフォルダに loading.tsx を置くだけで読み込み中に自動表示される。手で state を持たなくてよい。
    node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/loading.md と node_modules/next/dist/docs/01-app/01-getting-started/10-error-handling.md
  • error.tsx でのエラー表示（use client 必須）
    サーバー方式でエラー境界を作るには error.tsx を置く。だが先頭に 'use client' が必須で、これを書かずに詰まる人が最多。データ取得自体はサーバーのままでよく、エラー表示の部品だけがクライアントになる、という分担を理解する。
    node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/error.md（『Error boundaries must be Client Components』の記述）と node_modules/next/dist/docs/01-app/01-getting-started/10-error-handling.md（L212付近）
  • リンクと画面遷移（next/link の Link）
    上部の /register・/history への導線に使う。<a> タグだとページ全体が再読込される。Link を使うとクライアントサイド遷移＝全体を再読込せず速い。
    node_modules/next/dist/docs/01-app/01-getting-started/04-linking-and-navigating.md（Link を next/link から import する節と、なぜ a タグでなく Link かの説明）
  • books テーブルに本当にデータがあるかを自分で確認する方法
    一覧が空のとき『バグなのか、データが無いだけなのか』を自分で切り分けるため。PM待ちにせず自走できる。
    Supabase ダッシュボード → Table Editor → books テーブルを開いて行があるか見る。前提の #3（サンプル投入）が済んでいるか確認。

つまずきポイントと対処

error.tsx を作ったのにエラーになる／効かない

error.tsx は先頭に 'use client' が無いと動きません（公式docsに『Error boundaries must be Client Components』と明記）。さらに error.tsx は throw された例外しか捕まえません。Supabase の select は失敗を error フィールドで返すだけで throw しないことが多いので、その error を自分で throw するか、サーバー内で error を見て自前表示するかを選んでください。

await したらエラーになる

現状の page.tsx は async ではありません。サーバーで await してDBを読むなら export default async function Home() に変える必要があります。ここを忘れると await が使えずエラーになります。

画面が真っ白／一覧が空

(1) ターミナルに env 関連のエラーが出ていないか（.env.local の作成漏れ・キー名違い。正は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY）。(2) データが0件なだけかも：Supabaseダッシュボード → Table Editor → books テーブルに行があるか自分で見る。行が無ければ #3（サンプル投入）の前提が未完了。

  • 列名は title / author / shelf_level / status / id。スペルや大文字小文字を間違えると取れない（DBスキーマ通りに）。
  • status は available か on_loan の2値。それ以外を想定した色分けは不要。
  • DELETE は誰もできない仕様（RLSにDELETEポリシー無し＝全削除事故防止）。この画面では読み取りだけなので関係ないが覚えておくと安心。
  • リンクは <a> でなく next/link の Link。<a> だと毎回ページ全体が再読込される。
