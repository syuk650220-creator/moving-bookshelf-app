▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

本詳細画面(S-2) を作ります。URL は /books/[id]（[id] はその本の uuid）。やることは4つ：(1) URL の id で books を1冊だけ取得して表示、(2)『借りる』を押すと loans に1行 insert ＋ books.status を on_loan に update、(3)『返す』を押すと未返却の loans に returned_at/returned_by を入れて books.status を available に戻す、(4) この本の貸出履歴(loans) も一覧表示。

このIssueの山場

1回の操作で 2つのテーブルを続けて更新する のが初めての回です。『片方だけ成功して片方が失敗したらどうなる？』を意識するのがこのIssueの一番の学びどころ。

編集するのは基本 app/books/[id]/page.tsx の1枚。現状は仮の文章だけのスタブ（仮置きファイル）です。DB接続の窓口 lib/supabaseClient.ts は import するだけで中身は触りません。前提は #1（本一覧）— 一覧から本をタップしてこの詳細に来る流れです。

このNext.jsは最新版(16系)

あなたの知っている書き方と違う可能性があります。特に 動的ルートの [id] の受け取り方（params が Promise になった）は昔と変わっています。答えのコードを覚えるのではなく、必ず node_modules/next/dist/docs/ の該当ガイドを開いて『今の正しい書き方』を確認しながら進めてください。下の『調べる・学ぶこと』が道案内です。

完了の定義（Done）

  • URL の id（/books/[id] の [id]）で books を1件 select して、タイトル/著者/段/状態を表示できる
  • 『借りる』で loans に book_id・borrower_type(今回は "guest")・guest_name・borrowed_at を入れて1行 insert できる（必須列が埋まっていること）
  • 『借りる』成功後に books.status を on_loan に update でき、画面の状態表示も最新になる
  • 『返す』で未返却の loan（returned_at が null）に returned_at と returned_by を入れて update できる
  • 『返す』成功後に books.status を available に戻し、画面の状態表示も最新になる
  • この本の貸出履歴(loans) が新しい順などで一覧表示される
  • 更新が失敗したとき、画面にエラーメッセージが出る（onClick 内なので try/catch + useState で出す。error.tsx では出ない）
  • PRを出して Closes #7 を本文に書く
Done の確認は Table Editor でも

『insert できた』は画面の見た目だけでなく、Supabase ダッシュボード → Table Editor → loans を開いて、book_id・borrower_type・borrowed_at が実際に埋まった行ができているかを目で確認しましょう。必須列が空のまま『できた』と思い込むのを防げます。

進め方ステップ

おすすめの進め方（最初の分かれ道を消す）

この Issue は『借りる/返す』ボタン（onClick）と状態の出し分けが主役なので、ページを use client にして id は use(params) で受け、データ取得も useEffect + select で統一すると一本道です。サーバー方式(await params)も可能ですが、ボタン操作が多いこの画面ではクライアント方式の方が迷いません。

  1. ステップ0：環境準備（未済なら）
    clone → npm install → .env.local を「環境構築ガイド.html」の通りに自分で作る（正しいキー名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY。古い資料の …ANON_KEY は誤り）。前提 #1（本一覧）が動く状態だと、一覧→詳細の流れで確認しやすい。
  2. ステップ1：ブランチを切る
    main へ直接書かない。feature/borrow-return のような自分のブランチを作る。コマンドは下の『動作確認のしかた』に。
  3. ステップ2：このページを use client にして id を受け取る
    app/books/[id]/page.tsx の先頭に 'use client' を書き、use(params) で id を取り出す。⚠16系では params が Promise なので、昔のように params.id と直接読むと undefined になる。受け取り方は下の研究『動的ルート [id] と params』とdocsで必ず確認。
  4. ステップ3：取得処理を『関数』にまとめる（あとで再利用する）
    id で books を1件 select する処理と、その本の loans を一覧 select する処理を、呼び直せる関数として書く。これを useEffect で初回に1回呼ぶ。⚠重要：あとのステップ6で『更新成功後にこの関数を呼び直す』ので、最初から関数にしておくと後がラク。結果は useState に入れて画面に出す。
  5. ステップ4：本1件と貸出履歴を表示する
    取れた本のタイトル/著者/段/状態を表示。loans は新しい順（borrowed_at 降順など）で一覧に並べる。0件なら『履歴なし』等を出す。
  6. ステップ5：『借りる』ボタンの処理を書く
    onClick の中で順に：(1) loans に insert（book_id・borrower_type="guest"・guest_name（仮名でOK）・borrowed_at）→ (2) その error を if で確認 → (3) 成功していれば books を status='on_loan' に update（id で eq して対象を絞る）。⚠ #6(ゲスト名) が未実装の今は borrower_type="guest" + guest_name に仮の名前を決め打ちで入れて動かす（理由は下の研究参照）。
  7. ステップ6：成功したら表示を最新化する
    借りる/返すが成功したら、ステップ3で作った取得関数を呼び直す。これで status と履歴が最新になる。『再描画』とは具体的にこれ＝前ステップの取得関数の再呼び出し、と覚える。
  8. ステップ7：『返す』ボタンの処理を書く
    onClick の中で：(1) この本の未返却 loan（book_id 一致 ＆ returned_at が null）を .is('returned_at', null) で絞る → (2) 通常は1件のはずだが、複数あり得るので まず select で対象を1件特定 → その loan の id で update の2段が安全（または borrowed_at 降順で1件に絞る）→ (3) returned_at(今の時刻) と returned_by を update → (4) 成功したら books を status='available' に update（id で eq）→ (5) ステップ6の最新化。
  9. ステップ8：失敗時のエラー表示を入れる
    ⚠ onClick 内のエラーは error.js / エラー境界では捕まりません（公式docsに明記）。各 supabase 呼び出しの error を if で見て、try/catch で囲み、useState に持ったエラーメッセージを画面に出す。これがクライアント方式での正しい失敗表示。
  10. ステップ9：動作確認して PR を出す
    下の『動作確認のしかた』で借りる→返すを通しで確認 → コミット → push → PR。本文に『何をしたか / Closes #7 / 動作確認』。マージは PM(水谷) 承認後 Squash and merge。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心です。各ステップで詰まりやすい概念を「何を / なぜ / どこで調べるか」でまとめました。答えのコードはあえて載せません。下のdocs/URLを自分で開いて、今のNext.js16・Supabaseの書き方を確認してください。

  • 動的ルート [id] と params の受け取り方（16系では params が Promise）
    URL の id を取らないと『どの本か』が決まらず、最初の一歩で止まる。16系では params が Promise になり、昔の params.id 直読みは undefined になる。この Issue はボタン操作中心なので use client + use(params) で受けるのが一本道。
    node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/dynamic-routes.md の『In Client Components』節（'use client' ＋ React の use(params) で受け取る例）と冒頭の『params は Promise。await か use で開く』の記述。クライアントなので await ではなく use(params) を使う。
  • Supabase の select で1件だけ取る／履歴を並べて取る
    id で books を1件、book_id で loans を複数（履歴）取る。1件取得・並び替え（order）・null 絞り込みの文法を知る必要がある。
    https://supabase.com/docs/reference/javascript/select 。1件なら id で eq、履歴は borrowed_at の降順で order。戻り値は { data, error }（error は throw されず戻り値に入る点に注意）。
  • Supabase の insert に何を入れるか（loans の必須列を schema に即して）
    loans は borrower_type が NOT NULL かつ CHECK(in 'user','guest')、book_id も NOT NULL。これを入れずに insert すると『null value in column』や『violates check constraint』という原因の分かりにくいエラーで詰まる。誰が借りるかで guest_name か user_id のどちらを使うかも変わる。
    supabase/schema.sql の loans 定義（41〜47行）で必須列を確認。insert 文法は https://supabase.com/docs/reference/javascript/insert 。⚠#6(ゲスト名)が未実装の今は、borrower_type="guest" ＋ guest_name に仮の名前を決め打ちで入れて動かす（user_id はログイン機能後の話なので今は使わない）。
  • Supabase の update で対象を eq/is で絞る（borrow=on_loan, return=available）
    books の status を変えるとき、必ずその本だけに絞らないと全件書き換えてしまう。返却では『未返却の loan』を null 判定で絞る必要がある。
    https://supabase.com/docs/reference/javascript/update 。books は .eq('id', 本のid)。未返却 loan は .eq('returned_at', null) では効かず .is('returned_at', null) を使う（null は eq でなく is）。複数該当時は『まず select で1件の id を特定 → その id で update』の2段、または borrowed_at 降順で1件に絞る。
  • 2つのテーブルを続けて更新するときの考え方と、片方失敗時の注意
    借りる/返すは loans と books を続けて変える。1つ目が成功して2つ目が失敗すると、データがちぐはぐ（loans は入ったのに status が変わっていない等）になる。学生プロジェクトでは完全な取り消し(トランザクション)までは求めず、『各 error を必ず確認 → 失敗を検知してユーザーに知らせる』で十分。
    考え方の参照として node_modules/next/dist/docs/01-app/01-getting-started/07-mutating-data.md。⚠ただしこの docs は Server Functions / Server Actions（'use server'）+ revalidatePath / router.refresh の世界で、このプロジェクトの『クライアントから Supabase JS を直接叩く方式』とは噛み合いません。docs からは『更新後にデータを最新化するという発想(Refresh data)』だけ拾い、Server Function 部分はそのまま使わないこと。最新化は次の項目（取得関数の呼び直し）で行う。
  • 更新後に画面を最新化する（クライアント方式の具体）
    insert/update しても、画面の useState は自動では変わらない。『再描画』が何を指すのか分からず止まりやすい。
    具体策：ステップ3で作った取得関数（books 1件＋loans 一覧を取り直す関数）を、借りる/返す成功後に呼び直す。これで status と履歴が最新になる。docs(07-mutating-data.md)の router.refresh() の“更新後に取り直す”という考え方が近いが、サーバー関数/revalidatePath ではなくクライアントで select を呼び直して useState を更新するのがこの構成の基本。
  • onClick 内のエラーをどう画面に出すか（error.js では捕まらない）
    借りる/返すは onClick（イベントハンドラ）の中で動く。Error boundary / error.js はイベントハンドラ内のエラーを捕まえないので、error.js で失敗を見せようとしても動かず混乱する。
    node_modules/next/dist/docs/01-app/01-getting-started/10-error-handling.md の『Error boundaries don't catch errors inside event handlers』の節（手動で catch して useState/useReducer に入れUIに出す、と明記）。つまり try/catch ＋ useState でメッセージを表示するのが正解。
  • 未返却 loan の絞り込み（returned_at is null）と複数該当への対処
    返却対象は『まだ返していない loan』＝ returned_at が null の行。同じ本を何度も借りた履歴があると、null の行が複数になり得る。order/limit が無いと該当全行を書き換える危険がある。
    https://supabase.com/docs/reference/javascript/using-filters の .is()（null 判定は .is('returned_at', null)、.eq では効かない）。複数対策は『select で borrowed_at 降順 → 先頭1件の id を取り → その id で update』。通常は貸出中=1件の前提だが、安全のため id 特定の2段がおすすめ。
  • books に本当にその id の本があるか自分で確認する
    詳細が表示されないとき『バグなのか、id が存在しないだけか』を自分で切り分けるため。
    Supabase ダッシュボード → Table Editor → books で対象の id を探す。URL の id とコピー一致しているか確認。前提 #1（本一覧）の行から正しい id でリンクされているかも見る。

つまずきポイントと対処

update で eq/is を付け忘れると全件書き換え

books の status を update するとき .eq('id', 本のid) を付け忘れると全部の本の status が書き換わります。しかもこのプロジェクトは DELETE ポリシーが無く、消して作り直せません（全削除事故防止の仕様）。だから update する前に必ず select で『対象が1件か』を確かめる癖をつけましょう。返却の未返却 loan も同様に必ず絞ってから update。

insert で『null value in column "borrower_type"』『violates check constraint』が出る

loans の borrower_type は NOT NULL ＆ CHECK(in 'user','guest') です。これを入れずに insert すると『null value in column "borrower_type"』、'user'/'guest' 以外を入れると『violates check constraint』が出ます。今回は #6 前なので borrower_type="guest" ＋ guest_name に仮名 を入れてください。book_id も NOT NULL なので必須。

params をそのまま読むと undefined になる

16系では params が Promise です。params.id と直接読むと undefined。use client にして use(params) で開いてから id を取り出してください（dynamic-routes.md の In Client Components 節）。

error.js を作ったのに失敗が表示されない

借りる/返すは onClick（イベントハンドラ）の中。error.js / エラー境界はイベントハンドラ内のエラーを捕まえません（10-error-handling.md に明記）。各 supabase 呼び出しの error を if で見て、try/catch ＋ useState でメッセージを画面に出してください。

  • null の絞り込みは .is('returned_at', null)。.eq('returned_at', null) は効かない（定番の詰まり所）。
  • 未返却 loan が複数になることもある（同じ本を何度も借りた履歴）。order(borrowed_at 降順)＋先頭1件、または select で id を特定してから update。
  • 列名は loans: book_id / borrower_type / user_id / guest_name / borrowed_at / returned_at / returned_by、books: id / status(available|on_loan)。スペル・大文字小文字を schema 通りに。
  • loans と books の2回更新は続けて走る。1回目の error を確認せず2回目に進むと、片方だけ反映されてちぐはぐになる。各 error を必ず if で確認。

eq / is絞り込み条件。eq は等しい、is は null 判定用（returned_at が null は .is を使う）。
