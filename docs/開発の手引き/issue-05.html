このIssueは何？

本棚に並べる本を アプリから1冊ずつ登録する画面 (S-3) を作ります。やることは「ISBN(本の裏のバーコードの数字)を入力 → openBD という無料の書誌APIでタイトル・著者を自動取得 → 段(1〜5)を選ぶ → books テーブルに1件 insert → ホームへ戻る」の流れです。

今回のうれしさ

外部API(openBD)を初めて触る回です。「ネット上のサービスにfetchで問い合わせて、返ってきたJSONから欲しい値を取り出す」という、これから何度も使う基本パターンを身につけます。前提Issueは #2(テーブル作成), #3(サンプル投入), #4(リポジトリ雛形) です。

ファイルはもう存在します(新規作成ではない)

編集する app/register/page.tsx は #4(リポジトリ雛形)で すでに仮画面(プレースホルダ) が置かれています。中身は「本登録 / S-3」という見出しと『担当者がこの画面を実装します』という説明文だけです。同じ名前のファイルを新しく作るのではなく、この既存ファイルの中身を自分の実装に置き換えます。

完了の定義（Done）

  • ISBNをテキスト入力できる(まずは手入力でOK。バーコードカメラは将来拡張)
  • 「取得」ボタンを押すと openBD からタイトル・著者が自動でフォームに入る
  • openBDで見つからない時は、タイトル・著者を手で入力できる(フォールバック)
  • 段(shelf_level)を1〜5から選べる
  • 「登録」を押すと books に1件増える(status は available)
  • 登録に成功したらホーム(/)へ自動で戻る

進め方ステップ

  1. ブランチを切る
    main に直接書かないこと。git switch -c feature/register-book のように feature ブランチを作ってから作業します(コマンドは下のcodeブロック参照)。1 PR = 1 Issue が目安です。
  2. 既存スタブを置き換える準備をする
    app/register/page.tsx には すでに仮画面が入っています。新規作成ではなく、(1)ファイル先頭(use client ディレクティブ と import 群)と (2)return の中身 を、自分の実装に置き換えていきます。このページはユーザーが入力したりボタンを押したりする画面なので『クライアントコンポーネント』にする必要があります。先頭に書く合図(ディレクティブ)が何かは docs で確認(研究ブロック参照)。
  3. 入力フォームと state を用意する
    ISBN・タイトル・著者・段(shelf_level) の4つを React の state(useState)で持ちます。段は 1〜5 の select(プルダウン)で選ばせると楽です。ただし select の値は『文字列』で返ってくるので、insert する時に数値へ変換が必要になります(詳しくは『つまずきポイント』)。入力欄を value と onChange で state につなぐ『制御コンポーネント』の考え方を docs/React で確認。
  4. 「取得」ボタンで openBD を呼ぶ(ここが核心・小さく分ける)
    (a) ボタンの onClick を async 関数にする。(b) fetch(\https://api.openbd.jp/v1/get?isbn=${isbn}\) を await する。(c) さらに res.json() を await してJSONに変換する(なぜ await が2回要るか=本文の読み取りも非同期だから)。(d) 返ってきた配列の中身から title / author を取り出して setTitle / setAuthor で state に入れる。取得中・失敗時の表示は別の小さな state(例: loading, message)で管理すると親切です。
  5. 見つからない時のフォールバックを作る
    openBDは登録の無い本だと [null] を返します(=配列の中身が null)。その場合は『見つかりませんでした。手で入力してください』と出すだけにして、タイトル・著者欄はそのまま手入力できるようにします。fetch自体が失敗(オフライン等)した場合と、見つからない([null])場合は分けて扱えると更に良いです。
  6. 「登録」ボタンで books に insert する
    supabase は import { supabase } from '@/lib/supabaseClient' のように読み込みます(雛形で用意済み)。supabase.from('books').insert({...}) で1件追加します。列名は schema.sql の通り: isbn, title, author, shelf_level(数値に変換), status は 'available' を入れる。insert の戻り値の error をチェックして、あればメッセージを出す。
  7. 成功したらホームへ遷移する
    画面遷移には useRouter を使います。本体の先頭で一度 const router = useRouter() と呼んでおき、insert 成功後に router.push('/') でホームへ戻します。import 元が今回の最大の罠です(研究ブロック参照)。
  8. 動作確認 → コミット → PR
    下の『動作確認のしかた』で確認できたら、git add → git commit → git push して GitHub で Pull Request を作成。PR本文は『何をしたか / Closes #5 / 動作確認』。マージは PM(水谷)承認後に Squash and merge。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心です。答えのコードは載せません。下の「何を・なぜ・どこで」を見て、自分でdocsを開いて現行の書き方を確かめながら書いてください。

  • use client ディレクティブ(このページをクライアント化する)
    入力やボタン操作・useState・useRouter を使うページは『クライアントコンポーネント』にする必要がある。ファイルの一番上に書く1行が何かを確認する。
    node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-client.md と node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md
  • フォーム入力とReactのstate(useState)
    ISBN・タイトル・著者・段 をユーザー入力に合わせて保持し、画面に反映するため。useState は import { useState } from 'react' で読み込む(import元を間違えやすい)。入力欄を value と onChange で state につなぐ『制御コンポーネント』の形を覚える。
    React公式 https://react.dev/reference/react/useState と https://react.dev/learn/reacting-to-input-with-state
  • fetchで外部API(openBD)を呼ぶ基本と注意
    openBDは鍵不要の公開APIなので、今回はブラウザ(クライアント)から直接 fetch してOK。async/await の使い方、fetch の結果に対して res.json() をもう一度 await する理由(本文の読み取りも非同期処理だから)を理解する。
    MDN https://developer.mozilla.org/ja/docs/Web/API/Fetch_API/Using_Fetch ／ 参考: node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md。なお 05-server-and-client-components.md の箇条書きにある『Use API keys, tokens, and other secrets without exposing them to the client.(鍵やトークンをクライアントに晒さない)』という記述は、鍵が要るAPIはサーバーで呼ぶという《逆ケース》の話。今回は公開APIなので当てはまらない対比として読むこと。
  • openBDのレスポンス構造(配列 → summary → title/author)の取り出し方
    返り値は『配列』で、要素が見つからない時は null。多くの本は summary.title / summary.author の両方で取れる(例: 9784101010137 は summary.author に値が入っている)。ごく一部 summary.author が空の本だけ、その時に onix(Contributor の PersonName)を見る、という例外扱い。断定せず、必ず自分の目で実物を見てキー名を確認する。なお同じ著者でも summary 側はスペース無し・onix 側はスペース有りで表記が違うことがあるので、取得元は summary.author に統一しておくと動作確認の期待値とそろう。
    openBD APIガイド https://openbd.jp/api/ ／ まずブラウザで https://api.openbd.jp/v1/get?isbn=9784101010137 を開き、title と著者がどのキー(summary.title / summary.author 等)に入っているか目視確認する。見つからない例は https://api.openbd.jp/v1/get?isbn=9784101010136 (これは [null] が返る)。
  • Supabaseのinsert(books に1件追加)
    取得・入力した値を DB に保存するため。列名は schema.sql に厳密一致させる(isbn, title, author, shelf_level, status)。status には 'available' を入れる。insert の戻り値の error を必ず確認する。
    Supabase JS docs https://supabase.com/docs/reference/javascript/insert ／ 列名・制約は supabase/schema.sql(books: shelf_level は 1..5 の int, status は available|on_loan)
  • 登録後の画面遷移(useRouter)
    insert 成功後にホーム(/)へ戻すため。最大の罠は import 元。import { useRouter } from 'next/navigation' が正しい(古い学習データの next/router は旧版で別物。16系では navigation を使うことを docs で確認)。使い方は本体の先頭で一度 const router = useRouter() と呼んでから router.push('/') を使う、の2段階。
    node_modules/next/dist/docs/01-app/03-api-reference/04-functions/use-router.md と node_modules/next/dist/docs/01-app/01-getting-started/04-linking-and-navigating.md

つまずきポイントと対処

useRouter の import 元(最頻出の罠)

import { useRouter } from 'next/router' と書くと動きません。16系(App Router)では import { useRouter } from 'next/navigation' が正しい。古いネット記事・学習データは next/router を案内していることが多いので注意。

.env.local を直したらサーバー再起動

supabase が undefined・環境変数が読めない時、原因は2つ。(1)キー名が古い(正は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY。古い anon は誤り)。(2)もっと頻発するのが『.env.local を作った/直した後にサーバーを再起動していない』。Next.js は env を起動時にしか読まないので、npm run dev を一度 Ctrl+C で止めて起動し直す。

title/author が undefined になる: openBDのレスポンスは配列で、見つからない時は中身が null。配列の0番目を取り出してから .summary.title… とたどる必要がある。著者も多くは summary.author で取れる(例: 9784101010137 は summary.author に値あり)が、ごく一部 summary.author が空の本だけ onix 側を見ることになるので、必ず実レスポンス( https://api.openbd.jp/v1/get?isbn=9784101010137 )を見てキー名を確認してから取り出すこと。

shelf_level でエラー / 段が変な値で保存される: HTMLの select の value は『文字列』で返ってくる(例: "3")。books.shelf_level は 1..5 の int(数値)なので、insert の前に Number(...) で数値へ変換する。変換し忘れると型エラーや 1..5 制約違反になりがち。

「取得」を押しても何も起きない: onClick の関数を async にし忘れ/await を付け忘れていると、fetch の途中の値(Promise)を使ってしまう。fetch を await、さらに res.json() を await、の2回 await を確認する。

見つからない例で『コードが壊れた』と勘違い: テストで 9784101010136 を入れると openBD は [null](見つからない)を返します。これはバグではなく仕様。ヒットする例は 9784101010137(『こころ』夏目漱石/新潮文庫)。

動作確認のしかた

  1. npm run dev を実行して開発サーバーを起動する(.env.local を直した直後なら必ず再起動)。
  2. ブラウザで http://localhost:3000/register を開き、ISBN欄に 9784101010137 を入れて「取得」を押す → タイトル『こころ』と著者『夏目,漱石,1867-1916』(これは summary.author の値=スペース無し)が自動で入ることを確認(著者はカンマ区切りで生没年まで入るのは正常。なお onix 側から取るとスペース有りの『夏目, 漱石, 1867-1916』になり表記が違うので、取得元は summary.author にそろえる)。
  3. 次に 9784101010136 を入れて「取得」→『見つかりません』と出て、タイトル・著者を手入力できる(フォールバック)ことを確認。
  4. 段(shelf_level)を1〜5から選び、「登録」を押す → エラーが出ず、ホーム(/)へ戻ることを確認。
  5. Supabase の管理画面 → Table Editor → books テーブルを開き、いま登録した行が1件増えていることを確認する(ここが本番の確認ポイント)。
ホーム(S-1)では今はまだ見えない

登録できたかの確認は『Supabase の Table Editor で books に行が増えたか』を主軸にしてください。ホーム(本一覧 S-1 / app/page.tsx)も現状はスタブ(仮画面)で、本を一覧表示しません(別Issue)。画面に出ないからといって『登録できていない』と早とちりしないこと。

用語ミニ辞典

言葉かんたんな意味
ISBN本に付いた世界共通の番号。裏表紙のバーコード下段(978…で始まる13桁)の数字。
openBDISBNから書名・著者などを無料で返してくれる公開API(鍵不要)。
fetchブラウザからネット上のAPIに問い合わせる命令。結果はあとから返るので await で待つ。
JSONAPIがデータをやりとりする時の文字の形式。res.json()でJavaScriptの値に変換する。
state(useState)画面に表示する『今の値』をReactで保持する仕組み。入力値の管理に使う。
クライアントコンポーネントブラウザ側で動く部品。入力やボタン操作・useStateを使うページに必要。先頭に合図のディレクティブを書く。
shelf_level本を置く『段』。books列で1〜5の整数。selectの値は文字列なので数値変換が要る。
フォールバック本命(openBD自動取得)が使えない時の代替手段。ここでは手入力のこと。
