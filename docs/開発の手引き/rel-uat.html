▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴール: 開発者以外のゼミ生に、配布した本番(Vercel)のアプリを実際に使ってもらい、主要操作が問題なく回ることを確認する。出てきた使いにくさやバグを記録し、優先度をつけて直す。

なぜ必要? 作った本人は「動く前提」「正しい順番」で操作してしまうので、不具合や分かりにくさに気づきにくい。受入テスト(UAT)＝ 作る側でなく使う側の目線で『これは要件（やりたかったこと）を満たしているか』を確かめる最後の関門。ここで拾った声がそのまま MVP（まず動く最小限の完成品）の仕上げになる。

受入テストとは

プログラムの細かい部品が正しいかを見るのではなく、『ゼミ生が普段やりたい操作（本を探す・借りる・呼ぶ）が最後までスムーズに通るか』を、実際の利用シーンで確かめるテストのこと。

完了の定義（Done）

  • テスター（ゼミ生）3人以上に、本番URLで主要操作を実際に試してもらえた
  • 主要操作（探す→借りる/返す→ロボ呼び出し→ログイン）が実利用で最後まで回ることを確認できた
  • 出た不具合・使いにくさが GitHub Issue（バグ報告テンプレ）に記録された
  • 記録した不具合に『必ず直す / 余力で / 見送り』の優先度が付いた
  • 『必ず直す』に分類した不具合の修正PRがマージされ、本番で直っていることを再確認できた

進め方ステップ

  1. 0. テスターを集めて段取りする
    ゼミ生3人に声をかけ、日時を確保する（1人15〜20分が目安）。当日全員が入れないと困るので、事前に1人だけ Google アカウントでログインできるか疎通確認しておく（Supabase Auth の許可設定漏れがあると当日全員ログインできない事故になる）。
  2. 1. テスト対象と前提を確定する
    対象は『開発者が本番(Vercel)にデプロイ済みのアプリ』。テスターはローカル構築不要で、配布する Vercel の URL をブラウザで開くだけ。clone も .env.local も要らない。あわせて『ロボ呼び出し』を試すなら、stop_points（席データ）が入っているかPM/担当に確認する（空だと呼び出しが必ずエラーになる。これは仕様であってテスト失敗ではない）。
  3. 2. テストシナリオを書く
    『何をする → どうなれば成功か』を箇条書きにする。例: 『本を探す→検索で出る』『借りる→借りた状態になる』『返す→戻る』『ロボを呼ぶ→呼び出しが受け付けられる』『Googleログイン→トップに入れる』。テスターはこの紙の順になぞるだけにしておく。
  4. 3. テスターに触ってもらう（観察する）
    横で見て、迷った場所・指が止まった場所をメモする。ここでは口を出さず手伝わない（手伝うと『分かりにくさ』が隠れてしまう）。終わったら『一番使いにくかったところ』を一言もらう。
  5. 4. 出た声をその場でメモにまとめる
    『どの画面で・何をしたら・どうなった・どうなってほしかった』をセットで書く。スクショやスマホ写真も撮っておく（後でバグ票に貼ると伝わりやすい）。
  6. 5. バグ票（GitHub Issue）に起票する
    リポジトリの GitHub で New Issue → バグ報告 を選ぶと、症状 / 再現手順 / 期待する動き の枠が出る。1つの不具合＝1 Issue。再現手順は他人がなぞれる粒度で書く。
  7. 6. 優先度をつけて取捨選択する
    全部を直すのは3ヶ月では無理。『必ず直す（使えない・データが壊れる）/ 余力で（不便だが回避できる）/ 見送り（今回はやらない）』の3段階に仕分け、PMと合意する。
  8. 7. 直す → 再テストする
    『必ず直す』を feature ブランチ→PRで直す（1 PR=1 Issue、Closes #番号 を書く）。マージ後、Vercel の自動デプロイ完了（数分）を待ってから本番URLで再確認する。デプロイ状況は Vercel のダッシュボードで見られる。直ったら Issue を Close する。

📚 進めながら調べる・学ぶこと

  • 受入テスト(UAT)とは／MVPが要件を満たすかの確認
    『細かいバグ探し』ではなく『やりたかったこと（要件 F-◯/C-◯）が実利用で通るか』を見るのが目的。目的を取り違えると、重箱の隅つつきで時間を使ってしまう。
    Web検索『受入テスト UAT とは 初心者』。あわせて同フォルダの基本設計／要件一覧（S-◯ 画面・F-◯/C-◯ 要件）を見て、何が『満たすべきこと』かを確認する。
  • テスト項目／シナリオの作り方
    テスターが迷わず同じ手順をなぞれるように『操作→期待結果』の形でシナリオを用意する必要がある。口頭だけだと人によって試す内容がバラバラになる。
    Web検索『テストシナリオ 書き方 例』『テストケース 操作 期待結果』。本Issueの『動作確認のしかた』のOK目安テーブルをそのままシナリオ表の雛形に使ってよい。
  • フィードバックの集め方（観察＋ヒアリング）
    『使いにくい』は本人が言葉にしづらい。横で観察して『止まった場所』を拾う方が正確。手伝いすぎると問題が隠れる、という勘所も学ぶ。
    Web検索『ユーザビリティテスト 観察 進め方』『think aloud（思考発話）テスト』。
  • バグ票（Issue）への落とし込み方
    『なんか変』では直せない。再現手順・期待する動き・実際の動き・環境(ブラウザ/PC)が揃って初めて直せるバグ票になる。
    リポジトリの .github/ISSUE_TEMPLATE/bug.md を使う。 GitHub で New Issue → バグ報告 を選ぶと『症状 / 再現手順 / 期待する動き』の枠が自動で出る。現状このテンプレには『実際の動き』『スクショ』『環境（ブラウザ/PC）』の欄が無いので、症状欄にそれらも書き足すと直す人が助かる。
  • 優先度付けと取捨選択（MoSCoW と本Issueの3段階の対応）
    限られた期間で『何を直し何を見送るか』を全員が同じ基準で判断するため。用語が混ざると混乱するので、本Issueでは下の対応表の言葉に統一する。
    Web検索『MoSCoW 優先度 とは』。本Issueの『用語ミニ辞典』の対応表（必ず直す＝Must など）に合わせる。
  • GitHub のラベルでの整理
    起票した不具合を後から優先度で見分けられるようにするため。ただしメンバーが各自で乱立させると逆に混乱する。
    Web検索『GitHub ラベル 付け方』。運用は『ラベルは PM(水谷)が最初に priority:high / priority:low / wontfix の3つだけ作る。メンバーは付けるだけ』に固定する（作成権限・乱立対策）。
  • 本番(Vercel)への反映の仕組みと反映待ち
    main マージで自動デプロイされるが反映に数分の時間差がある。『マージ直後に本番URLを見たがまだ直っていない』で混乱しやすい。
    CLAUDE.md/AGENTS.md のデプロイ記述、Vercel ドキュメント https://vercel.com/docs/frameworks/nextjs 。デプロイ状況は Vercel ダッシュボードで確認。

つまずきポイントと対処

ロボ実機が無いと robot_status は変わらない（正常）

アプリとロボは Supabase のテーブル越しに疎結合でつながっている。アプリは robot_calls に insert するだけで、robot_status を更新するのはロボ側（rclpy/supabase-py）。現状 robot/ は README のみで実機もシミュレータも無いので、アプリ単体テストでは robot_status は変わらないのが正常。テスターには『呼び出しが受け付けられればOK。ロボが動かないのは想定どおり』と先に伝える。

ロボ呼び出しは席データが無いと必ずエラーになる

robot_calls.seat_id は stop_points への外部キーで必須。schema.sql に席の seed が無く scripts にも投入処理が無いため、stop_points が空だと呼び出し insert が外部キー違反で失敗する。これはバグではなく仕様。テスト前にPM/担当へ『席データは入っているか』を確認する。空ならロボ呼び出しシナリオは飛ばす。

本番DBは削除できない＝テストデータは“片付ける”しかない

RLSで DELETE ポリシーが無く、誰も行を消せない（全削除事故の防止）。一方 update は全テーブル全員OK。なので消す代わりに、テストで作った robot_calls は status を canceled に、借りた本は返却操作で available に戻して論理的に片付ける（行自体は残る）。ゲスト名は先頭に [TEST] を付けて後で見分けられるようにする。本のダミーも [TEST] を付ける。

ログインの名前表示は users テーブル由来とは限らない

users テーブルは RLS 有効だが select/insert/update ポリシーが無く、アプリのキー(anon)からは読み書きできない。名前表示は auth セッション情報(メール等)由来の可能性が高い。受入テストの合格基準は『ログインが完了してトップ画面に入れる』程度にとどめ、『DBの名前が出るか』までは判定軸にしない（実装が何由来で名前を出しているかは開発者に確認）。

貸出は user か guest か先に決める

loans への記録で borrower_type を 'user' にすると user_id（users 参照）が要るが、users がアプリから扱えない可能性がある。テストでは guest（borrower_type='guest' / guest_name）で借りる運用が安全。どちらで試すか事前に決めておく。

手伝いすぎない

テスターが迷ったとき、つい『そこはこう押すよ』と教えたくなるが、それをやると『分かりにくさ』が消えてしまう。迷った事実こそが一番の収穫。グッとこらえて観察する。

動作確認のしかた

各シナリオで『何を見ればOKか』。アプリ単体テスト時の判定が基本。robot_status の変化は実機/シミュレータが動いている時だけ確認できる。DBは Supabase の Table Editor で行が増えたか見られる。

やることOKの目安（アプリ単体テスト時）
本を探す検索/一覧で目的の本が表示され、画面が崩れずエラーも出ない
借りる（guest推奨）借りた本が on_loan 表示になり、loans に行が1件増える（Table Editor で確認）
返す本が available 表示に戻り、loans の returned_at が入る
席を選んでロボ呼び出し席(stop_point)を選んでから呼ぶと、robot_calls に status=queued の行が1件増える（Table Editor で確認）
（実機/シミュレータ稼働時のみ）ロボ状態robot_status.state が idle→moving 等に変わる。実機が無い間は変わらないのが正常
Googleログインログインが完了し、トップ画面に入れる（セッションが確立する）

最終OK判定: テスター3人以上が上記の主要操作を最後まで通せて、出た不具合がすべて Issue 化され、『必ず直す』分が本番で直っていること（=Done のチェックが全部埋まる）。

🤖 AIコーチの使い方

AIには答え（完成コードや完成シナリオ）を書かせず、考え方とチェックの相棒として使う。下をコピペして使う。

あなたはコーチです。答えそのものを書かず、私が自分で考えられるよう質問と観点で導いてください。

状況: 学生プロジェクトの『移動本棚ロボ アプリ』で受入テスト(UAT)をします。テスターはゼミ生3人、本番(Vercel)のURLをブラウザで開くだけで触ります。アプリとロボはSupabaseのテーブル(robot_calls/robot_status)経由の疎結合で、ロボ実機はまだありません。本番DBは削除不可・更新は可です。

お願い:
1) 主要操作（探す/借りる/返す/ロボ呼び出し/ログイン）の受入テストシナリオを『操作→期待結果』形式で自分で書きたい。漏れがちな観点を質問で気づかせてください（答えは書かない）。
2) テスト中に出た声を GitHub の bug.md テンプレ（症状/再現手順/期待する動き）に落とすとき、私の下書きに足りない情報を指摘してください。
3) 集まった不具合を『必ず直す/余力で/見送り』に仕分ける判断軸を、私に問いかける形で整理させてください。
まず、私のシナリオ案のどこを詰めるべきか質問から始めてください。

用語ミニ辞典

言葉かんたんな意味
受入テスト(UAT)使う側の目線で『やりたいことが実利用で通るか』を確かめる最後のテスト
MVPまず動く最小限の完成品。今回ゼミ生に触ってもらう対象
テストシナリオ『この操作をすると、こうなれば成功』を並べた手順表
バグ票(Issue)不具合を直せる形で記録した GitHub の票。症状/再現手順/期待する動き を書く
疎結合アプリとロボが直接つながらず、DBの表を介して間接的に連携する設計
論理的に片付ける行は消せないので、status を canceled/返却済みに update して無効化すること
MoSCoWMust(必ず)/Should(できれば)/Could(余力で)/Won't(やらない) の優先度分類

優先度の言葉はこの対応で統一する（用語の混乱防止）:

困ったら・参照

  • バグ票テンプレ: リポジトリの .github/ISSUE_TEMPLATE/bug.md（症状/再現手順/期待する動き）。GitHub の New Issue → バグ報告 から使う

  • 環境構築ガイド.html: （テスターは不要。開発者が本番にデプロイ・ローカル確認する際の .env.local 設定用）
