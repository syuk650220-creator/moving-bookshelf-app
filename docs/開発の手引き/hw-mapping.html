実機マッピング（SLAMで地図作成→席座標採取）

SLAMで部屋の地図を作って保存し、席3つの座標(x,y,theta)を実機から採取して stop_points テーブルに入れる。これが後でロボが席まで自走するための「地図」と「目的地リスト」になる。

このIssueは何？

ゴールは2つ。(1) SLAM で部屋の地図を作って保存する、(2) 各席の座標 (x, y, theta) を採って stop_points テーブルに入れる。これができると、後のIssueでロボが Nav2 を使って「3番の席まで行って」と指示されたとき、地図上の座標をめがけて自走できるようになる。

SLAM(スラム)＝Simultaneous Localization And Mapping。ロボがLiDAR(レーザーで周囲の距離を測るセンサ)で周りを見ながら走り回り、地図を作るのと自分が地図のどこにいるかを同時に推定する技術。今回は ROS2 の slam_toolbox を使う。

このIssueの立ち位置（準備系）

これは「ロボが自走する(Nav2)」の前準備。地図と席座標が無いとNav2は目的地を持てない。逆に言うと、本Issue完了時点ではまだNav2で実際に走らせる必要はない（座標の最終的な正しさはNav2導入後に確定する）。Sprint2の走行・Nav2系Issueと並走する位置づけ。

⚠️要確認(PM)：ROS2ディストロ／OSの組み合わせ

本資料は ROS2 Humble 前提で source /opt/ros/humble/setup.bash や apt 導入を書いているが、Humble の公式対応OSは Ubuntu 22.04(Jammy)。土台ガイド hw-pi5-setup は Ubuntu Server 24.04 指定で、24.04 に正式対応するのは ROS2 Jazzy であり整合していない。『22.04+Humble』『24.04+Jazzy』『24.04+Humble(非公式)』のどれで行くかは着手前にPM(水谷)と確定すること。本ガイド中の /opt/ros/humble・humble という記述・apt パッケージ名（ros-humble-… 等）は、確定したディストロ名 /opt/ros/<distro> に読み替える。

前提タスクにぶら下がっている

本Issueは「走行系(/cmd_vel に値を流すと実機が動く＝オドメトリ odom→base_link の TF も出る)」「LiDARセットアップ(型番確定・ros2で /scan が出る)」が先に確立していることが前提。⚠️注意：この2つを単独で扱う専用ガイドは現状この手引き内にまだ存在しない（hw-motor はモータ選定・配線まで、/cmd_vel を出すROS2駆動ノードやオドメトリpublishは未カバー）。揃っていなければ別途PM(水谷)に確認のうえ起票・整備するか、本Issue内のステップ1〜の最小確認で代替する。「既存ガイドへ戻る」先を探しても見つからないので注意。

完了の定義（Done）

  • SLAM(slam_toolbox)を起動し、ロボを走らせて部屋の地図ができる（RVizで部屋の形が見える）
  • 地図がファイルに保存できる（2系統・各2ファイル：Nav2で使う形式なら .pgm(画像)+.yaml(設定)の2点、slam_toolbox独自保存(serialize)なら .posegraph+.data の2点が一組で残る）
  • 保存した地図が実際の部屋とおおむね合っている（合格目安：主要な壁2面と本棚の位置が判別できる／白い未探索の大穴が無い）
  • 席3つの座標 (x, y, theta) が実機から採れている（TFまたは推定姿勢から数値を読み取れた）
  • 採った3つの座標が stop_points テーブルに入っている（SQL Editorで insert 済み、select で3行見える）
「実機で席へ行かせる」検証はこのIssueでは任意

座標が本当に正しいか（行かせて確かめる）は Nav2が動く別Issue完了後の追検証。本Issue段階でNav2未導入ならスキップ可。ここで行かせ方が分からなくても自分の失敗ではない。

進め方ステップ

  1. ステップ0：ROS2ワークスペースの基本（最初に押さえる）
    apt で入れたパッケージ（slam_toolbox等）はそのまま ros2 run/launch で動くが、自分でgit cloneした/自作したノードは colcon build → source install/setup.bash をしないと使えない。「コマンドが見つからない(command not found / package not found)」ときは、まず source し忘れを疑う。新しいターミナルを開くたびに source /opt/ros/<distro>/setup.bash（例: /opt/ros/humble/setup.bash。⚠️ distro名は上記『要確認(PM)』のとおり Humble/Jazzy で確定値が変わるので hw-pi5-setup の確定値に読み替える）と、必要なら自分のワークスペースの setup.bash が要る。詳しくはresearch『ROS2の起動準備』。
  2. ステップ1：前提を確認する
    (a) 走行できるか＝『/cmd_vel に値を流すと実機が動く』が達成済みか。ros2 topic info /cmd_vel で購読ノード(モーター制御)がいるか確認。(b) オドメトリが出ているか＝ros2 topic echo /odom や ros2 run tf2_ros tf2_echo odom base_link で odom→base_link の TF が出ているか確認（後述のとおりSLAMが地図化に必須）。(c) LiDARが点群を出すか＝ros2 topic echo /scan（または使用ドライバのトピック）でデータが流れるか確認。使用LiDARの型番が分からなければ、まずPM(水谷)・ハ設に聞く（型番が決まらないとドライバも調べられない／型番は本資料・robot/READMEに未記載）。⚠️ これら走行系・LiDARセットアップを単独で扱う専用ガイドは現状この手引き内に存在しないので、未完了ならPMに確認のうえ整備・起票する（戻る先のガイドは無い）。
  3. ステップ2：TFツリーが繋がっているか確認する
    SLAMが動くには map → odom → base_link → laser(scanのフレーム) という TFツリーが揃っている必要がある（map→odom は slam_toolbox が出すが、odom→base_link は走行系(駆動ノードのオドメトリpublish)、base_link→laser は robot_state_publisher/URDF が出す）。ros2 run tf2_tools view_frames で frames.pdf を出し、想定どおり繋がっているか確認する。TFが欠けると source 忘れとは別系統で『地図ができない/No transform』になる。欠けていれば走行系・robot_state_publisher 側を先に整える。
  4. ステップ3：slam_toolbox を起動する
    slam_toolbox を apt で入れ、オンライン(逐次)マッピング用の launch を起動する。起動の正確なコマンド名/launchファイル名は research『slam_toolboxの起動』で公式READMEを見て確認する（答えは書かない）。起動できると、地図作成モードが立ち上がる。
  5. ステップ4：RVizを開いて地図を見られるようにする
    別ターミナルで RViz を起動し、Map 表示と LiDAR の LaserScan 表示を追加して、地図が育っていく様子を見られるようにする。最初は自分の周りだけ地図ができている。フレームは map を基準にする。
  6. ステップ5：teleop（キーボード操作）でロボを動かす準備
    ロボを動かすのはteleop(キーボード遠隔操作)で行う。teleop_twist_keyboard パッケージを使い、押したキーが /cmd_vel(速度指令トピック)に流れ、それをモーター制御ノードが受けて実機が動く、という経路。まず ros2 topic info /cmd_vel でモーター側がそのトピックを購読しているか確認してから teleop を起動する。起動方法と操作キーは research『teleopの起動と確認』を参照。
  7. ステップ6：ゆっくり走らせて地図を作る
    teleopでゆっくり部屋を一周する。急加速・急旋回はLiDARが追従できず地図がズレる。壁ぎわや本棚の前を通って、RViz上で地図が部屋全体に広がるまで走る。白い未探索の穴が埋まり、壁が一本につながればよい。
  8. ステップ7：地図を保存する
    Nav2が後で読む地図(.pgm+.yaml)を作るなら ros2 run nav2_map_server map_saver_cli（=Nav2側のツール。パッケージ名込みで覚える）を使う。注意：slam_toolbox には独自の保存(serialize → .posegraph+.data の2点で一組、RVizのSlamToolboxプラグインのパネルからも保存可)もある。Nav2で使うなら map_saver で .pgm/.yaml を出す。どちらを採るかは research で確認。
  9. ステップ8：席の座標を採る（x, y, theta）
    ロボを採りたい席の位置・向きに実際に置く。そのうえで地図座標系での自己位置を数値で読む。第一候補は ros2 run tf2_ros tf2_echo map base_link（=map からロボ本体への変換＝ロボの位置と向き）。ここに出る x, y が席の座標。向きはクォータニオン(x,y,z,w の4値)で出るので、theta(yaw, ラジアン)へ変換が要る（research参照）。先に自分のロボのベースフレーム名を確認：base_link とは限らず base_footprint 等のこともある（view_frames/RVizのTFツリー/robot_state_publisher で確認）。違うフレーム名なら読み替える。No transform/frame does not exist になる場合は (1)フレーム名違い (2)odom→base_link の TF 欠如（走行系未整備）が原因。採取の運用：ロボを止めて数秒待ち、値が落ち着いてから読む（online中は map→odom がドリフトで揺れる）。数回読んで大きく違わないか確認するとよい。slam_toolbox使用中で AMCL を使っていない場合は tf2_echo か slam_toolbox の推定位置を使うのが現実的（AMCLを使う構成なら ros2 topic echo /amcl_pose でも読める）。3席ぶん繰り返す。
  10. ステップ9：stop_points に座標を入れる（SQL Editor）
    Supabase の SQL Editor から stop_points に insert。列は id, label, x, y, theta（created_at はdefaultありなので省略可）。id は自動採番ではない手動採番の int。席ごとに 1, 2, 3 など重複しない番号を自分で決めて入れる。この id が後で robot_calls.seat_id の参照先（＝アプリの席選択で使う番号）になるので、番号の割り当てはアプリ担当と取り決めておく。SQLは下のコードブロック参照。

📚 進めながら調べる・学ぶこと

  • SLAM の考え方（slam_toolbox / Cartographer の違い）
    「地図を作る」と「自分の位置を知る」を同時にやるのがSLAM。仕組みのイメージが無いと、なぜゆっくり走るのか・なぜオドメトリが要るのかが腑に落ちず、ズレた地図をそのまま使ってしまう。
    slam_toolbox 公式README: https://github.com/SteveMacenski/slam_toolbox （online/offline モードの違い）。比較対象として Cartographer の概念も。ROS2 Humble docs: https://docs.ros.org/en/humble/ 。
  • ROS2の起動準備（source / colcon build）
    「コマンド/パッケージが見つからない」の最頻原因が source 忘れ。新しいターミナルごとに環境を読み込む必要があり、自作ノードは build も要る。ここを知らないと最初の一歩で詰まる。
    ROS2 Humble: https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Configuring-ROS2-Environment.html および Colcon ビルドのチュートリアル（同サイト内 Beginner: Client libraries）。
  • LiDARのセットアップ（型番確定→ros2ドライバ）
    LiDARが点群(/scan)を出さないとSLAMは何もできない。製品ごとにドライバ(ros2パッケージ)が違うため、まず使用LiDARの型番を確定させてから『型番 + ros2 driver』で探す必要がある。
    まずPM(水谷)・ハ設に使用LiDARの型番を確認する（型番は本資料・robot/READMEに未記載。LiDARセットアップを単独で扱う専用ガイドはこの手引き内に現状無いため、人に聞くのが確実）。型番確定後に『(型番) ros2 driver』で公式リポジトリを検索。レーザークラスは製品仕様書で確認（多くの屋内2D LiDARはClass1=目に安全だが要確認）。
  • teleopの起動と確認（/cmd_vel への経路）
    ロボを動かせないとマッピングできない。teleopでキー入力を /cmd_vel に流し、モーター制御ノードがそれを受けて動く、という経路を理解・確認する。地図ズレの本質は『手押しかteleopか』ではなく『odom→base_link の TF が出ているか』。teleopで /cmd_vel を流しても駆動ノードが車輪エンコーダ等から odom を publish していなければ同様にズレるし、逆に外部から odom TF が供給されていれば手押しでも地図化はできる。まず odom が出ているかを確認すること。
    まず ros2 topic info /cmd_vel で購読ノード(モーター制御)がいるか確認 → ros2 run teleop_twist_keyboard teleop_twist_keyboard の操作キー。teleop_twist_keyboard: https://github.com/ros-teleop/teleop_twist_keyboard 。Nav2 Getting Started: https://docs.nav2.org/getting_started/index.html 。
  • 地図の保存（map_server / slam_toolbox serialize の2系統）
    保存手段が2系統あり出力形式も違う（各2ファイル）。Nav2で使うなら nav2_map_server の map_saver で .pgm+.yaml、slam_toolbox独自(serialize)なら .posegraph+.data の2点で一組。混同すると「コマンド/サービスが無い」で迷う。
    Nav2 map_server: https://docs.nav2.org/ （Map Server / map_saver_cli）。slam_toolbox の serialize/save 機能: https://github.com/SteveMacenski/slam_toolbox の README（Saving/serialization の節）。
  • 席座標 (x, y, theta) を正しく読む手段（TF / 推定姿勢）
    RVizの『2D Pose Estimate』は座標を読むツールではなく、自己位置をクリックで“与える”入力ツール（/initialpose にpublish）。これで座標を読もうとすると確実に詰まる。正しくは TF か推定姿勢から数値を取る。
    TF/tf2_echo: ROS2 Humble TF2 チュートリアル https://docs.ros.org/en/humble/Tutorials/Intermediate/Tf2/Tf2-Main.html （ros2 run tf2_ros tf2_echo map base_link）。ベースフレーム名は base_link とは限らない(base_footprint 等)ので先に view_frames/RVizのTFツリーで確認して読み替える。No transform になるなら odom→base_link の TF 欠如が原因（走行系未整備）。online中はドリフトで値が揺れるので静止後に読み複数回確認する。AMCL構成なら Nav2 docs の /amcl_pose。x,yの当たりだけなら RViz の『Publish Point』（数値はターミナルログに出る／向きthetaは出ない）。
  • クォータニオン → yaw(theta) 変換と座標系の向き
    ROSの向きは内部でクォータニオン(x,y,z,w の4値)で持つ。tf2_echoや/amcl_poseもクォータニオンで出る。stop_points.theta に入れる1つの角度(yaw, ラジアン)へは変換が必要。『thetaが無い』と混乱しがちな所。基準は地図の+x方向、反時計回りが正。
    REP-103(座標系・単位の標準): https://www.ros.org/reps/rep-0103.html 。tf2の変換: https://docs.ros.org/en/humble/Tutorials/Intermediate/Tf2/Tf2-Main.html 。yaw換算の考え方(euler_from_quaternion 等)を確認（コードは書かず考え方として）。
  • stop_points へ SQL Editor で入れる理由とid採番
    「insertポリシーが無い＝誰も入れられないのでは？」と矛盾に見えて不安になる。SQL Editorは管理者権限(service role)で動くためRLSをバイパスして入れられる。一方アプリのブラウザ(匿名/publishableキー)はポリシーが無いので弾かれる、という非対称が仕様。id採番ルールも要確認。
    Supabase RLS / service role: https://supabase.com/docs/guides/database/postgres/row-level-security （bypass / service_role の節）。schema.sql(supabase/schema.sql)で stop_points に select ポリシーのみ・insertポリシー無し・id が手動採番int であることを実物で確認。

つまずきポイントと対処

症状・勘違い原因・対処
コマンド/パッケージが見つからないsource忘れが最頻。新ターミナルごとに source /opt/ros/humble/setup.bash（自作ノードなら自分のワークスペースの install/setup.bash も）。自作ノードは先に colcon build。
地図が大きくズレる・二重に見える走るのが速すぎ、またはodom(odom→base_link の TF)が出ていないのが主因。teleopでも車輪エンコーダ等のオドメトリ source が無ければ同様にズレる。まず ros2 topic echo /odom や ros2 run tf2_ros tf2_echo odom base_link で odom が出ているか確認し、出ていなければ走行系(駆動ノードのオドメトリpublish)を先に整える。その上でゆっくり走らせる。
『2D Pose Estimate』をクリックしても座標が出ないそれは座標を“読む”ツールではなく自己位置を“与える”入力ツール。座標は ros2 run tf2_ros tf2_echo map base_link や /amcl_pose から読む（ステップ8）。
thetaが見当たらない（x,y,z,wの4値が出る）向きはクォータニオンで出る。stop_points.theta には yaw(ラジアン)へ変換した1つの角度を入れる。変換はtf2 docs/REP-103参照。
map_saver でコマンド/サービスが無いと言われるmap_saver_cli は Nav2(nav2_map_server) のツール。ros2 run nav2_map_server map_saver_cli。slam_toolbox独自保存(serialize)とは別系統なので混同しない。
RVizが映らない/重い（別PCで見たい）別PCで見るときは両方のターミナルで export ROS_DOMAIN_ID=同じ数字 を実行（.bashrcに書くと毎回不要で楽）。同一Wi-Fi/LAN(同一サブネット)であること。繋がらないときはまずDOMAIN_IDの不一致、次にファイアウォールを疑う。
SQL Editorで入れていいのか不安（insertポリシー無し）SQL Editorは管理者権限(service role)でRLSをバイパスするので入る。アプリの匿名キーは弾かれる、という非対称が仕様。schema.sqlの通りで正しい。
stop_points.id をどう決める？自動採番ではない手動採番のint。1,2,3 など重複しない番号を自分で決める。robot_calls.seat_id とアプリの席選択UIの番号と整合させるため、アプリ担当と取り決める。
安全注意（実機を動かす作業）

実機を動かす際は 転倒・挟み込み・人やモノへの突進に注意。teleopは低速から。緊急停止の手段（電源スイッチの位置・キー操作での停止）を動かす前に必ず確認しておく。床の段差・ケーブルの巻き込みにも注意。LiDARのレーザーは使用製品のレーザークラスを仕様書で確認（多くはClass1=目に安全だが要確認）。

動作確認のしかた

  1. RVizで地図を見ながら一周し、白い未探索の大穴が無く、壁が一本につながり、本棚の位置が分かること（地図が部屋とおおむね一致）。
  2. 地図を保存 → ファイルが残ること（Nav2用なら .pgm+.yaml の2点が生成される）。保存した画像を開いて部屋の形に見えればOK。
  3. ロボを各席に置いて tf2_echo（または /amcl_pose）で x,y を読み、向きを yaw に変換して theta を得る。3席ぶん数値がメモできていること。
  4. Supabase SQL Editor で select * from stop_points; を実行し、入れた3行(席1/2/3)が x,y,theta 付きで見えること。idが1,2,3など重複なく振られていること。
  5. 【任意・後日】Nav2が動く別Issue完了後に、1席だけ実機で行かせて座標が妥当か追検証。本Issue段階でNav2未導入ならスキップ可（座標の最終確定はNav2導入後）。

用語ミニ辞典

言葉かんたんな意味
SLAM走りながら地図を作り、同時に自分の位置も推定する技術。
slam_toolboxROS2でSLAMを行う代表的パッケージ。今回これを使う。
LiDAR(ライダー)レーザーで周囲の距離を測るセンサ。地図と自己位置の元データ(点群/scan)を出す。
RVizROSの可視化ツール。地図やセンサ値、ロボの位置を画面で見る。
teleopキーボード等での遠隔手動操作。teleop_twist_keyboardで/cmd_velに速度指令を流す。
/cmd_velロボへの速度指令トピック。teleopやNav2がここに値を流し、モーター制御が受け取る。
オドメトリ車輪の回転などから推定する『自分がどれだけ動いたか』。SLAMはこれ(odom→base_link の TF)を併用して地図を整える。odom が出ていないと teleopでも地図がズレやすい。
TF / tf2_echo座標系どうしの位置・向きの関係。tf2_echoでmap→base_linkの位置(x,y)と向きを読める。
map / base_linkmap=地図の基準座標系、base_link=ロボ本体の座標系（ロボによっては base_footprint 等のことも）。両者の関係＝地図上のロボの位置。tf2_echo で読むときは自分のベースフレーム名を確認して読み替える。
クォータニオン向きを4つの数(x,y,z,w)で表す方式。stop_pointsに入れるyaw(角度)に直して使う。
theta(yaw)地図上での向き。ラジアン、基準は+x方向で反時計回りが正。クォータニオンから変換して得る。
map_saver_cliNav2(nav2_map_server)の地図保存ツール。.pgm(画像)+.yaml(設定)を出す。
.pgm / .yamlNav2が読む地図ファイル。.pgmが画像、.yamlが縮尺や原点などの設定。
ROS_DOMAIN_ID同じ番号同士のROS2機だけが通信できる仕分け番号。別PC連携時は両機で同じ値に揃える。
stop_points席(停車ポイント)の座標表。id,label,x,y,theta を持つ。idは手動採番。
service role / RLSバイパス管理者権限。SQL Editorはこれで動くのでRLSを無視して書き込める。アプリの匿名キーは弾かれる。
