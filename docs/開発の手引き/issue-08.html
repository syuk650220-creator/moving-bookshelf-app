このIssueは何？

アプリ（Web）とロボ（Raspberry Pi）は直接つながりません。間に Supabase のテーブルを置き、アプリは「呼んで！」を robot_calls に1行 insert し、ロボはそのテーブルを見張って動きます。この「テーブルを見張ってロボの仕事に変える係」がブリッジ（橋渡し役）です。このIssueはその最小版を作ります。

今回のゴールは1つだけ：robot_calls に status='queued'（順番待ち）の行が入ったらPythonで検知し、その行の status を moving（移動中）→ arrived（到着）→ done（完了）と順に書き換えながらログを出す。Nav2（ロボの自律移動）との連携は次の #9 でやるので、今回は「検知してログ＋status更新」まででOKです。

「ブリッジ＝橋渡し役」の出どころ

この役割は実在ファイル robot/README.md の「持ち場」と「つなぎ目（疎結合の原則）」に書かれています：『アプリとロボは直接やり取りせず、Supabase のテーブル（robot_calls / robot_status など）を介してつながる』。※docs/基本設計書.md は現時点ではプレースホルダ（4行のみ・中身は補遺で確定予定）なので、仕様の出どころは robot/README.md と supabase/schema.sql を正典にしてください。

Piが無くても大丈夫

このIssueはPCだけで完結できます（やることはSupabaseに繋いでテーブルを読み書きするだけ）。Pi実機での実行確認は #9 以降でOKです。

完了の定義（Done）

  • Python仮想環境(venv)を作り、supabase-py をインストールできた
  • 接続情報を環境変数（コードに直書きしない）で渡し、Supabaseに繋げた
  • robot_calls を見張る（ポーリング or Realtime購読）コードが robot/ 配下に置いてある（例: robot/bridge_node.py）
  • アプリ or SQLで robot_calls に1件 insert すると、ブリッジ側のログにその行が出る
  • 検知した行の status を queued→moving→arrived→done と更新でき、各段階がログに出る（実機なしでOK）
  • FIFO（先に来た順）で1件だけ掴んでいる（created_at 昇順で先頭1件）
  • 接続キー・URLを直書きしておらず、git status に .env が出てこない（コミットされない）ことを確認した
  • ブランチ（feature/◯◯）を切ってPRを出した（Pi実機での実行確認は #9 以降でOK。Piが手元にある人は git pull→実行まで試す）
robot_status は今回のスコープ外

スキーマには robot_status（id=1固定・ロボの現在状態を1行で持つ表）もありますが、今回の最小実装では robot_calls の status のみ更新し、robot_status.state / current_call_id の更新は次段階（#9以降）の対象とします（範囲をわざと絞ります）。robot_status を誰がいつ更新するかは設計確定待ちなので、迷ったらPM(水谷)に確認を。

進め方ステップ

  1. Python仮想環境(venv)を作る
    robot/ で作業用の仮想環境を作ります。venv＝そのプロジェクト専用のPython部屋（他と汚し合わないため）。コマンドは python -m venv .venv → 有効化(Win: .venv\Scripts\activate / Mac: source .venv/bin/activate)。詳しい意味は下の『調べる・学ぶこと』の venv 項目へ。
  2. supabase-py を入れる
    有効化した状態で pip install supabase。これがPythonからSupabaseを操作するライブラリ。pip freeze > requirements.txt で何を入れたか記録しておくとチーム共有が楽。
  3. 使うSupabaseキーの種別を決める（重要）
    今回は publishable key（旧 anon に相当する読み書き可キー）で十分です。理由：RLSで select/insert/update は全員OK設計なので、最小権限で足ります。service_role キーは使いません（RLSを無視する強権キーで、漏れると全データ操作可能になるため）。ロボ側で使う具体的なキー名は環境構築ガイドにまだ記載が無いので、正式なキー名はPM(水谷)に確認してください。
  4. 接続情報を環境変数で渡す準備
    URLとキーをコードに直書きせず、robot/ に .env を作って読み込む方針にします。キー名は仮に SUPABASE_URL / SUPABASE_KEY のように NEXT_PUBLIC_ 接頭辞なしで。NEXT_PUBLIC_ はブラウザに公開する用の名前なので、サーバー的に動くロボ側Pythonでは付けません。実際のキー名はPM確認の結果に合わせること。
  5. Supabaseに繋いでみる（疎通テスト）
    環境変数からURL/キーを読み、クライアントを作って robot_calls を select するだけの短いコードで『繋がるか』を先に確認。エラーが出たらまずキー名・値・RLSを疑う。
  6. 検知の方式を選ぶ：まずはポーリング
    見張り方は2通り：(A)ポーリング＝数秒ごとに自分でテーブルを見に行く、(B)Realtime購読＝変化が来たら通知される。最小実装は実装が単純な(A)ポーリングを推奨。※注意：robot/README.md の最終的な持ち場は『Realtime購読』なので、これは“まず動かすための暫定”。後でRealtimeに差し替える前提で進めます。違いは下の研究項目で。
  7. FIFOで先頭1件だけ掴む
    robot_calls から status='queued' の行を created_at 昇順で並べ、先頭の1件だけ取得（order by created_at + limit 1 の考え方）。複数あっても一度に1件だけ処理するのがロボ1台の前提。
  8. status を順に進める：moving→arrived→done
    掴んだ行の id を指定して update。実機が無いので各段階の間は『擬似的に少し待つ(sleep数秒)』か『キー入力で1段階ずつ進める』どちらかでOK。各更新の前後で print ログを出し、3段階が見えるようにする。canceled（取消）は今回の範囲外＝来ても掴まない程度の認識でOK。
  9. robot/ に置いてブランチ→PR
    ファイルは robot/bridge_node.py など robot/ 配下へ。feature/◯◯ ブランチを切って push → PR（本文に Closes #8 と動作確認を書く）。Piがある人は別マシンで git pull→実行まで確認。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心。各ステップで詰まりやすい所を「何を / なぜ / どこで調べるか」で並べます。答えのコードは載せません。自分の手で書いて学んでください。

  • Python仮想環境(venv)の作り方と有効化
    Git/Python初心者が最初に必ずぶつかる関門。venvを使わないとPCのPython本体を汚し、ライブラリ衝突の原因になる。
    Python公式 venv: https://docs.python.org/ja/3/library/venv.html / pip入門: https://packaging.python.org/ja/latest/tutorials/installing-packages/
  • supabase-py のセットアップと使い方（create_client / select / update）
    PythonからSupabaseを操作する土台。クライアントの作り方とテーブル操作の書き方を知らないと一歩も進めない。
    supabase-py 公式リポジトリ: https://github.com/supabase/supabase-py / JS版だがメソッド名の対応が分かる select: https://supabase.com/docs/reference/javascript/select
  • Realtime購読 と ポーリング の違い
    今回はポーリング採用だが、robot/README.md の最終形はRealtime購読。違いと『なぜ最小実装ではポーリングか／いつ差し替えるか』を理解しておくと後の #9 がスムーズ。
    Supabase Realtime: https://supabase.com/docs/guides/realtime / ポーリングは『一定間隔でselectを繰り返す』だけなので supabase-py の select を time.sleep と組むイメージ
  • FIFOで先頭1件を取る（order by created_at limit 1）
    queuedが複数あってもロボは1台。先に来た順（FIFO）で1件だけ処理しないと割り込み・取りこぼしが起きる。schema にも専用インデックス idx_calls_queue（created_at, status='queued'）がある。
    supabase-py の order/limit の書き方（公式リポジトリのREADME/サンプル）: https://github.com/supabase/supabase-py / 根拠は supabase/schema.sql 73行（FIFO処理用インデックス）
  • status の update（行を特定して書き換える）
    検知だけでなく moving→arrived→done と状態を進めるのがブリッジの仕事。どの行を更新するかを id で絞る eq の使い方が肝。
    supabase-py 公式リポジトリ: https://github.com/supabase/supabase-py / 更新の考え方（JS版update）: https://supabase.com/docs/reference/javascript/update
  • PythonでSupabaseに繋ぐ環境変数の扱い（どのキーを使うか）
    キーをコードに直書きすると漏洩→全データ操作のリスク。さらに『どの種別のキーを使うか』を間違えると権限不足や権限過大になる。
    ロボ側のキー名は環境構築ガイドにまだ記載が無いため、まずPM(水谷)に確認。方針：publishable key（読み書き可・RLS適用）で足り、service_role は使わない/コミットしない。環境構築ガイド5-3が service_role の共有・コミットを固く禁じている点を参照。Pythonでの.env読み込みは python-dotenv: https://pypi.org/project/python-dotenv/
  • robot_calls の status 遷移の意味
    queued/moving/arrived/done/canceled が何を表すかを取り違えると、アプリ側の表示や次のロボ動作とズレる。
    出どころは supabase/schema.sql 55-56行の check 制約（status in queued|moving|arrived|done|canceled）。意味は下の用語ミニ辞典に整理済み。robot_status の state（idle|moving|arrived|returning）は同 65-66行（※今回は更新対象外）。

つまずきポイントと対処

外部キー違反でinsertが失敗する

robot_calls.seat_id は stop_points(id) への外部キー、book_id は books(id) への外部キー（schema 53-54行）。stop_points や books が空だと『存在しないIDを指している』とエラーになり、原因不明で止まりがち。先に #3（サンプル投入）が済んでいるか、最低1件ずつデータを用意してから試すこと。

接続キーを絶対にコミットしない

使うキーは publishable（RLS適用の読み書き可キー）に限定。service_role は使わない（RLSを無視する強権キーのため）。robot/ に .env を作るなら、ルートの .gitignore は .env* を無視対象にしているので robot/.env も対象のはず。ただし必ず git status で .env がコミット候補に出てこないことを目視確認すること。

  • ModuleNotFoundError: supabase：venvを有効化せずに python を実行している/別の環境に入れた。activate し直して pip install を確認。
  • 繋がらない・401や権限エラー：URLかキーの値ミス、またはキー種別違い。キー名はPM確認の値と一致しているか、publishable を使っているかを確認。
  • queued が検知されない：insert した行の status が本当に queued か（既定値は queued なので status を指定しなければOK）、ポーリング間隔が長すぎないか、フィルタ条件（status='queued'）が合っているか。
  • robot_status はいつ更新するの？という疑問：今回は更新しません（範囲外）。次段階で扱います。混乱したらDoneのcalloutを再読。

動作確認のしかた

実機なしでPCだけで確認します。前提：books と stop_points に最低1件ずつデータがあること（#3サンプル投入が済んでいるか、手動で1件用意）。

  1. 実在IDを確認する
    SupabaseのSQL Editorで、insertに使う実在のIDを先に調べる。
  2. ブリッジを起動する
    venvを有効化して python robot/bridge_node.py（ファイル名は自分の付けた名前）。起動後、見張りが回り始めるログが出るはず。
  3. robot_calls に1件 insert する
    別の場所（SQL Editor または アプリのボタン）から1件追加。status は省略すれば自動で queued。
  4. ログを確認する
    （自動で進める方式の場合）数秒ごとに moving→arrived→done と3段階のログが順に出ればOK。（キー入力で進める方式の場合）案内に従ってキーを押すたびに1段階ずつ進み、3回でdoneになればOK。どちらの方式にしたかで操作が変わる点に注意。
  5. DBの値も確認する
    SQL Editorで対象行の status が done になっているかを見る。ログとDBの両方で確認できれば完了。
-- ① 実在IDを先に確認（コピペ用）
select id, label from stop_points;
select id, title from books;

-- ② 上で出た実在の id に置き換えて1件作る（手順イメージ。完成文を丸暗記せず列名の意味を理解して）
--   robot_calls に必要なのは book_id（booksの実在id）・seat_id（stop_pointsの実在id）・requested_by（誰が呼んだか/文字列）。
--   status は省略すると自動で 'queued'。
insert into robot_calls (book_id, seat_id, requested_by)
values ('ここにbooksの実在id', ここにstop_pointsの実在id, 'test');

-- ③ 進み具合を確認
select id, status, created_at from robot_calls order by created_at desc limit 5;
