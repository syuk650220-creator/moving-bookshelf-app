このIssueは何？

本を借りたい席が決まると、ロボはその席まで自走します。席の場所は DB の stop_points（停車ポイントの座標表）に x, y, theta（位置と向き）で入っています。このIssueは、その座標を Nav2（ROS2 のナビゲーション基盤）が理解できる「ゴール」の形に変換する関数の素案 を作るのがゴールです。

全体の流れは #8 ブリッジ（最小実装では Supabase の robot_calls の queued 行をポーリングで検知してロボに伝える。Realtime 化は後段）→ このIssue #9 で座標をゴールに変換 → Nav2 がそこまで走る、という順番です。robot/README ではブリッジに変換も含めて記述されていますが、本プロジェクトでは #8＝検知・状態更新、#9＝座標→PoseStamped 変換、と分担します（最終的な担当境界は PM 確認）。

これは下調べ中心のIssue

実機にマップを作らせて座標を確定させる工程（資料では W5＝プロジェクト第5週の『実機マッピング』と呼びます。具体的な日付はスケジュール表を参照）より前なので、いまは「実際の座標値」はまだありません。なので #9 は『仕組みを調べて、値が決まったらすぐ動かせる関数の雛形を用意しておく』のが目的です。完璧な実装より、考え方を整理してメモと素案を残すことを重視してください。

進め方ステップ

  1. 全体像をつかむ
    『席の座標(x,y,theta) → Nav2 のゴール(PoseStamped)』という変換が必要、という1本の流れを頭に入れます。入口は #8 から渡ってくる seat_id（または座標）、出口は Nav2 に送る PoseStamped です。
  2. ROS2 の座標系と姿勢を調べる
    x,y は平面上の位置、theta は向き（ヨー角）。ROS2 は向きを内部では quaternion（4つの数 x,y,z,w）で持ちます。なぜ theta だけで足りないのか、theta と quaternion の対応を調べてメモします。
  3. Nav2 のゴール送信の形を調べる
    Nav2 へは NavigateToPose というアクションでゴールを送ります。ゴールは PoseStamped 型（header に座標系名 frame_id と時刻 stamp、pose に位置と向き）。frame_id は通常 "map" です。型の構造をメモします。
  4. theta → quaternion 変換の方法を調べる
    平面ヨー1軸なら数式は qz = sin(theta/2), qw = cos(theta/2), qx = qy = 0 だけです。tf_transformations の quaternion_from_euler を使う手もありますが、入っているか先に確認します（つまずきポイント参照）。
  5. stop_points から座標を引く方法を #8 と決める
    座標を DB から取る役を #8（ブリッジ）側にするか #9 側にするかを相談します。このIssueの既定は『取得は #8、#9 は x,y,theta を受け取って PoseStamped を返す純粋関数』です。違う結論なら理由をコメントに残します。
  6. 関数の雛形だけ書く（形を先に）
    robot/nav_goal.py に、関数名・引数(x,y,theta)・戻り値の型(PoseStamped)・docstring・中身は # TODO だけを書きます。まず『形』を先に作ります。中身はまだ空でOK。
  7. 数値変換部分だけ実装して単体確認
    theta → qz,qw の計算だけを標準ライブラリ(math)で書き、手元PC（ROS2 無し）で値を出して確認します。PoseStamped を組み立てる部分は Pi/ROS2 環境で後から埋めます。
  8. 調べたことをメモにまとめてPRに添える
    座標系・NavigateToPose・quaternion変換・入力の決め方を箇条書きでまとめ、素案コードと一緒にPRを出します。Done のチェックと突き合わせます。

📚 進めながら調べる・学ぶこと

ここがこの資料の核心です。各ステップで詰まりやすい概念を『何を / なぜ / どこで』で示します。答えは載せません。自分で開いて確認してください。

  • ROS2 の座標系と角度の単位（REP-103）
    x,y,theta の theta が『度』か『ラジアン』かを取り違えると、向きが何十倍もずれてロボが明後日を向きます。schema.sql の theta は default 0 で単位コメントが無いので、チームで単位を確定する必要があります。
    ROS の規約 REP 103（https://www.ros.org/reps/rep-0103.html）。ここに『角度は必ずラジアン・右手系』と明記されています（ROS2 では度は使わない）。これを根拠に『theta はラジアンで統一』とチームで合意しメモに残します。
  • 姿勢の表し方（theta ↔ quaternion）
    DB は theta（1つの角度）で持ちますが、ROS2 のゴールは quaternion（4つの数）を要求します。なぜ ROS が quaternion を使うのか、どう変換するのかを理解しないと、向きを正しくセットできません。
    geometry_msgs/Pose の orientation の型定義（https://docs.ros2.org/latest/api/geometry_msgs/msg/Pose.html）と、Quaternion の意味（https://docs.ros2.org/latest/api/geometry_msgs/msg/Quaternion.html）。平面なら qz,qw だけで足りる理由も合わせて確認します。
  • Nav2 のゴール送信（NavigateToPose と PoseStamped）
    ゴールの『入れ物』が PoseStamped だと分からないと、関数の戻り値の型が決められません。特に header.frame_id（通常 "map"）と stamp（現在時刻）を入れ忘れるとゴールが無視されて動きません。
    Nav2 公式 docs（https://docs.nav2.org/ の Navigating / NavigateToPose のページ）と geometry_msgs/PoseStamped の型定義（https://docs.ros2.org/latest/api/geometry_msgs/msg/PoseStamped.html）。header(stamp+frame_id) と pose(position+orientation) の入れ子構造をメモします。
  • theta → quaternion の変換ユーティリティの有無を確認
    tf_transformations の quaternion_from_euler は便利ですが、ROS2 Humble では標準で入っておらず別途インストール（python3-transforms3d 依存）が要ることがあります。知らずに import すると ModuleNotFoundError で必ず詰まります。
    ros-humble-tf-transformations の有無は ROS Index（https://index.ros.org/p/tf_transformations/）と GitHub（https://github.com/DLu/tf_transformations）で確認。apt list --installed | grep tf-transformations で導入済みかも確認。入っていなければ apt で追加するか、平面ヨー1軸なら qz=sin(theta/2), qw=cos(theta/2) を自前計算します。
  • stop_points から座標を引く方法（DB取得の担当分け）
    座標を DB から取る役を #8 と #9 のどちらが持つかを決めないと、関数の入力が seat_id か x/y/theta かが定まらず Done 判定がぶれます。
    Supabase JS は使わずロボ側は supabase-py（https://github.com/supabase/supabase-py）で stop_points を select します。列名は schema.sql（supabase/schema.sql の stop_points: id, label, x, y, theta）で正確に確認。既定は『取得は #8、#9 は x,y,theta を受け取る純粋関数』。
  • 実機が無い段階でシミュレータ(Gazebo)で試す選択肢
    W5 の実機マッピングより前でも、シミュレータ上で NavigateToPose にゴールを送る一連を体験しておくと、本番で慌てません。ただしバージョン差で手順が変わります。
    Nav2 公式の Getting Started（https://docs.nav2.org/getting_started/index.html）。TurtleBot3 + Gazebo の例が現行の Humble 版で Gazebo Classic / Ignition のどちらの手順かを確認してから始めます（版違いで詰まりやすいので、自分の環境の docs の版に合わせる）。
  • ロボ側コードの運用ルール（PCで書いて Pi で git pull）
    ロボ側は『PCの VS Code で書いて push → Pi で git pull して実行』という流れになります。CONTRIBUTING.md はアプリ向けの npm run dev 前提なので、この Pi 運用と混同しないことだけ注意します。
    この運用は Pi5 セットアップガイド(hw-pi5-setup の『この先のワークフロー』)と #8(ブリッジ最小)で既に説明されています。まずそれらを読んでください。robot/README.md にはこの運用への1行リンクを足すと親切ですが、二重記述になるので本文の作り込みは不要です。

つまずきポイントと対処

角度の単位（度とラジアン）

ROS2 は REP-103 で角度は必ずラジアンです（度は使わない）。theta=0 は『無回転＝マップの x 軸の正方向を向く』を意味します。schema.sql の theta は単位コメントが無いので、チームで『ラジアン統一』と決めてメモに残しておかないと、誰かが度で入れて事故ります。

tf_transformations が import できない

ROS2 Humble では tf_transformations / quaternion_from_euler が標準で入っていないことがあります。import で ModuleNotFoundError が出たら、apt で ros-humble-tf-transformations（python3-transforms3d 依存）を入れるか、平面ヨーなら qz=sin(theta/2), qw=cos(theta/2)（qx=qy=0）で自前計算します。

frame_id の設定漏れでゴールが効かない

PoseStamped の header.frame_id を空のまま送ると、Nav2 はどの座標系のゴールか分からずゴールが無視されます。通常は "map" を入れ、stamp には現在時刻を入れます。これは初心者の最頻つまずきです。

手元PC(Windows・ROS2無し)で関数全体を動かそうとする

PoseStamped など ROS2 の型を使う部分は ROS2 環境が無いと import できず ModuleNotFoundError になります。手元で確認できるのは theta→qz,qw の純粋な数値計算だけ。型を組み立てる部分は Pi/ROS2 環境で確認します（動作確認の節を参照）。

  • Pi で直接コードを書いてしまう → 運用は『PCで書いて push → Pi で git pull』が基本（Pi5 セットアップガイド hw-pi5-setup と #8 に明文化済み）。
  • x と y を取り違える / 向きと位置を混同する → x,y は位置、theta は向き、と図を描いて整理する。
  • quaternion の順番ミス → ライブラリやメッセージで (x,y,z,w) の並びが決まっている。型定義で順番を必ず確認する。

動作確認のしかた

このIssueはロボ側(Python)なので npm run dev は使いません。代わりに次のように確認します。

  • 数値変換部分（theta → qz,qw）は ROS2 非依存・標準ライブラリ(math)だけで書けるので、手元PC（Windows・ROS2無し）で関数を呼んで値を出し、想定どおりか確認する（例: theta=0 なら qz=0, qw=1）。
  • 型を組み立てる部分（PoseStamped を作る）は ROS2 が必要なので、Pi または ROS2 環境で確認する。手元PCでは ModuleNotFoundError になるのが正常。
  • 設計指針: 『純粋な数値変換』と『PoseStamped 組み立て』を別関数に分けておくと、前者だけは手元PCで単体テストでき、後者は ROS2 環境で確認、と切り分けられて楽。

余裕があれば、Nav2 の Getting Started に沿って Gazebo シミュレータでゴールを1つ送ってみると、PoseStamped が正しい形か体感で確認できます。

用語ミニ辞典

言葉かんたんな意味
theta（ヨー角）平面上の向きを表す角度。ROS2 では必ずラジアン（REP-103）。theta=0 は無回転で、マップの x 軸の正方向を向くこと。
quaternion（クォータニオン）向きを4つの数(x,y,z,w)で表す方法。ROS2 の姿勢は内部でこれを使う。平面ヨーだけなら qz,qw の2つだけで足りる。
PoseStampedNav2 のゴールの入れ物。header(時刻 stamp ＋ 座標系名 frame_id、通常 "map")と pose(位置 position ＋ 向き orientation) の入れ子。frame_id 漏れだとゴールが無視される。
NavigateToPoseNav2 に『ここまで行って』と頼むアクション(時間のかかる依頼の仕組み)。ゴールとして PoseStamped を渡す。
stop_points席（停車ポイント）の座標表。列は id, label, x, y, theta。seat_id でこの行を引いて座標を得る。
frame_id座標がどの座標系のものかを示す名前。ロボのナビでは通常 "map"。
W5プロジェクト第5週。実機にマップを作らせて座標(x,y,theta)を確定させる工程。具体的な日付はスケジュール表を参照。
