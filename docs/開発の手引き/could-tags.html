▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴール: 本に自由なタグを付けられるようにし、本詳細でタグを表示、検索画面でタグによる絞り込みができるようにする（要件 C-02）。

タイトルや著者だけでは「機械学習の本」「卒研で使った本」みたいな探し方ができない。books.tags（text[] = 文字列の配列カラム）を使うと、1冊の本に複数のラベルを自由に付けられて、後から「このタグの本」とまとめて探せる。

スキーマ上は最初からタグ用の列が用意されている（supabase/schema.sql の books.tags text[] not null default '{}'）。だからDBの変更は不要で、アプリ側で『付ける・見せる・絞る』の3つを足すのがこのIssue。

COULD（余力で）

これは MUST/SHOULD の後にやる『余裕があれば』機能。完璧を目指さず、まず『1個のタグで絞り込めた』を最小ゴールにすると進めやすい。表記ゆれ対策などの凝った正規化は後回しでOK。

完了の定義（Done）

  • 本にタグを付けて保存できる（保存先は books.tags）→ 動作確認の手順1で確認
  • 本詳細（app/books/[id]/page.tsx）でその本のタグが表示される → 動作確認の手順2で確認
  • 検索画面（app/search/page.tsx）でタグを指定すると、そのタグを持つ本だけに絞り込める → 動作確認の手順3で確認
  • タグの前後の空白が除かれている（『 AI 』を入れても『AI』として保存される）→ 動作確認の手順4で確認
  • 空文字（空白だけ）のタグは保存されない → 動作確認の手順4で確認
  • 同じタグが重複して入らない（『AI』を2回足しても1個）→ 動作確認の手順4で確認
  • ×などでタグを1個外して保存すると、その本のタグだけが減る（本そのものは消えない）→ 動作確認の手順5で確認

進め方ステップ

前提（先に読む）

この機能は《本登録 S-3》の保存処理(insert)、または本を編集して保存する処理(update)が先に動いていることが前提。今の register / books/[id] / search はどれも静的な雛形で、まだ保存処理もフォームも無い（自分の目で各 page.tsx を開いて確認した上で着手すること）。保存処理がまだ無ければ、(A)本登録Issueの担当と連携して insert に乗せる か、(B)自分で『既存の本を編集して保存(update)』する小さい画面/処理から始める、のどちらかを選ぶ。

  1. 現状を自分の目で確認する
    app/register/page.tsx・app/books/[id]/page.tsx・app/search/page.tsx・app/page.tsx を開く。どれも今は文章だけの雛形。『どこに保存処理があるか』ではなく『どこに保存処理をこれから足すか』を決めるのが最初の一歩。
  2. Supabase に練習用のタグを直接入れて、表示から作る
    いきなり入力UIを作らず、まず Supabase の SQL Editor で既存の本に tags を直接入れてみる。text[] の配列リテラルは波カッコで書く（例: update books set tags = '{"AI","論文"}' where id = '対象のid'; ／ id は Table Editor で確認）。これで『DBに入ったタグをアプリで読んで表示する』だけに集中できる。なお SQL Editor は管理者権限で動くので RLS に関係なく入れられる（アプリ側の anon キー経由とは別）。
  3. 本詳細でタグを表示する
    app/books/[id]/page.tsx で、その本の tags（string[] で返ってくる）をループして表示する。まずは「読んで見せる」だけ。data fetching の現行の書き方は同梱docsで確認（下の調べるセクション参照）。
  4. タグ入力UIを作る（付ける）
    テキスト入力＋『追加』で配列に足し、各タグ横の×で外す『チップ入力』を作る。useState で『今の本のタグ配列』を持ち、追加=新しい配列を作って set、×で消す=その要素を除いた新しい配列を作って set。useState や onClick を使うコンポーネントには 'use client' が要る（理由は調べるセクション）。
  5. 保存する（insert か update を選ぶ）
    前提で選んだ方に乗せる。新規登録時は insert の値に tags を含める。既存の本の編集時は、その本の tags 列を『組み立てた新しい配列でまるごと上書き(update)』する。×で消した結果も『減った配列で上書き』。部分更新ではなく毎回まるごと差し替えが基本動作。
  6. 保存の直前に正規化する
    保存する配列を作るときに必ず通す処理: 各タグを trim（前後の空白除去）→ 空文字を除外 → 重複を除外。この3つだけまず確実に。表記ゆれ統一（小文字化など）は余力で。
  7. 検索画面でタグ絞り込みを足す
    app/search/page.tsx で、選んだタグを使って Supabase の select に絞り込み条件を付ける。1個のタグで絞れたら成功。複数タグは contains / overlaps の違いを理解してから（調べるセクション参照）。
表示対象の範囲（一覧について）

『一覧』はトップ app/page.tsx（S-1 本一覧）を指すが、これも今は雛形で他Issueの範囲。一覧側のタグ表示は対象外でOK。このIsseでは まず本詳細(app/books/[id]) でのタグ表示を確実にすれば Done を満たせる。一覧が実装済みなら余力で同じ表示を足す。

📚 進めながら調べる・学ぶこと

  • 配列カラム text[] にデータを入れる／更新する方法
    tags は普通の文字列カラムではなく配列。JS側では string[]（文字列の配列）を渡す。一番のつまずきは『タグの追加/削除のたびに、その本の現在のtags全体を新しい配列に組み立て直して、tags列をまるごと上書きupdateする』という考え方。1個だけ足す/消す部分更新ではない。これが分かると後述『タグを消したつもりが消えない』も防げる。supabase-js v2 では select した text[] はそのまま JS の string[] で返るので、表示・編集での型の不安は少ない。
    Supabase JS の update/insert: https://supabase.com/docs/reference/javascript/update と https://supabase.com/docs/reference/javascript/insert / 配列カラムの考え方は schema.sql の books.tags の定義も合わせて読む。
  • タグでの絞り込み: contains と overlaps の違い
    PostgREST/supabase-js には配列用の絞り込みがある。contains（cs / @>）は『指定したタグを“すべて”含む』(AND的)、overlaps（ov / &&）は『指定したタグの“どれか”を含む』(OR的)。重要: この差が出るのは複数タグを同時に指定したときだけで、タグ1個なら両者は同じ結果になる。まず1個で動かしてから複数の挙動を試すと混乱しない。
    Supabase JS フィルタ: https://supabase.com/docs/reference/javascript/select の Filtering（contains / overlaps の項）。
  • タグ入力UI（チップ入力）と 'use client'
    入力して配列に足す・×で消す、という操作は useState と onClick が必要。Next.js 16 では state やイベントを使うコンポーネントには 'use client' 宣言が要る。なぜサーバー/クライアントを分けるのか、どこに 'use client' を書くのかを現行の書き方で確認する（学習データと挙動が違う可能性があるので必ず同梱docsを見る）。
    node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-client.md（先頭の 01-app/ を忘れない。03-api-reference 単体のパスは存在しない）。
  • Next.js 16 でのデータ取得・更新の現行の書き方
    タグの表示＝本データの取得、保存＝更新。このNext.jsは16系で、取得/更新の書き方が学習データと違う可能性がある。答えのコードを写すのではなく、現行ガイドのパターンを読んで自分で組み立てる。
    取得: node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md / 更新: 07-mutating-data.md / DB接続は lib/supabaseClient.ts の supabase を import して使う。
  • タグの正規化（trim・空除外・重複除外、表記ゆれ）
    正規化は『保存する配列を作る瞬間』に効かせる。最小ラインは trim＋空文字除外＋重複除外の3つだけ。表記ゆれ統一（toLowerCase など）は、(1)英字にしか効かず日本語タグ（機械学習/論文 等）や全角半角ゆれ(AI/ＡＩ)には無力、(2)小文字化して“保存値そのもの”を変えると画面表示まで小文字になる副作用がある。やるなら『保存値を変えるのか、比較だけ小文字にするのか』を決めてから。迷ったら後回しでOK。
    JS の配列/文字列処理（Array の filter / new Set による重複除去 / String.trim）を MDN で確認。COULDなので最小ラインで割り切る。
  • 本一覧画面がどれかの確認と、本登録の保存処理の依存
    『一覧でタグ表示』が指す画面の特定と、タグ保存が乗る土台(insert/update)の有無を最初に確かめないと着手位置を間違える。app配下は page.tsx / register / search / books/[id] / history / login のみで、トップ app/page.tsx が S-1 本一覧、いずれも現状は雛形。これを踏まえ、本詳細(books/[id])の表示を主目標にし、保存は本登録Issueの担当と前提をすり合わせる。
    app/page.tsx（S-1 本一覧）・app/books/[id]/page.tsx・app/register/page.tsx・app/search/page.tsx を実際に開いて現状確認。要件は基本設計の F-02 / C-02、関連Issue『本登録(S-3)』。

つまずきポイントと対処

削除の操作を先に覚える

×で消す＝その本のtags配列から『その要素を除いた新しい配列』を作って update する（本そのものは消さない）。タグの追加も削除も毎回『新しい配列を作ってまるごと上書き』が基本。これは行の delete ではなく books 行の update（tags列の上書き）。そもそも schema.sql には delete ポリシーが1つも無く、アプリ(anonキー)からは誰も行を削除できない設定なので、タグを外しても本そのものが消える事故は起きない。下の表はその再確認。

症状・勘違い原因対処
タグを消したつもりが消えない配列の一部だけ消そうとしている／古い配列のまま update している消したいタグを除いた新しい配列を作り、その配列で tags 列をまるごと上書き update する
タグが文字列としてくっつく（例: "AI,論文"が1個に）配列ではなく1本の文字列として入れているtags は text[]。string[]（要素ごとに分かれた配列）で渡す
空白だけのタグや重複が保存される正規化を保存直前に通していない保存配列を作るときに trim→空文字除外→重複除外を必ず通す
onClick / useState でエラーになる'use client' が無いそのコンポーネント先頭に 'use client' を書く（node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-client.md で確認）
日本語タグの表記ゆれが揃わないtoLowerCase は英字専用で日本語/全角半角には効かないCOULDなので無理に揃えない。最小は trim＋重複除外。統一は余力で
Enterで同じタグが2回入る（日本語入力中）IME変換確定のEnterと追加のEnterが両方発火している重複除外があれば実害は減る。気になれば変換確定中のEnterを追加扱いしない工夫を（余力で）
select した tags の型が不安text[] が何で返るか分からないsupabase-js v2 では string[] でそのまま返る。配列としてループ表示してよい

動作確認のしかた

まず package.json の scripts を開いて dev スクリプトを確認してから起動する（このプロジェクトでは dev= next dev なので npm run dev でOK）。

  • 手順1: タグを付けて保存 → Supabase の Table Editor で対象の本の tags 列に配列として入っていること（Doneの1項目目に対応）
  • 手順2: 本詳細(/books/[id]) を開くと、その本のタグが並んで表示されること（Doneの2項目目に対応）
  • 手順3: 検索画面でタグを1個指定 → そのタグを持つ本だけに絞り込まれること（Doneの3項目目に対応）
  • 手順4: 『 AI 』『AI』『（空白だけ）』を入れて保存 → tags は『AI』1個だけ、空白タグは入っていないこと（Doneの4・5・6項目目に対応）
  • 手順5: タグを1個×で外して保存 → その本のタグだけが1個減り、本そのものや他の本は消えていないこと（Doneの7項目目に対応）
OKの目安

『1個のタグで本を絞り込めて、本詳細にタグが見えて、重複・空白・前後空白が入らない』までできていれば COULD として十分。
