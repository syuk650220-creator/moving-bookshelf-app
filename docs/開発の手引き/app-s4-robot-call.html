▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴールは、「この本を、この席へ呼ぶ」という操作を作ること。利用者が本と席を選んで「呼ぶ」を押すと、robot_calls テーブルに1行追加され（status='queued'）、ロボへの呼び出し依頼になる。さらに、今並んでいる呼出（queued/moving）の一覧も画面に出す（F-09）。

設計の肝（疎結合 / アプリ⇄ロボはDB経由）

アプリはロボと直接通信しない。アプリは robot_calls にinsert（行を足す）だけ、robot_status を読むだけ。ロボ側が robot_calls を購読して動き、robot_status を更新する。だから今回作るのは「robot_calls に1行入れる＋キューを読む」側だけでよい。

S-2（本詳細）は今スタブ。book_idは自分で用意して始める

本来この画面は S-2（app/books/[id]/page.tsx）から book_id を受け取って開く想定。でも S-2 は現状『担当者がこの画面を実装します』だけのスタブで、まだ book_id を渡せない。なので最初の動作確認では book_id を自分で用意して進める（やり方はステップ1）。S-2 からの受け渡し（URLクエリ/props）は別Issue・後回しでOK。

置き場所（最初の一歩）: まずは専用ページ app/robot-call/page.tsx を新規に作り、ここ単体で「席を選ぶ→呼ぶ→キューに出る」を完成させる。S-2 から開くモーダル/受け渡しは、動いてから後で繋げばよい。

完了の定義（Done）

  • app/robot-call/page.tsx（クライアントコンポーネント）を作り、画面が表示できている
  • 席（stop_points）を select してリスト/ドロップダウンで選べる（先にシードSQLで仮の席を入れておく・ステップ2参照）
  • 本（book_id）が決まっている（当面は Table Editor からコピーした実在の id を使う・ステップ1参照）
  • 本と席の両方が選ばれるまで「呼ぶ」ボタンは押せない（無効）／揃ったら押せる
  • 「呼ぶ」で robot_calls に1行 insert される（book_id, seat_id, status='queued', requested_by）
  • requested_by は今は仮の固定文字列（例 "guest" や自分の名前）でOK（ログイン実装後に置き換え）
  • 現在の呼出キュー（status が queued か moving）を created_at 昇順で一覧表示できる（F-09）
  • insert に成功したらキュー一覧に反映される（再 select するか Realtime で）
  • 失敗時（未選択や不正値）はエラーが画面で分かる（一度わざと失敗させて確認済み・動作確認参照）

進め方ステップ

  1. 1. まず book_id を1つ手に入れる（S-2待ちにしない）
    Supabase の Table Editor → books を開き、実在する本の id（uuid）を1つコピーする。books が空なら、まず適当な本を1行 insert（title と shelf_level は必須）。当面はこの id を、画面上のテキスト入力に貼るか、コードの定数として持っておく。これで book_id 問題は解決。S-2 からの受け渡しは後回し。
  2. 2. 先に席データを入れてから席を select する
    stop_points が空だと席一覧は当然カラ。まず『動作確認のしかた』のシードSQLを Supabase の SQL Editor で実行して仮の席（例: 1番席/2番席）を入れる。そのうえで lib/supabaseClient.ts の supabase を import し、stop_points を select（id, label を取得）して画面にドロップダウン/ボタンで出す。
  3. 3. 「選んだ席」を state で覚える
    React の useState で選択中の seat_id を保持する。ドロップダウンなら <select> の onChange、ボタン並びなら各ボタンの onClick で state を更新。本（book_id）と席（seat_id）が両方そろうまで「呼ぶ」ボタンを無効（disabled）にする。
  4. 4. 「呼ぶ」で robot_calls に insert
    ボタンを押したら robot_calls に insert する。入れる中身は { book_id, seat_id, status:'queued', requested_by }。requested_by は今は固定文字列（例 "guest" や自分の名前）でOK。status は省略してもDB側の既定で 'queued' が入るが、学習のため明示してよい。
  5. 5. 呼出キューを取得して表示（F-09）
    robot_calls を select し、status が queued または moving の行だけに絞り、created_at 昇順で並べて一覧表示する。insert が成功したら、この select をもう一度呼んで一覧を更新する（一番手堅い方法）。
  6. 6.（任意）Realtime で自動更新にする
    余裕があれば、再 select の代わりに Supabase Realtime で robot_calls を購読し、行が増えたら自動でキューを更新する。ただし事前設定が必要（『つまずき』『調べること』参照）。まずは5の手動再 select で完成させてから挑戦するのが安全。
  7. 7.（後で）S-2 と繋ぐ
    専用ページで動いたら、S-2（本詳細）から book_id を渡してこの操作を開く形に発展させる。S-2 が未実装の今は無理にやらず、別Issue/後回しでよい。

📚 進めながら調べる・学ぶこと

ここが核心

答えのコードを写すのではなく、下の『何を/なぜ/どこで』を自分でたどって書くこと。AIには答えでなく道案内を頼む（AIコーチの使い方参照）。

  • stop_points を select して席選択UIを作る
    席を選ばせるには、まずDBから席の一覧（id, label）を取ってくる必要がある。これがキュー insert の seat_id の元になる。
    Supabase JS の select 公式: https://supabase.com/docs/reference/javascript/select （.select('id,label') の書き方・戻り値の data/error）。座標やラベルの列名は supabase/schema.sql の stop_points（id, label, x, y, theta）で確認。
  • robot_calls に insert する（book_id, seat_id, status, requested_by）
    このIssューの本体。アプリ→ロボは『robot_calls に1行足す』だけで成立する（疎結合）。列名・値を間違えると入らない。
    Supabase JS の insert 公式: https://supabase.com/docs/reference/javascript/insert 。入れる列の正解は supabase/schema.sql の robot_calls（seat_id は int で stop_points.id、book_id は uuid で books.id、status は queued/moving/arrived/done/canceled のいずれか、requested_by は text・NULL可・FK無し）。
  • 選んだ席を state に保存し、本＋席が揃うまでボタンを無効にする
    React 未経験だと『クリックや選択でデータを覚える』『揃うまでボタンを押せなくする』のやり方で必ず詰まる。フォーム入力と state を繋ぐのが今回の山。
    useState: https://react.dev/learn/state-a-components-memory 。イベントで state を更新: https://react.dev/learn/responding-to-events 。<select> など制御コンポーネント入門: https://react.dev/learn/reacting-to-input-with-state 。disabled は React の <button> に disabled 属性を渡すだけ（条件は『本と席が両方ある』）。
  • キューを status で絞り created_at 昇順で取得（F-09）
    『今並んでいる呼出』だけ見せたい＝queued/moving に絞り、古い順に並べる。絞り込みと並べ替えの書き方を知る必要がある。
    Supabase JS のフィルタ（.in('status',[...])）と並べ替え（.order('created_at', { ascending: true })）: https://supabase.com/docs/reference/javascript/select と https://supabase.com/docs/reference/javascript/order 。スキーマの idx_calls_queue も created_at 順なので昇順で正しい。
  • データ更新時の今どきの作法（mutating data）を知る
    Next.js 16 では『データを変える操作』の流儀（Server Actions など）が学習データと違う可能性。今の正解の形を一度見ておく。
    node_modules/next/dist/docs/ の 01-app/.../07-mutating-data.md を開く。ただし: このIssューは学習目的で、まずクライアント側（'use client' + supabase JS で insert/再select）で完結させてよい。07-mutating-data.md は Server Actions 中心だが『今の流儀を知る』ための参考であって、必ず Server Actions に置き換える必要はない。どちらでやるか迷ったら、まずクライアント側で動かす。
  • 'use client'（クライアントコンポーネント）
    useState・onClick・onChange・ブラウザでの supabase 呼び出しはクライアント側でしか動かない。ファイル先頭の宣言が要る。
    node_modules/next/dist/docs/ の 03-api-reference/01-directives/use-client.md と 01-app/.../05-server-and-client-components.md 。
  • （任意）Realtime で呼出キューを自動更新
    他の人が呼んだ呼出も、リロードせず一覧に出せると気持ちいい。F-09 の発展。
    Supabase Realtime: https://supabase.com/docs/guides/realtime 。重要な前提: 対象テーブル robot_calls を supabase_realtime パブリケーションに追加しないとイベントが一切飛ばない（Dashboard → Database → Replication で robot_calls を有効化）。コードを書くだけでは届かない、最頻出のつまずき。

つまずきポイントと対処

症状・勘違い原因と対処
insert で book_id を渡せず詰まるS-2 がスタブで book_id をくれないのが原因。最初は Table Editor の books から実在 id（uuid）をコピーして使う（ステップ1）。S-2 連携は別Issue・後回しでよい。
席一覧が空っぽstop_points にデータが無いだけ。先に『動作確認』のシードSQLを SQL Editor で実行してから select する（順番が大事・ステップ2）。
robot_status や stop_points に insert しようとしてエラーこの2つには insert ポリシーが無く、アプリ（匿名キー）からは追加できない仕様。アプリが insert するのは robot_calls だけ。
でもシードSQLでは stop_points に入った？矛盾では？矛盾ではない。SQL Editor からの insert は管理者権限で実行されRLSを通り抜けて入る。アプリ（匿名キー）からは RLS に弾かれて入らない、という違い。
insert は通った気がするのに data が空/エラーが分からないsupabase の戻り値は { data, error }。error を必ず受けて画面に出す。何も表示しないと成功も失敗も分からない。
Realtime を購読したのに何も来ないrobot_calls が supabase_realtime パブリケーションに未追加なのが定番原因。Dashboard → Database → Replication で robot_calls を有効化する。コードだけでは届かない。
onChange/onClick が動かない・useState が使えないファイル先頭に 'use client' が無い／サーバーコンポーネントのまま書いている。クライアント宣言を確認。

動作確認のしかた

先に席の仮データを入れる。Supabase の SQL Editor で次を実行（席は本来ハード/結合側が決めるが、開発中の仮データとして自分で入れてよい）:

-- 開発用：仮の席（停車ポイント）を入れる
insert into stop_points (id, label, x, y, theta) values
  (1, '1番席', 0.0, 0.0, 0.0),
  (2, '2番席', 1.5, 0.0, 0.0),
  (3, '窓側席', 1.5, 2.0, 0.0)
on conflict (id) do nothing;
  • ページを開くと席のドロップダウン/ボタンに『1番席/2番席/窓側席』が出る
  • 本未選択 or 席未選択のとき「呼ぶ」が押せない（disabled）。両方そろうと押せる
  • 「呼ぶ」を押したあと、Supabase の Table Editor → robot_calls に1行増えている（book_id, seat_id, status=queued, requested_by が入っている）
  • 画面のキュー一覧（queued/moving）にその呼出が出る（古い順）
  • 【失敗も一度確認】わざと未選択のまま、または存在しない seat_id（例 999）で呼んでみて、エラー表示がちゃんと出ることを確認する（999 は stop_points に無いので外部キー違反になる）
  • 【任意・Realtime】robot_calls の Replication を有効化したうえで、Table Editor から手動 insert した行が、リロードなしで一覧に出れば成功
失敗パスも必ず一度試す

成功した時しか試さないと『エラー表示が本当に出るか』が未検証のまま Done にしがち。わざと失敗させてエラーが画面で分かることを1回確認しておく。

用語ミニ辞典

言葉かんたんな意味
robot_callsロボへの呼び出し依頼を1件1行で貯めるテーブル。アプリはここに insert する。
seat_id呼ぶ先の席。stop_points.id（int）を指す。robot_calls の列名は seat_id。
book_id呼ぶ本。books.id（uuid）を指す。
status（呼出）呼出の進行状態。queued→moving→arrived→done（または canceled）。新規は queued。
queue（キュー）順番待ちの列。ここでは status が queued/moving の呼出たち＝『今並んでいる呼出』。
queued並んだばかり・まだ動いていない状態。insert 時の初期値。
stop_points席（停車ポイント）の座標表。アプリは select するだけ（insert 不可）。
requested_by誰が呼んだか（text・NULL可）。今は仮文字列でOK、ログイン後に置き換え。
疎結合アプリとロボが直接つながらず、DBのテーブル越しに間接連携する設計。
RLS行レベルセキュリティ。テーブルごとに誰が select/insert できるかを決める仕組み。
Realtime（任意）DBの変化をリロードなしで受け取る仕組み。対象テーブルの有効化が前提。
