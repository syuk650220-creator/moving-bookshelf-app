コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴールは、5スロット（5段）の本棚部を作って、各段に NeoPixel を1個ずつ組み込み、「5段に本が入る」「各段のLEDを段ごとに別々に光らせられる」状態を作ること。これは要件 F-10（利用者が呼んだ本の段を光らせて教える） の物理的な土台になる。

このIssueの守備範囲はハード（機構＋配線＋組込）まで。「アプリの呼出に連動して光らせる本番ロジック」は別Issue（ソフト側のLED点灯Issue）。ただし後述のとおり、動作確認のために『各段を順に光らせるだけの使い捨てスクリプト』は自分で用意してよい（製品ロジックではないので答えを写すのではなく、選んだライブラリのサンプルを参考に自作する）。

最初に必ず読む：Raspberry Pi 5 は『定番ライブラリがそのまま動かない』

本機の頭脳は Raspberry Pi 5。Pi 5 は新I/Oチップ（RP1）を採用したため、従来Piで定番だった rpi_ws281x / Adafruit_NeoPixel（PWM＋DMA方式）が そのままでは動かず ws2811_init failed with code -3 (Hardware revision is not supported) で失敗するのが2025年時点でも知られた既知問題。配線も対応表も正しいのにソフトで必ず壁にぶつかる。だから『まず Adafruit の旧チュートリアル通りに』進めると詰む。Pi5対応の手段（SPI経由のドライバ等）を最初に確認すること（→ 進め方ステップ／研究セクション）。

設計の肝（疎結合 / ハード→ソフトの受け渡し）

「どの段（shelf_level 1〜5）が、どのLED番号（インデックス 0〜4）で、どのGPIO/データ線につながっているか」という対応表が、このハードIssueからソフト側へ渡す唯一の成果物。これがズレると『2段目を呼んだのに4段目が光る』事故になる。対応表は口頭でなく文書＋PRで残すのがこのIssueの肝。

完了の定義（Done）

  • 5段（5スロット）の本棚部ができ、各段に研究室で使う想定の本が物理的に入る（段の高さ・奥行き・幅を実測した本サイズで決めた）
  • 各段に NeoPixel を1個ずつ（計5個）固定して組み込んだ
  • 5個を電源(+5V)・データ・GND の3線で数珠つなぎ（DOUT→次のDIN）に配線した
  • 【電源】NeoPixel は Pi の5Vピンからではなく外部5V電源/バッテリーから給電し、GNDだけ Pi と共通にした。外部電源の容量がLED最大電流（5個ぶん：自分で計算）を上回っていることを確認した
  • 各段のLEDを 1段ずつ個別に点灯できることを、確認用の使い捨てスクリプトで実機確認した（Pi5で光らせられない事情がある場合は『回避ルート』参照）
  • 【受け渡し】shelf_level(1〜5) → LEDインデックス(0〜4) → GPIO/データ線 の対応表を robot/ 配下の固定ファイル（例 robot/led_mapping.md）に表で書き、PRに含めてPM（水谷）レビューを通した
  • 配線・LEDが走行や振動で外れないよう取り回しを固定した（線が機構や車輪に干渉しない）
Doneの考え方

ハード（5段＋LED5個＋配線＋電源＋対応表）が揃えばこのIssueは完了。『アプリの呼出と連動』は別Issueなので、ここではできなくてOK。逆に対応表をPRで残すところまでがこのIssueの責任範囲。

進め方ステップ

  1. 本のサイズを実測して、1段の寸法を決める
    ロボに載せる想定の本（研究室の本）の高さ・幅・厚みを何冊か実測し、最大級に合わせて1段の内寸（高さ・奥行き・幅）を決める。5冊ぶんで全体の高さ・重さがどれくらいになるかもメモ（移動ロボなので重心・重量が後で効く）。
  2. 5段構造を仮組みする（接着の前に必ず仮）
    段板や仕切りを仮で組んで、実際に本を5段それぞれ出し入れできるか確認する。本は出し入れする運用（接着しない）。LEDを入れる位置（各段のどこに付けると本に隠れず見えるか）もこの段階で目星をつける。
  3. 【Pi5の壁を先に確認】NeoPixelをPi5でどう光らせるか調べて方式を決める
    配線する前に、Pi 5でNeoPixelを光らせる現行手段を確定させる（→研究セクション）。定番のrpi_ws281x系はPi5で動かないことが多いので、SPI経由のドライバ等を選ぶ。ここを後回しにすると、組み上げたあとに『光らせられない』で全部やり直しになる。ソフト側Issueと並行で詰めるのが安全。
  4. 【最重要・電源をどこから取るか決める】
    NeoPixel は Piの5Vピンからではなく外部5V電源/バッテリーから給電し、GNDだけPiと共通にするのが基本（→用語『電流容量』）。理由：5個を全白点灯すると 1個あたり最大約60mA×5＝最大約300mA級のピーク電流が流れ、Piの5Vピンから取ると不足・不安定・最悪Pi破損の恐れ。本機はバッテリー駆動の移動ロボなので外部給電が前提。5個ぶんの最大電流を自分で計算し（1個あたり最大電流×個数）、用意する電源/バッテリーの容量がそれを上回るか確認する（→研究『電流計算と電源容量』）。
  5. 1個だけで点灯テスト（机上）
    いきなり5個つながず、まずNeoPixel1個を選んだ方式・電源・データ線でつなぎ、確認用スクリプトで光らせる。ここでロジックレベル（3.3V→5V）の問題が出ることがある（→つまずき）。1個が安定して光ってから次へ。
  6. 5個を数珠つなぎにして全段テスト
    1個目のDOUT→2個目のDIN…と順に5個つなぎ、各個に+5VとGNDも配る。確認用スクリプトで0番→1番→…→4番を順に単独点灯させ、どのインデックスがどの物理段かを記録する。これがそのまま対応表の元になる。
  7. テストが通ったら各段にLEDを固定して配線を取り回す
    各段の決めた位置にNeoPixelを固定し、電源/データ/GNDの3線を機構・車輪に干渉しないよう束ねて固定する。走行の振動で抜けないようコネクタ/結束を確実に。最初のLED前にデータ線へ直列抵抗（信号保護の定番、目安数百Ω）、電源にコンデンサ（突入電流対策の定番）を入れる是非も調べる。
  8. 対応表を作ってPRでPM承認を得る
    shelf_level(1〜5) → LEDインデックス(0〜4) → GPIO/データ線（とSPIならどのSPIか） を表にして、robot/ 配下の固定ファイル（例 robot/led_mapping.md）に書き、PRに含めてPM（水谷）のレビューを通す。口頭合意で終わらせない（食い違いの温床）。PCで書いて push → Pi で git pull の流れ。

📚 進めながら調べる・学ぶこと

  • Raspberry Pi 5（RP1チップ）でのNeoPixel/WS2812 制御の現行手段
    このIssue最大の落とし穴。Pi 5 は新I/Oチップ(RP1)のため、従来定番の rpi_ws281x / Adafruit_NeoPixel（PWM+DMA方式）が ws2811_init failed with code -3 で動かない。これを知らずにAdafruitの旧チュートリアル通り進めると、配線が正しくても必ず詰む。SPI経由のドライバ等、Pi5で動く方式を最初に選ぶ必要がある。
    SPI経由でPi5対応のライブラリ Pi5Neo: https://github.com/vanshksingh/Pi5Neo 。Raspberry Pi 公式フォーラムで『Raspberry Pi 5 neopixel ws2811_init failed code -3』を検索: https://forums.raspberrypi.com/ 。※CLAUDE.md記載の Adafruit『NeoPixels on Raspberry Pi』(https://learn.adafruit.com/neopixels-on-raspberry-pi) は旧Pi向けでPi5の壁に触れていないので、配線の参考にはなるが点灯方式はPi5用を別途調べること。
  • NeoPixel 5個ぶんのピーク電流の見積りと外部5V電源/バッテリーの容量選定
    Piの5Vピンから給電すると容量不足で不安定・Pi破損の恐れ。1個あたり満白点灯で約60mA、5個で最大約300mAが目安。本機はバッテリー駆動の移動ロボなので外部5V電源/バッテリーから給電し、GNDだけPiと共通にするのが定番。用意する電源容量がこの最大電流を上回るか自分で計算して確認する。
    Adafruit NeoPixel Überguide『Best Practices』（電源・電流の節）: https://learn.adafruit.com/adafruit-neopixel-uberguide/best-practices 。1個あたりの最大電流と、電源を別系統にしてGNDだけ共通にする理由を確認する。
  • Pi GPIO 3.3V ロジックと WS2812 5V DIN のレベルシフト（電圧変換）の要否
    PiのGPIOは3.3Vロジック、WS2812のデータ入力(DIN)は約5Vロジックを期待するため、直結だとDINの閾値に届かず『最初のLEDだけ色化け』『全体がチラつく』『光らない』という超頻出症状になる。配線も向きも合っているのに色がおかしい、の主因がこれ。レベルシフタIC、または給電・配線方法での回避を調べる。
    Adafruit NeoPixel Überguide のロジックレベル/Best Practices の節: https://learn.adafruit.com/adafruit-neopixel-uberguide/best-practices （『Logic Level』『3.3V→5V』の記述）。選んだPi5用ライブラリのREADMEにもレベルシフトの注意があるか確認。
  • shelf_level(1〜5) と LEDインデックス(0〜4) の対応設計（オフバイワン）
    DBの books.shelf_level は 1始まり（1〜5、schema.sqlでcheck制約）、NeoPixelのインデックスは 0始まり（0〜4）。この1ズレ（オフバイワン）を明文化しないと『2段目を呼んだのに3段目が光る』事故になる。さらに物理的な段の上下とインデックスの向きも決めておく必要がある。
    列定義の正は supabase/schema.sql の books.shelf_level int not null check (shelf_level between 1 and 5)。LED側のインデックスは選んだライブラリのサンプル（配列の何番目か）で確認。対応のルールを自分で表に落とす。
  • 対応表をどこに・どの形式で残し、PRでPM承認を得るかの共有運用
    robot/README.md には『共有テーブルの形はPM(水谷)が決める』とあり、置き場所が曖昧だと口頭合意で終わり食い違う。文書＋PRで残して初めて『共有した』ことになる。
    robot/README.md（説明文）は読むだけにして、対応表は robot/ 配下に新規ファイル（例 robot/led_mapping.md）を作って表で記載 → PRに含めてPMレビュー。開発フローは CLAUDE.md／AGENTS.md（PCで書いてpush→Piでgit pull、main直push禁止・PR・PM承認後Squash）。
  • NeoPixelの3線（電源/データ/GND）配線と数珠つなぎ、保護部品（抵抗・コンデンサ）
    5個を安定して光らせる物理配線の基本。DOUT→次のDINで数珠つなぎし、各個に+5VとGNDを配る。データ線の直列抵抗（信号反射対策の定番）、電源のコンデンサ（突入電流のならし）を入れる是非を判断するため。
    Adafruit NeoPixel Überguide『Best Practices』: https://learn.adafruit.com/adafruit-neopixel-uberguide/best-practices （直列抵抗・コンデンサ・配線長の節）。Raspberry Pi のGPIO配置: https://www.raspberrypi.com/documentation/computers/raspberry-pi.html 。
  • 確認用の使い捨て最小点灯スクリプトの作り方（製品ロジックではない）
    ハードが正しいか確かめるには『各LEDを順に点けるだけ』の最小コードが要る。本番の点灯ロジックは別Issueだが、確認手段は自分で持つ必要がある。答えを写すのではなく、選んだライブラリのサンプルを参考に自作してよい。
    選定したPi5対応ライブラリの examples/（例：Pi5Neo の examples）。supabase等は使わず、ローカルで各インデックスを順に点灯するだけの短いスクリプトでよい。

つまずきポイントと対処（安全注意つき）

症状・落とし穴原因・対処
ws2811_init failed with code -3 (Hardware revision is not supported)Pi5最大のハマり。定番 rpi_ws281x/Adafruit_NeoPixel（PWM+DMA）はPi5(RP1)非対応。SPI経由のPi5Neo等、Pi5対応の方式に切り替える（→研究1）。配線は悪くないので作り直さない。
配線も向きも合っているのに最初のLEDだけ色化け／全体がチラつくPiの3.3Vデータ信号がWS2812 DINの5V閾値に届かないロジックレベル不足の可能性。レベルシフタ(3.3V→5V)や給電・配線の見直しを調べる（→研究3）。
LEDが暗い／不安定／たまに消える、Piが落ちる電流容量不足。Piの5Vピンから給電していないか確認。外部5V電源/バッテリーから給電しGNDだけ共通に。5個ぶん最大電流(目安~300mA)を計算し容量が足りる電源を使う（→研究2）。
2段目を呼んだのに別の段が光るshelf_level(1始まり)とLEDインデックス(0始まり)のオフバイワン。対応表で変換ルールと段の向きを明文化（→研究4）。
対応表が伝わらない／後で食い違う口頭合意のまま。robot/ 配下のファイル（例 led_mapping.md）に表で書き、PRでPM承認まで取る（→研究5）。
走行したら配線が抜けた／LEDが取れた固定不足。振動前提で結束・コネクタを確実に。線が車輪や機構に干渉しない取り回しに。
【安全】配線中・通電中の注意通電したまま配線を抜き差ししない（ショートでLED/Pi破損）。+5VとGNDを逆/短絡させない。半田付けは換気・やけど注意。バッテリー使用時は端子の短絡に特に注意（発熱・発火の恐れ）。

動作確認のしかた

前提：ハード確認には最小の点灯コードが要る。製品ロジックは別Issueだが、各LEDを順に点灯するだけの使い捨てスクリプトは、答えを写すのではなく選定したPi5対応ライブラリのサンプル（examples）を参考に自分で用意してよい。

  1. 5段それぞれに対象の本を実際に入れて、無理なく出し入れできることを確認する。
  2. 外部5V電源/バッテリーから給電し、GNDがPiと共通になっていることを目視・テスタで確認する。
  3. 確認用スクリプトでLEDインデックス 0→1→2→3→4 を1個ずつ単独点灯させ、どのインデックスがどの物理段か（上から/下から）を記録する。
  4. 全白点灯（5個同時に最大輝度）させても暗くならない・チラつかない・Piが落ちないことを確認する（電源容量がOKの証拠）。
  5. 色化け/チラつきが出たらロジックレベル（3.3V→5V）と給電を見直す。
  6. 対応表（shelf_level→インデックス→GPIO/データ線）とこの実測結果が一致していることを確認し、ファイルに反映してPRを出す。
完成判定

「5段に本が入る」かつ「各段のLEDを1段ずつ個別に光らせられる」かつ「外部電源で5個同時点灯しても安定」かつ「対応表をPRでPM承認」まで来たらこのIssueは完了。Pi5でどうしても光らせられない事情がある場合は次の『回避ルート』参照。

回避ルート（Pi5で詰まったら）

ハード単体だけ先に保証する

Pi5でのドライバ選定が長引いても、ハードの完成（機構・配線・電源）は止めない。まず旧Pi / Arduino / 開発PC など『確実に点く環境』でNeoPixel5個が個別点灯することをハード単体として保証し、Pi5でのドライバ選定はソフト側Issueと並行で詰める。こうすれば『配線が悪いのか、ドライバが悪いのか』の切り分けもできる。

LEDインデックスNeoPixel配列の何番目か（0〜4、0始まり）。shelf_levelと1ズレるので対応表で変換する。
オフバイワン1始まりと0始まりのズレなど『1だけずれる』定番バグ。対応表で明文化して防ぐ。
ロジックレベル信号のHIGH/LOWを表す電圧。Piは3.3V、NeoPixelのDINは約5Vを期待するので変換が要ることがある。
レベルシフト信号電圧を変換すること（ここでは3.3V→5V）。足りないと色化け/チラつきの原因になる。
電流容量電源が供給できる電流の上限。足りないとLEDが暗い・不安定、Piが落ちる。NeoPixel5個で最大~300mAが目安。
ピーク電流全白最大輝度など、最も電流が流れる瞬間の値。電源容量はこのピークを上回らせる。
RP1Raspberry Pi 5 の新I/Oチップ。これが原因で従来のNeoPixelライブラリがそのまま動かない。
直列抵抗 / コンデンサデータ線の保護（信号の反射対策・数百Ω目安）／電源の突入電流をならす定番部品。入れる是非を調べる。
