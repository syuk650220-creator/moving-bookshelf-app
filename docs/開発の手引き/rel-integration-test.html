このIssueは何？

ゴール：アプリとロボを実際につなぎ、本を選ぶ → 席へ呼ぶ → ロボが自走 → 到着で LED → 借りる → 返す の一連の流れ（フルフロー）を最初から最後まで一気に通す。これが結合テスト（=部品が単体で動くだけでなく、つなぎ目まで含めて全体が動くかを見るテスト）です。

なぜ必要か：このプロジェクトではアプリとロボは直接通信せず、Supabase のテーブル（robot_calls / robot_status）を介して 疎結合（=お互いを直接知らず、間にあるデータだけで連携する） でつながります。各班がバラバラに作ったものを最後に合流させるので、テーブルというつなぎ目で値が正しく受け渡されるかは、通してみないと分かりません。ここが本プロジェクトの山場で、全員参加です。

境界（テストで一番見る場所）

アプリとロボの境界はちょうど Supabase のテーブルです。だから不具合の切り分けも『アプリが robot_calls に正しく書けたか』『ブリッジがそれを読んで robot_status を正しく更新したか』をテーブルの中身を見て判断します。テーブルが共通言語です。

完了の定義（Done）

  • 前提Issue（ブリッジ・LED制御・席座標→Nav2ゴール変換）が全て完了し、ロボ側が起動できる状態であることを確認した
  • 開始状態がそろっている：books に available の本、stop_points に席が1件以上、robot_status は id=1 が state=idle
  • アプリで本を選んで席へ呼ぶと robot_calls に1行 insert される（status=queued）
  • ブリッジがそれを拾い、robot_calls.status が moving に、robot_status.state が moving に進む
  • ロボが該当の席まで自走して止まる
  • 到着で robot_calls.status=arrived、robot_status.state=arrived になり、LED が点灯する
  • アプリで借りる→books.status=on_loan、loans に1行（借りた人＝user か guest の区別つき）
  • アプリで返す→books.status=available、loans の returned_at と returned_by が入る
  • 一連の終了で robot_calls.status が done に、robot_status.state が idle に戻る（誰がいつ更新するかを確認済み）
  • 通している間に見つけた不具合・気づきを GitHub Issue に起票した
done の責務をはっきりさせる

robot_calls.status の queued→moving→arrived→done への前進は『アプリ』ではなく『ロボ側（ブリッジ）』が更新する想定です（RLS 上は誰でも update 可なので、決めておかないと曖昧になる）。正常系では到着・貸出処理が終わった後に done へ落ちます。もし本フローで done を使わず arrived のまま終わらせる設計にするなら、それを Step0 でチームで決めて1行メモし、Done の最後の項目を『その決定どおりに終端状態になっている』に読み替えてください。enum（5値）と実運用がズレたまま進めないことが大事です。

進め方ステップ

  1. Step0：前提が揃っているかゲート確認（ここを通らなければ着手しない）
    本Issueは『各部品が出来てから』の最終合流です。(1) ブリッジ（robot_calls を Realtime 購読→Nav2 ゴール変換）、(2) LED 制御、(3) 席座標→Nav2 ゴール変換、(4) アプリの呼出/貸出/返却 がそれぞれ完了していることを確認します。robot/ はまだプレースホルダなので、ブリッジの『起動コマンド・ログの見方』はハード制御班の手順書か該当Issueから入手してください（本書には載っていません）。あわせて『status を done まで進めるか arrived で止めるか』『誰が status を進めるか（ブリッジ）』をチームで一度決めてメモします。
  2. Step1：開始状態をそろえる（シード＆リセット）
    テストは毎回同じスタート地点から始めないと再現できません。Supabase の SQL Editor（service_role 権限で動く）を使い、books に available の本を入れ、stop_points に席を1件以上入れ、robot_status の id=1 を state=idle に update します。stop_points / users / robot_status はアプリ（anon キー）からは insert できない（RLS に insert ポリシーが無い）ので、必ず SQL Editor で行います。books は insert 可。座標 x/y/theta は実機の地図に対応する実測値なので、ハード制御班・席座標Issュー担当に確認して埋めます。
  3. Step2：アプリ側だけで『呼ぶ』を確認
    ブラウザでアプリを開き、本を選んで席へ呼びます。Supabase の Table Editor で robot_calls を開き、新しい行が status=queued / book_id・seat_id が正しい値で入っているかを目視します。ここで増えなければアプリ↔Supabase の段で詰まっています（切り分け早見表参照）。
  4. Step3：ブリッジが拾うかを確認（Realtime のつなぎ目）
    ロボ側でブリッジを起動した端末の出力（ログ）を見ます。robot_calls の新規行を受信したログが出るか、robot_calls.status が moving に、robot_status.state が moving に変わるかを Table Editor で確認します。受信しない場合は Realtime がそのテーブルで有効化されているか（後述）を疑います。
  5. Step4：ロボが自走→到着→LED を確認（まず低速・短距離）
    実機を動かします。最初は速度を落とし、短い距離で席まで行けるか個別に確認します。到着したら robot_calls.status=arrived・robot_status.state=arrived・LED 点灯がそろうかを見ます。実機を動かす前に必ず安全確認（下の callout）を読みます。
  6. Step5：借りる→返すを確認
    到着後、アプリで『借りる』を実行。借りる人を user（ログイン利用者）か guest（その場の名前入力）で出し分け、books.status=on_loan、loans に borrower_type と user_id または guest_name が入るかを見ます。次に『返す』を実行し、books.status=available、loans.returned_at・returned_by が入るかを確認します。
  7. Step6：本番距離で一気通貫
    Step2〜5 を低速・短距離で個別に通せたことを確認してから、最後に本番の距離・配置で Step1（リセット）→2→3→4→5 を止めずに通します。終了時に robot_calls.status と robot_status.state が決めた終端状態（done / idle 等）になっているかを最終確認します。
  8. Step7：異常系を1〜2本だけ通す
    つなぎ目の弱さは異常系で出ます。例『存在しない席IDで呼ぶ（FK違反になるか）』『到着前にもう一度呼ぶ（2件目が queued で積まれるか／弾かれるか）』を試し、期待どおりかをメモします。期待値はキュー設計（後述 research）から読み解きます。
  9. Step8：見つけた不具合を Issue 化
    再現手順・どの段で・どのテーブルの何がおかしかったか・期待と実際を1件ずつ Issue にします。番号はまだ振られていないので、画面名（S-◯）や要件、テーブル名で具体的に書きます。
実機を動かす前の安全確認（Step4・Step6）

(1) 緊急停止の手段を全員が事前に把握する：この機体の主電源スイッチの位置、ソフト停止の方法（ros2 のトピック/サービス等）。具体の停止操作はハード制御班に確認し、ホワイトボード等に書き出してから開始。(2) 非常停止役を1人立て、その人は手を空けておく。(3) 初回は必ず低速・短距離。一気通貫は『個別通過を確認してから本番距離』の順で、いきなり全速・長距離はしない。(4) バッテリー残量を開始前に確認（最低ライン・確認手段はハード制御班に確認）。残量不足のまま自走させない。(5) 走行経路に人や物を置かない。

📚 進めながら調べる・学ぶこと

  • 結合テストの観点（境界＝Supabase の各テーブル）
    単体では動く部品も、つなぎ目で値が欠ける・型が違う・タイミングがズレることで落ちる。どこを見れば『つながった』と言えるかを理解するため。
    状態遷移と enum の唯一の原典は supabase/schema.sql（基本設計書.md・要件定義書.md は現状プレースホルダで中身が無いため、整備されるまでは schema.sql を見る）。schema.sql の CHECK 制約 status in (queued,moving,arrived,done,canceled) と state in (idle,moving,arrived,returning) から遷移を読み取る。robot/README.md でつなぎ目（疎結合）の考え方を確認。
  • 各段階での確認方法（robot_calls / robot_status を見る）
    テーブルの中身が各段のゴール判定そのもの。目視で『今どの段か』を読めると切り分けが速い。
    Supabase ダッシュボードの Table Editor で robot_calls / robot_status / books / loans の行を直接見るのが主手段。クエリで確認したいときは SQL Editor。注意：リポジトリの scripts/check-supabase.mjs は『手早い疎通確認』に使えそうだが、現状そのままでは動かない（下のつまずきポイント参照）。まずは Table Editor を主に使う。
  • Realtime が該当テーブルで有効化されているか
    ブリッジは robot_calls を Realtime 購読する設計。Supabase 側で publication にテーブルが追加されていないと、insert しても通知が飛ばず Step3 で必ず落ちる。
    Supabase の Realtime の考え方は https://supabase.com/docs/guides/realtime 。このプロジェクトで robot_calls / robot_status が Realtime 有効か（どこで確認・有効化するか）はダッシュボードの Database → Replication / Publications を確認し、未有効なら PM に相談。
  • 不具合の切り分け（アプリ / ブリッジ / ロボ のどこか）
    全体が動かないとき、原因を3つの層のどこかに絞れると修正が速い。境界のテーブルを見れば層が特定できる。
    下の『切り分け早見表』。アプリ層はブラウザ F12 の Console / Network、ブリッジ層はロボ側で起動した端末のログ（起動コマンドはハード制御班の手順書）、ロボ層は ROS2 / Nav2 のログ（https://docs.nav2.org/）。
  • ログの見方（ブラウザ Console / Network）
    insert が失敗したとき、原因はネットワークのレスポンスに出ている。どのリクエストを見るかを知らないと大量のログに埋もれる。
    ブラウザ F12 → Network タブで robot_calls への POST（/rest/v1/robot_calls）を探し、ステータスコード（201=成功 / 4xx=失敗）とレスポンス本文のエラーメッセージを読む。Console タブに supabase の error が出ていないかも見る。Supabase JS の挙動は https://supabase.com/docs/reference/javascript/select 等。
  • 借りる人の出し分け（user / guest）と返却者の記録
    結合フローの『借りる→返す』では loans に誰が借り・誰が返したかを正しく入れる必要があり、これも結合テストの確認項目。
    schema.sql の loans 定義：borrower_type in (user,guest)、user_id（user のとき）、guest_name（guest のとき）、returned_at / returned_by。出し分けの確認はアプリの貸出/返却画面と Table Editor の loans 行を突き合わせる。
  • シードに必要な最小データ（特に stop_points の座標）
    robot_calls.seat_id は stop_points.id への外部キー。席が無いと呼出 insert が FK 違反で落ち、Nav2 ゴールも作れない。座標は実機の地図に対応する実測値が要る。
    schema.sql の stop_points（id, label, x, y, theta）と robot_calls の references。座標値と『誰が用意するか』は席座標→Nav2ゴール変換Issue の担当・ハード制御班に確認（依存関係あり）。
  • 同時に2回呼んだときの期待挙動（キュー設計）
    異常系（Step7）の期待値を実装の勘でなく設計から決めるため。robot_status は1行固定で current_call_id は1件しか指せない。
    schema.sql の idx_calls_queue（create index ... where status='queued'）は queued の複数行を想定＝FIFO キュー設計。robot_status は id=1 固定の1行。これらから『2件目は queued で積まれて待つのが自然』と読み取り、実際の挙動と突き合わせる。最終的にどう振る舞うかはブリッジ実装次第なので観測してメモ。
  • テストシナリオの作り方
    場当たりに触ると再現できない。前提→操作→期待結果を1本ずつ書くと、誰が見ても同じ判定ができる。
    正常系1本＋異常系1〜2本を表にする（下の段階表が正常系の雛形）。各行に『前提状態 / 操作 / 期待するテーブルの値』を書く。原典は schema.sql の enum と FK。
  • 見つけた不具合の Issue 化
    通しながら出た問題を口頭で流すと消える。再現手順付きで残すと担当が直せる。
    GitHub Issues。テンプレ（.github）に沿って、再現手順・どの段・どのテーブルの何がおかしいか・期待と実際を書く。番号は未付与なので画面名(S-◯)・タスク名・テーブル名で示す。

つまずきポイントと対処

scripts/check-supabase.mjs は現状そのままでは動かない

このスクリプトは疎通確認に便利そうだが、中身が古いキー名 NEXT_PUBLIC_SUPABASE_ANON_KEY を読んでいる（15行目 const key = env.NEXT_PUBLIC_SUPABASE_ANON_KEY）。実際の .env.local は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY しか持たない（ANON_KEY は誤り＝AGENTS.md の大原則）。そのまま実行すると『KEY:(なし)』で exit 1 となり、テーブル行数の表示まで到達しない。使うなら先にこのファイルのキー名を PUBLISHABLE_KEY に直す（別タスクとして起票推奨）。当面の疎通確認は Supabase の Table Editor で行数を見るのを主手段にする。

設計書を開いても中身が無い

docs/基本設計書.md と docs/要件定義書.md は現状『後で貼る』のプレースホルダで状態遷移も要件も書かれていない。状態遷移・enum を確認したいときは supabase/schema.sql の CHECK 制約を見る（整備後は設計書を参照）。

毎回スタート状態が違うと再現できない／削除できない仕様

RLS に DELETE ポリシーが無く誰も削除できない（全削除事故防止の仕様）。だからリセットは『消す』ではなく『update で戻す』。具体：robot_status は insert 不可・id=1 の1行固定なので、その行を state=idle に update（新しく insert しようとすると失敗する）。books はテストで on_loan になった行を status=available に update。loans はテストで作った行の returned_at を update で戻す（または無視して新しいテストとして進める）。これらは SQL Editor で行う。

stop_points / users / robot_status はアプリから insert できない

schema.sql 上、stop_points・robot_status・users には insert ポリシーが無い（select / 一部 update のみ）。さらに users は auth.users への外部キー。つまり『席を入れる』『利用者を入れる』はアプリ（anon キー）からは不可で、Supabase の SQL Editor（service_role）で行う必要がある。席が入れられない＝壊れた、ではなく仕様。robot_calls と books と loans は insert 可。

ブリッジのログがどこか分からない

robot/ はまだプレースホルダ。ブリッジの起動コマンドとログの出る場所はハード制御班の手順書／該当Issueにある。本書には載せていないので、ブリッジ担当に聞いて Step3 の確認場所を事前に押さえておく。

robot_callsロボ呼出キュー。status は queued→moving→arrived→done（中止は canceled）。本フローではブリッジが status を前進させる
robot_statusロボの現在状態。id=1 の1行固定。state は idle/moving/arrived/returning。insert 不可で update で戻す
seat_idrobot_calls の席。stop_points.id への外部キー（存在しない席IDだと insert が落ちる）
Realtimeテーブルの変化を購読側に即通知する Supabase の仕組み。ブリッジが robot_calls を購読する
RLS行レベルセキュリティ。本プロジェクトは DELETE ポリシー無し＝誰も削除不可
シードテストの開始状態を作る初期データ投入。ここでは SQL Editor で行う
ブリッジrobot_calls を購読して Nav2 ゴールに変換し、robot_status を更新するロボ側の橋渡しプログラム
service_roleSupabase の SQL Editor が使う強い権限。RLS を越えて insert/update できる
