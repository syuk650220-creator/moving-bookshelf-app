▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴール: 手元(ローカル)でしか動いていないアプリを、インターネット上の公開URL（例 https://moving-bookshelf-app.vercel.app）で誰でも開ける状態にする。さらに main ブランチにマージするたびに自動で本番が更新される仕組みを作る。

なぜ必要? ゼミ生に触ってもらう受入テストも、最終発表も、URLを送るだけで全員が同じアプリを開けるからこそ成り立つ。各自がローカルで npm run dev する状態のままでは、開発者本人しか使えない。デプロイ＝書いたコードをサーバーに置いて『誰でもアクセスできる状態』にすること。

前提: 先に画面が“ローカルで動く”状態にしておく

現状リポジトリの app/page.tsx や app/search/page.tsx は『担当者が実装します』のプレースホルダで、Supabase からデータを取るコードがまだ入っていない場合がある。デプロイはあくまで『ローカルで動くものをそのまま公開する』作業。本一覧/呼び出し画面の実装Issueが終わり、ローカルで本のデータが表示できている状態になってから、もしくは並行して進める。デプロイ自体は画面が空でも成功するが、Doneの『データが表示される』はローカルで表示できていないと満たせない。

Vercelとは

Next.js を作っている会社が運営する、Next.js を一番ラクに公開できるホスティングサービス。GitHub リポジトリとつなぐと、push/マージのたびに自動でビルドして公開してくれる。無料枠で学生プロジェクトには十分。

完了の定義（Done）

  • 公開URL（https://〜.vercel.app）が発行され、ブラウザで開いてアプリが表示される
  • 公開URL上で本の一覧などSupabaseのデータが実際に表示される（環境変数が正しく効いている証拠）
  • Vercelに環境変数を2つ（NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY）登録した
  • その2つの変数が Production 環境に入っていることを確認した（Settings → Environment Variables で対象に Production が含まれている）
  • 本番(Production)に紐づくブランチが main になっていることを確認した
  • PR を立てると Preview URL が自動生成される（自動デプロイの仕組みが効いている）ことを確認した
  • 公開URLをチーム(PM/メンバー)に共有した

進め方ステップ

  1. 1. まずローカルで本番ビルドが通るか確認する
    プロジェクトのルート（package.json があるフォルダ）で npm run build を実行する。前提として、先に npm install 済み・.env.local 作成済みであること（環境構築ガイド.html 参照）。Vercel は中でこのビルドを行うので、ローカルで通らないものは本番でも必ず失敗する。成功サイン: 最後に『Compiled successfully』やビルド完了の表示が出てプロンプト（入力待ち）に戻ればOK。赤い『Error』で途中停止したら失敗なので、その内容を読んで先に直す。
  2. 2. Vercelアカウントを用意する（PMと一緒に）
    本番の所有はPM(水谷)アカウントが基本。各自が勝手に別アカウントで公開すると本番が複数できて混乱する。Vercel に GitHub アカウントでサインアップ/ログインする。この時に出る『リポジトリへのアクセス許可』画面（All repositories か Only select か）で、必ず moving-bookshelf-app へのアクセスを許可する。
  3. 3. GitHubリポジトリをImport（取り込み）する
    Vercel ダッシュボードで『Add New… → Project』→ GitHub の一覧から moving-bookshelf-app を選んで Import。一覧に出てこない場合は、リポジトリの所有者（個人かOrganizationか）とアクセス権をPMに確認する（Organization所有だと個人アカウントからは見えないことがある）。Framework は Next.js が自動認識されるので、ルートディレクトリ等はそのままでよい。
  4. 4. 環境変数を2つ登録する（最重要）
    Import 画面（または後から Settings → Environment Variables）で、Key と Value を2組登録する。Key は NEXT_PUBLIC_SUPABASE_URL と NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY（コードが読む名前と完全一致が必須）。値は各自の .env.local と同じもの＝URLは https://〜.supabase.co の形、キーは sb_publishable_ で始まる文字列。正しい2行は同フォルダ 環境構築ガイド.html のステップ4に実際の値が載っているので、それと同じものを貼ると取り違えない。環境(Environment)の選択欄では Production・Preview・Development すべてにチェックを入れておくと安全（特に Production を必ず含める）。ここを間違えると本番URLでだけ変数が効かず、データが表示されない。
  5. 5. Deploy（デプロイ）を実行する
    Deploy ボタンを押すとビルドが始まる。数分待つとビルドログが流れ、完了すると公開URLが表示される。失敗したら赤字のログを上から読む（ステップ1で通っていればたいてい原因は環境変数まわり）。
  6. 6. 公開URLを開いて動作確認する
    発行された URL をブラウザで開く。アプリが表示され、本一覧などのデータが出れば成功（=環境変数が効いている）。データが出ない・エラーになる場合は『動作確認のしかた』と『つまずきポイント』を見る。
  7. 7. 自動デプロイの仕組みを確認する
    確認目的なら、まず適当な feature ブランチで小さな変更のPRを立てると、Vercel がそのPR用の Preview URL を自動生成する。これで『push/マージで自動デプロイが効く』ことが確認できる。本番(Production)を更新する main へのマージは、勝手に行わずPM承認の上で（main直push禁止・マージはPM承認後Squash の開発フローに従う）。main にマージされたものが自動で本番URLに反映される。
  8. 8. 公開URLをチームに共有する
    発行された本番URLをPM/メンバーに共有し、受入テスト(関連Issue)に進める。

📚 進めながら調べる・学ぶこと

  • VercelとGitHubリポジトリの連携／Import の流れ
    デプロイの出発点。リポジトリをVercelに取り込んで初めて自動ビルドが始まる。連携時のアクセス許可で対象リポジトリを含め忘れると、そもそもImport一覧に出てこず最初の一歩で詰まる。
    Vercel公式 https://vercel.com/docs/frameworks/nextjs （Next.jsのImport手順）。GitHub側のアクセス許可（All / Only select repositories）の概念は https://vercel.com/docs/git のGit連携の節。一覧に出ない時はリポジトリ所有者・アクセス権をPMに確認。
  • Vercelの環境変数と『適用環境（Production/Preview/Development）』の選択
    本Issue最大の詰まり所。NEXT_PUBLIC_SUPABASE_* を登録しても、Production にチェックを入れ忘れると本番URLでだけ変数が undefined になり、Supabaseに接続できずデータが出ない。どこにチェックを入れるかを必ず確認する。
    Vercel公式 https://vercel.com/docs/environment-variables の Environments(適用環境)の節。値の正しい中身は同フォルダ 環境構築ガイド.html のステップ4。
  • Supabaseの新形式 publishable key（sb_publishable_...）の見分け方
    Supabaseダッシュボードには新形式の publishable key と旧来のキーが別々に並ぶことがあり、どれをコピーするか迷う。違うキーを貼ると本番でだけ認証エラーになる。
    正典は同フォルダ 環境構築ガイド.html のステップ4（実際の2行＝https://〜.supabase.co と sb_publishable_... がそのまま載っている）。自分で確認するなら Supabase ダッシュボード → Project Settings → API。
  • NEXT_PUBLIC_ プレフィックスの意味（なぜブラウザから使えるのか）
    Next.js では NEXT_PUBLIC_ で始まる変数だけがビルド時にブラウザ側コードへ埋め込まれる。これを理解していないと『変数を登録したのにクライアントで読めない』『逆に秘密のキーをうっかり公開してしまう』を防げない。
    node_modules/next/dist/docs/01-app/02-guides/environment-variables.md を開く（App Router=01-app配下が現行16系）。NEXT_PUBLIC_ の項と、env変更後はRedeployが必要な旨を確認。
  • プレビュー環境(Preview)と本番環境(Production)の違い
    PRごとに作られるPreview URLで安全に確認してから、main経由で本番(Production)を更新する流れを理解するため。確認はPreviewで、本番反映はmainマージで、という使い分けがわかる。
    Vercel公式 https://vercel.com/docs/deployments/environments 。補足: Vercelはデフォルトで Local / Preview / Production の3環境を持ち、環境変数は Production / Preview / Development の単位で割り当てる（環境名と環境変数の適用先の名前は少しズレるので注意）。
  • Next.jsのビルド設定（Vercelが自動でやってくれる範囲）
    Vercelは Next.js を自動認識し、ビルドコマンドや出力先を自分で設定する必要はほぼ無い。逆に手で触ると壊すことがあるので『触らなくてよい』ことを確認しておく。
    Vercel公式 https://vercel.com/docs/frameworks/nextjs 。ローカルでのビルド挙動は node_modules/next/dist/docs/01-app/02-guides/ 配下のビルド/環境変数ガイド。
  • 本番ブランチ(Production Branch)の設定
    どのブランチへのマージが本番URLを更新するかを決める設定。これが main になっていないと、mainにマージしても本番が変わらない/別ブランチで意図せず公開される。
    Vercel ダッシュボード → 対象プロジェクト → Settings → Git の Production Branch。説明は https://vercel.com/docs/git 。
  • デプロイ失敗時のログの見方
    ビルドは赤字エラーで止まることがある。ログの読み方を知らないと『なぜ失敗したか』が分からず止まってしまう。だいたいは環境変数の名前ミスか、ローカルでビルドが通っていないかのどちらか。
    Vercel ダッシュボード → 対象プロジェクト → Deployments → 失敗したデプロイを開く → Build Logs。エラー行を上から読む。ローカルで npm run build を再実行して同じエラーが出るかも切り分けに使う。

つまずきポイントと対処

症状ありがちな原因対処
公開URLは開けるがデータが出ない／真っ白環境変数を登録したが Production にチェックが入っていない、または Key 名が違うSettings → Environment Variables で2つの Key 名が NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY と完全一致か、適用先に Production が含まれるかを確認
環境変数を直したのに本番に反映されないenv変更はビルド時に埋め込まれるため、変更後に再ビルドしないと効かないDeployments タブから対象を Redeploy する（これだけで十分。再デプロイのためにmainへ直pushはしない）
Import一覧にリポジトリが出てこないリポジトリがOrganization所有、または連携時にアクセス許可していないGitHub連携のアクセス許可で moving-bookshelf-app を含める。それでも出なければ所有者・権限をPMに確認
ビルドが赤字で失敗するローカルでも npm run build が通っていない（型エラー等）先にローカルで npm run build を通す。同じエラーが出るはず。Vercelの Build Logs と突き合わせる
どのキーを貼るか分からないSupabaseに新旧のキーが並んでいて取り違える環境構築ガイド.html ステップ4の2行と同じものを貼る。キーは sb_publishable_ で始まる文字列
service_role キーは絶対にVercelの NEXT_PUBLIC_ に入れない

Supabaseには全権限を持つ service_role キーが別に存在する。これを NEXT_PUBLIC_ で登録するとブラウザに丸見えになり、誰でもDBを好き放題できてしまう。アプリが使うのは publishable key（sb_publishable_...）だけ。NEXT_PUBLIC_ に入れてよいのは publishable key と URL の2つだけ。

本番のmainマージは勝手にやらない

main にマージすると本番URLが即更新される。開発フロー（main直push禁止／feature/◯◯ → PR → PM承認後Squash）に従い、本番を動かすマージはPM承認の上で。自動デプロイの仕組み確認だけならPRのPreview URLで十分。

動作確認のしかた

『公開URLが開ける』だけでなく『環境変数が効いてデータが出る』まで見るのがこのIssueのゴール。確認は端末を分けると分かりやすい。

  1. 自分のPC以外（スマホや別の人のPC）で公開URLを開く → ローカル環境に依存せず本当に公開されているかを確認（見た目チェック）
  2. アプリが表示され、本の一覧などSupabaseのデータが出る → 環境変数が Production で効いている証拠
  3. 自分のPCのブラウザでも公開URLを開き、F12 → Console で赤いエラーが無いか確認（Supabase接続エラーが出ていないか）
  4. Vercel ダッシュボード → Settings → Environment Variables で2つの変数が Production に入っているか確認
  5. feature ブランチで小さな変更のPRを立て、VercelがそのPRの Preview URL を自動生成するか確認（自動デプロイの仕組みのチェック）。本番反映を伴う main マージはPMと行う

最終OK判定: 別端末から開いた公開URLでアプリが動き、データが表示され、Productionに2つの環境変数が入っていて、PRでPreview URLが自動生成されること（=Doneのチェックが全部埋まる）。
