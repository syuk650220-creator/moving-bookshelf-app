このIssueは何？

ゴール: まだ返していない貸出（returned_at が空）のうち、借りてから一定日数（例: 14日）を超えたものを見つけて「返却忘れかも」と画面に警告を出す。要件 C-04。

なぜ必要か: 研究室の本は紙の台帳がなく、借りた人が返し忘れても誰も気づけない。「○日経っても返却されていない」を自動であぶり出せば、声かけや督促のきっかけになる。まずは貸出履歴画面（S-5 = app/history/page.tsx）に警告バッジを出すところから始め、余力があれば全画面共通の件数表示やメール通知へ広げる、という段階設計のIssue。

学び方

答えのコードは載せません。「経過日数の計算」「未返却データの抽出」「どこで取得するか（Server/Client）」を自分で調べて手を動かすためのガイドです。AIには答えでなく“考え方”を聞きましょう。

完了の定義（Done）

  • 未返却（returned_at が NULL）の貸出だけを Supabase から取得できている
  • 各貸出の borrowed_at からの経過日数を計算し、しきい値（例 14日）を超えたものを「警告」と判定できる
  • S-5（貸出履歴 app/history/page.tsx）で、警告対象の行に「返却忘れかも」等のバッジ／色が出る
  • 本のタイトルも一緒に表示されている（loans 単体だと book_id しか無いため books をネスト取得）
  • テストデータ（20日前の未返却）で警告が出て、返却済みにすると警告が消えることを確認した
  • （余力）全画面共通の「未返却 N件」バッジ、またはメール/通知へ拡張

進め方ステップ

まず現状把握から。app/history/page.tsx を開くと、いまは見出しだけのスタブ（<h1>貸出履歴 / S-5</h1>）で、データ取得コードも "use client" も付いていません。ここに少しずつ機能を足していきます。

  1. 1. テストデータを仕込む（土台）
    Supabase の SQL Editor を開き、まず books から1件 id をコピーする（select id, title from books limit 1;）。その id を使って loans に「20日前に借りて、まだ返していない」行を1件作る。ポイントは2つ: returned_at を入れない（=NULL=未返却）、borrowed_at に過去の日時を入れる。過去日時は now() - interval '20 days' のように書く（このSQL書き方は次の code ブロック参照）。これが無いと何を作っても警告が出ず、入口で詰まるので最優先。
  2. 2. 未返却データを取る（タイトル付き）
    S-5 で supabase から loans を取得する。条件は returned_at が NULL の行だけ（.is("returned_at", null) のような絞り込みを調べる）。さらに loans は book_id で books を参照している（schema.sql の references books(id)）ので、select の文字列の中に books をネストして書くとタイトルも一緒に取れる。どう取得するかは下の research と「動作確認」を参照。
  3. 3. 経過日数としきい値を計算（置き場所も決める）
    lib/ に小さいファイルを新規作成する（例: lib/overdue.ts。supabaseClient.ts と同じ lib/ 配下）。そこに OVERDUE_DAYS = 14 のような定数と、isOverdue(borrowedAt: string): boolean のような判定関数を export し、history/page.tsx から import して使う。判定は『今 − borrowed_at が OVERDUE_DAYS 日を超えているか』。日時の比較方法は research のタイムゾーン項目を読んでから書く。
  4. 4. 画面に警告を出す（S-5）
    取得した各行を一覧表示し、isOverdue が true の行に「返却忘れかも」バッジや赤色を付ける。このプロジェクトの lib/supabaseClient.ts は NEXT_PUBLIC キーで作った『ブラウザ用クライアント』なので、まずは history/page.tsx の先頭に "use client" を付けたクライアントコンポーネントで取得するのが今ある仕組みに一番素直。useEffect で取得→useState に入れて表示、という流れを調べる。
  5. 5. （余力で）全画面共通バッジへ昇格
    app/layout.tsx に直接フェッチを足すとレイアウト全体がクライアント化したり全ページで毎回取得が走る副作用がある。なので『未返却 N件』だけを表示する小さなクライアントコンポーネント（例: components/OverdueBadge.tsx を新規作成。現状 components/ ディレクトリはまだ無いので自分で作る）を用意し、layout から呼ぶ形に分離する。
-- ① books から本を1件確認して id をコピーする
select id, title from books limit 1;

-- ② コピーした id を <BOOK_ID> に貼り、20日前の未返却 loans を1件作る
--    returned_at を入れない＝未返却。borrowed_at に過去日時を入れるのがミソ
insert into loans (book_id, borrower_type, borrowed_at)
values ('<BOOK_ID>', 'guest', now() - interval '20 days');

-- 確認: 未返却（returned_at が NULL）の行が取れるか
select id, book_id, borrowed_at, returned_at from loans where returned_at is null;

📚 進めながら調べる・学ぶこと

  • 未返却だけを抽出する（NULL 絞り込み）
    返却忘れ判定の前提は『まだ返していない＝returned_at が NULL』。等号(.eq)では NULL は拾えないので NULL 専用の書き方が要る。
    Supabase JS の select リファレンス https://supabase.com/docs/reference/javascript/select で『is』や null フィルタの項目を探す（.is("returned_at", null) のような書き方）。
  • 本のタイトルを一緒に取る（ネスト select / 外部キー）
    loans 単体だと book_id（UUID）しか無く、画面に本の名前を出せない。loans.book_id が books(id) を参照しているおかげで、1回のクエリでタイトルもネスト取得できる。
    Supabase JS の埋め込み(ネスト)select: https://supabase.com/docs/reference/javascript/select の『Query referenced tables / nested』。select の文字列の中にテーブル名(books)をネストして書く方法を読む。因果: schema.sql の book_id ... references books(id) があるから loans→books の関係が成立する。
  • データ取得をどこで行うか（Server / Client）
    Next.js 16系では Server Component と Client Component で書き方が違う。今回 lib/supabaseClient.ts は NEXT_PUBLIC キー＝ブラウザ用なので、まずはクライアント側で取得するのが無難（共有 supabase を Server Component で使うとクッキー/環境変数まわりで詰まりやすい）。
    まず node_modules/next/dist/docs/01-app/03-api-reference/01-directives/use-client.md（"use client" の付け方）。次に 01-app/01-getting-started/05-server-and-client-components.md と 06-fetching-data.md で取得場所の考え方。『Server で取得したいなら別途サーバー用 Supabase クライアントが必要（今は未用意）』という事実も押さえる。
  • 経過日数の計算とタイムゾーン
    borrowed_at は schema.sql で timestamptz（タイムゾーン付き）。Supabase JS は ISO 文字列（例 2026-06-18T...Z）で返すので、扱い方を知らないと比較を間違える。
    MDN の Date: https://developer.mozilla.org/ja/docs/Web/JavaScript/Reference/Global_Objects/Date 。new Date(文字列) は内部で UTC として正しく比較できるので、ミリ秒差→日数で計算する限り基本ズレない（『UTCだからズレる』は誤解）。表示用に日本時間へ直すかは別問題。
  • （任意）通知手段の選択肢
    C-04 の『通知』は画面バッジ／メール／Push と幅がある。まずは画面バッジが最小・確実。拡張時にどれを選ぶか判断材料が要る。
    Supabase Realtime（リアルタイム購読で件数更新）: https://supabase.com/docs/guides/realtime 。メール/定期実行に踏み込むなら Edge Functions + cron の発想（参考: Supabase Docs の Functions / Cron）。まずはバッジで Done を満たし、ここは余力で。
  • （任意）定期実行の仕組み（cron / Edge Functions）
    『毎朝チェックして通知』のような自動化に必要。ただし学生プロジェクトの初手では過剰になりがちなので最後でよい。
    Supabase の Scheduled Functions / pg_cron の解説（https://supabase.com/docs/guides/functions の周辺）。まずは画面を開いたら計算する方式で十分、と割り切る判断も学ぶ。

つまずきポイントと対処

  • 警告が一切出ない: たいてい『テストデータが無い/条件に合っていない』。ステップ1のSQLで returned_at を入れていない（=NULL）か、borrowed_at が本当に過去（20日前）かを SQL Editor で確認する。
  • NULL 絞り込みで .eq を使ってしまう: returned_at = null は効かない。NULL 専用フィルタ（is）を使う。
  • Server Component で client 用の処理を書いて詰まる: 共有 supabase はブラウザ用（NEXT_PUBLIC キー）。今回はファイル先頭に "use client" を付けてクライアント側で取得するのが素直。Server で取りたいなら別のサーバー用クライアントが必要（今は未用意）。
  • タイムゾーンで1日ズレる気がする: timestamptz は new Date() に渡せば内部で UTC として正しく比較できるので、ミリ秒差→日数で計算する限り基本ズレない。1日ズレが起きるのは『日付を 00:00 で切り捨てて数える』ときなので、まずは経過ミリ秒で判定すると安全。
  • 定数や関数の置き場所で迷う: lib/overdue.ts（supabaseClient.ts と同じ lib/）に置いて import する、と決め打ちでよい。ページの中に直書きしてもよいが、再利用しづらい。
  • layout に直接フェッチを足してレイアウトが重くなる/全クライアント化する: バッジは小さなクライアントコンポーネントに切り出して layout から呼ぶ。

動作確認のしかた

  1. npm run dev で開発サーバを起動し、ブラウザで http://localhost:3000/history を開く（package.json の scripts の dev = next dev）。
  2. ステップ1のテストデータ（20日前・未返却）の行に「返却忘れかも」バッジ／赤色が出ていることを確認する。
  3. その行を返却済みにする: Supabase の Table Editor で該当 loans 行の returned_at に now（現在時刻）を入れる、または SQL Editor で update loans set returned_at = now() where id = '<行のid>';（loans の update は upd_loans ポリシーで全員許可されているので通る）。
  4. 画面を再読み込みし、バッジが消える（returned_at が入った＝未返却ではなくなった）ことを確認する。
  5. （余力バッジを作った場合）どのページでも『未返却 N件』が出て、返却すると件数が減ることを確認する。
OK の目安

『20日前・未返却 → 警告が出る』『返却済みにする → 警告が消える』の往復が確認できれば判定ロジックは正しく動いています。
