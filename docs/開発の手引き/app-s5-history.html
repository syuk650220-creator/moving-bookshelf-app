このIssueは何？

ゴール: 全体の貸出履歴（loans テーブル）を新しい順に一覧表示するページ /history を作る。誰が・どの本を・いつ借りて／返したかがひと目で分かる状態にする。要件 F-07。

なぜ必要か: 本一覧（S-1）は「今どの本があるか」を見る画面。履歴ページは「過去に誰がいつ借りたか」という時間軸の記録を見る画面。返却忘れの確認や、よく借りられる本の把握に使う。loans には削除ポリシーが無い（誰も消せない）ので、ここは消えない記録の閲覧窓口になる。

編集する主なファイル: app/history/page.tsx（今は説明文だけの仮ページ）。ここを実装して履歴一覧にする。あわせて、本一覧か本詳細から /history へ行けるよう リンクを1本足す。リンクを足す候補ファイルは app/page.tsx（本一覧 S-1 = ルート /）か app/books/[id]/page.tsx（本詳細 S-2）。どちらか1つに <Link href="/history"> を足せばDone。

実ファイルパス注意

本一覧は app/page.tsx（ルート /）です。app/books/page.tsx は存在しません（app/books/ の下は [id]/ だけ）。存在しないファイルを探して詰まらないように。

完了の定義（Done）

  • /history を開くと貸出履歴の一覧が表示される
  • 並び順が新しい順（borrowed_at の降順）になっている
  • 各行に「本のタイトル」「借りた人」「借りた日時」「返した日時 または 未返却」が出ている
  • ゲストで借りた行は guest_name が表示される
  • （Auth実装後）ログイン利用者で借りた行は users の表示名が出る ※下記の注意参照
  • 本一覧（app/page.tsx）か本詳細（app/books/[id]/page.tsx）のどちらかから /history へ遷移できる
  • 履歴が0件のときは「履歴がありません」など空表示が出る
user表示名は条件付きDone

ログイン利用者（borrower_type='user'）の表示名は、現状のDB設定だと取得がブロックされます（後述の「つまずきポイント」参照）。当面はゲスト（guest_name）中心で動作確認し、user表示は Auth とポリシー整備の後でOKです。Doneの該当項目は「(Auth実装後)」の条件付きと考えてください。

進め方ステップ

  1. まず現状を見る
    アプリを起動して http://localhost:3000/history を開く。今は説明文だけの仮ページが出るはず。ここを書き換えていく。
  2. テストデータ（履歴）を1〜2件入れる
    Supabase の SQL Editor で loans にゲスト借りの行を入れる（下の『困ったら・参照』のシードSQL参照）。データが無いと一覧が空で確認できない。
  3. このページの取得方法を決める
    読み取り専用ページなので、まずは『クライアントコンポーネント（先頭に "use client"、useEffect/useState で取得）』が一番素直。判断に迷ったら下の研究項目を読む。
  4. loans を取得する
    supabase（lib/supabaseClient.ts の supabase）を import して loans を select。まずは並び替え無し・タイトル無しでも、行が取れて画面に出ることを先に確認する。
  5. 新しい順に並べ替える
    borrowed_at の降順で並べる。Supabase JS の .order() の正確な書き方は研究項目の order ページで確認。
  6. 本のタイトルを一緒に出す
    loans から books（title など）をネスト select で一緒に取る（研究項目参照）。books は閲覧ポリシーがあるので取得できる。
  7. 借りた人を表示する
    borrower_type が 'guest' なら guest_name を表示。'user' の表示名は現状ブロックされるので（後述）、まずは guest 行で見た目を完成させる。
  8. 日時を見やすく整形する
    borrowed_at / returned_at は ISO 文字列なので、そのまま出さず読みやすい形にする（研究項目の Date 整形参照）。returned_at が無い行は『未返却』と表示。
  9. 一覧UIを組む
    table か、行ごとのカード（map で並べる）で表示。Tailwind でざっくり整える。
  10. 空表示・読み込み中表示を足す
    クライアント取得なら、読み込み中は loading 状態、0件は data.length === 0 の分岐で出す（理由は研究項目参照）。
  11. 履歴への導線を足す
    app/page.tsx か app/books/[id]/page.tsx のどちらかに <Link href="/history">履歴を見る</Link> を1本足す。

📚 進めながら調べる・学ぶこと

このNext.jsは16系

学習データの書き方と違う可能性があります。コードを書く前に、必ず node_modules/next/dist/docs/ の該当ガイドを開いて現行の書き方を確認してください。AIには答えを書かせず、調べ方を聞くこと。

  • ページをサーバーで作るかクライアントで作るか
    取得方法でコードの形（"use client" の要否、データ取得の書き方、読み込み中表示の出し方）が変わる。最初に決めないと手が止まる。
    node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md と 03-api-reference/01-directives/use-client.md。このプロジェクトの supabase（lib/supabaseClient.ts）はブラウザ向けクライアントなので、読み取り専用ページならまず『クライアントコンポーネント（"use client" + useEffect/useState）』が一番素直で確認しやすい既定の道。サーバーで取りたくなったら同 docs を読んでから。
  • loans を select して borrowed_at 降順に並べる
    Done の核。並び順が違うと『新しい順』を満たせない。.order() の引数の置き場所を間違えやすい。
    https://supabase.com/docs/reference/javascript/select と https://supabase.com/docs/reference/javascript/order （order の専用ページ）。order("列名", { ascending: false }) の形。
  • 関連テーブルを一緒に取る方法（ネスト select）
    loans には book_id しか入っていない。本のタイトルを出すには books を一緒に取る必要がある。
    https://supabase.com/docs/reference/javascript/select の『Query foreign tables / nested』。重要: ネスト select は参照先テーブルにも SELECT ポリシー（RLS）が必要。books は sel_books があるので取れる。一方 loans→users（借りた人の表示名）も同じネスト書き方で取れるはずだが、users には現状 SELECT ポリシーが無いので user 行は取れず空になる（guest 行は guest_name を使う）。booksとusersで扱いが違う点に注意。
  • 日時（timestamptz）を見やすく整形する
    DB の値は 2026-06-18T03:21:00.000Z のような機械向け文字列。そのまま出すと読みにくい。
    JavaScript の Date と toLocaleString。MDN『Date.prototype.toLocaleString』を検索。未返却（returned_at が null）の判定とあわせて考える。
  • 未返却／返却済みの表現と borrower_type 分岐
    returned_at の有無で表示を変える。borrower_type が user か guest かでも借りた人の出し方が変わる。条件分岐の練習。
    React の条件付きレンダリング（公式 react.dev『Conditional Rendering』）。borrower_type==='guest' → guest_name、'user' → users.display_name（ただし現状ブロック、下記つまずき参照）。
  • 読み込み中表示と空表示の出し方
    取得方法によって正しい出し方が違う。loading.tsx を置いたのに読込中が出ない、という落とし穴がある。
    node_modules/next/dist/docs/01-app/01-getting-started/06-fetching-data.md と 03-api-reference/03-file-conventions/loading.md・error.md。クライアントで取得する場合の『読み込み中／0件』は基本コンポーネント内の状態分岐（loading 状態、data.length===0）で出す。loading.tsx はサーバー側取得時の仕組みで、クライアント取得では初回描画にしか効かない。

つまずきポイントと対処

user の表示名が空になる（最重要）

borrower_type='user' の行で users.display_name を出そうとしても、現状 schema.sql には users テーブルの SELECT ポリシー（sel_users）が無く、アプリは publishable(anon) キー＋RLS で接続するため取得がブロックされ空になります。原因不明のハマりになりがち。対処: (1) 当面はゲスト（guest_name）中心で確認する、(2) user 表示が必要なら PM／Supabase 管理者に users の SELECT ポリシー追加を相談する、(3) books のネストは sel_books があるので問題ない＝users とは別扱い、と切り分ける。『users 表示名が出ない時はまず RLS の SELECT ポリシー有無を確認』。

本のタイトルが出ない

loans は book_id しか持っていない。タイトルは books をネスト select で取る必要がある。出ない時は (1) ネストの書き方（select 指定文字列）と (2) FK/キー名（book_id 経由）を確認。books は SELECT ポリシーがあるので RLS でブロックはされない。

並び順が思った逆になる

.order() は order("borrowed_at", { ascending: false }) のように第1引数=列名、第2引数=オプション。{ ascending: false } の置き場所を間違えると昇順のままになる。

loading.tsx を置いたのに読込中が出ない

クライアント取得（useEffect）では loading.tsx は効きにくい。読み込み中は自分でコンポーネント内の loading 状態で出す。自分が選んだ取得方法とどちらの仕組みが噛み合うかを意識する。

テストデータが無くて何も出ない

loans が空だと当然一覧も空。先にシードSQL（下記）で1〜2件入れてから画面を作ると進めやすい。

動作確認のしかた

  • /history を開くと履歴の行が表示される → 出ない時: loans にデータが入っているか／取得エラーが console に出ていないか確認。
  • 新しく入れた行が一番上に来る（borrowed_at 降順） → 逆順なら .order() の { ascending: false } を確認。
  • 本のタイトルが各行に出ている → 空欄なら ネスト select のキー名／book_id 経由のFKを確認。
  • ゲスト借りの行で guest_name が出ている → 空なら borrower_type 分岐と guest_name の参照を確認。
  • （Auth後）user 借りの行で表示名が出る → 空なら users の SELECT ポリシー（RLS）有無をまず確認。
  • 未返却の行に『未返却』、返却済みの行に返却日時が出る → returned_at が null かどうかの分岐を確認。
  • loans を全部消した想定（空）で『履歴がありません』が出る → 0件分岐（data.length===0）を確認。
  • 本一覧 or 本詳細の『履歴を見る』リンクから /history に飛べる。

テスト用シードSQL（ゲスト借り）

Supabase の SQL Editor に貼って実行。下のように1回で実行できる形（サブクエリで book_id を引く）にしてあるので、idを手で貼り替える必要はありません。

-- 前提: books に title='テスト本A' の本が1冊ある想定（無ければ先に1冊 insert）
-- 例: insert into books (title, shelf_level) values ('テスト本A', 1);

-- ゲスト借り・未返却の履歴を1件（book_id はサブクエリで自動取得＝貼替不要）
insert into loans (book_id, borrower_type, guest_name)
select id, 'guest', '山田ゲスト' from books where title = 'テスト本A' limit 1;

-- ゲスト借り・返却済みの履歴も1件（returned_at を入れる）
insert into loans (book_id, borrower_type, guest_name, returned_at, returned_by)
select id, 'guest', '佐藤ゲスト', now(), '受付' from books where title = 'テスト本A' limit 1;
user借りのテストは難所

user 借りの履歴は users.id が auth.users(id) を参照するため、テスト行を作るには Auth 登録が絡みます。さらに users には SELECT ポリシーが無く表示名も取れません。Auth 未実装のうちは user 借りのテストは後回しでOK。まずは上のゲスト行で完成させましょう。
