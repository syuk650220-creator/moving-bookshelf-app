▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴールはフェイルセーフ（fail-safe＝何かが壊れても危なくない側に倒れる設計）。ロボが Supabase との通信を失ったとき、命令が来ない/状態が送れない状態のまま走り続けると、人や物にぶつかる危険がある。だから通信が切れたら、まずその場で安全に止まる。そして一定時間（この資料の暫定値で例：30秒）たっても復帰しなければ、ホーム位置への帰還を試みる。

このIssueの一番大事なこと（安全第一）

目的は『賢く動かす』ことではなく『危なくないこと』。迷ったら動かさない＝止める方を選ぶ。検知が遅れても、止まりさえすれば人や物にはぶつからない。逆に『止まらずに帰還を試みる』実装は最悪。必ず「止まる」を先に確実にする。

前提：仕様（D5）はまだ確定していない

古い資料に『設計判断 D5（切断時の振る舞い）を読んで方針を合わせる』とあるが、docs/基本設計書.md は現在4行のプレースホルダで D5 という記述は実在しない（姉妹Issue #8 と同じ注意）。正典は robot/README.md と supabase/schema.sql。待ち時間（例30秒）・帰還方針・ホーム位置はこの資料が暫定で置く初期値であり、正式化はPM（水谷）に確認／設計確定待ち。下の値は『仮置き、後で差し替え』前提で読むこと。

このIssueが動く“前提Issue”（依存関係）

『ゴールをキャンセルして帰還ゴールを出す』はNav2の自律走行とマップがすでに動いていて初めて意味を持つ。順番は #8 ブリッジ（robot_calls購読→ゴール変換）→ #9 座標→ゴール変換 → Nav2 自走 が立ち上がる、その後にこのIssueの『キャンセル/帰還』が効く。#9 の通り実座標は W5（実機マッピング）まで存在しない。それ以前でも『検知＋停止（速度ゼロ／ゴール発行を抑制）』だけで十分価値があるので、ここを先に確実に作る。

スコープの切り分け（重要）：Sprint3 の必達は (1)切断の検知 と (2)その場で停止 まで。(3)ホーム帰還は、ホーム座標とマップがそろってから（W5以降）の後続作業に格下げする。今は『帰還の方針メモ』と『帰還処理を書く関数の置き場（中身は後で）』を用意するだけでよい。

完了の定義（Done）

Doneは段階分け（今やる必達／環境が揃ってからの任意）

前提（Nav2自走・マップ・シミュレータ環境）が今は無いので、実走行を止める確認は今すぐには満たせない。下のように『最小Done（今できる）』と『拡張Done（後続）』に分ける。最小Doneが満たせれば、このIssueは前進として合格。

  • 【最小・必達】Supabaseへ届くか定期的に確かめる接続監視（最後に通信できた時刻からの経過で判定）が動き、切断したら検知ログが出る
  • 【最小・必達】切断を検知したら、停止指示（Nav2のゴールキャンセル、または速度ゼロの発行）をコード上で発行できることをログ/出力で確認できる
  • 【最小・必達】復帰（再び通信できる）を検知でき、検知ログが出る。復帰時の状態の戻し方を方針メモに書いてある
  • 【最小・必達】ホーム帰還処理は『関数の置き場＋方針メモ（座標未確定のため中身は後で）』として用意。ホーム座標・待ち時間（例30秒）はPM確認待ちのTODOとして明記してある
  • 【拡張・任意（環境が揃ってから）】わざとネットを切るテストで、実機（または Gazebo シミュレータ）で走り続けず止まることを確認
  • 【拡張・任意（W5以降）】復帰しないまま規定時間が過ぎたら、ホーム帰還ゴールを出して実際にホームへ戻る

進め方ステップ

  1. 前提の確認（依存Issueと未確定事項の棚卸し）
    まず #8（ブリッジ）/ #9（座標変換）が動いているか、Nav2 自走とマップがあるかを確認。無ければ『検知＋停止』だけのスコープで進める。あわせて 待ち時間・ホーム座標・robot_status の更新主体・ロボ側が使うSupabaseキー が未確定であることをPM（水谷）に確認・共有する（理由は『困ったら・参照』）。
  2. ロボ→Supabase の“生存確認（heartbeat）”を設計する
    『今ネットにつながっているか』を知る一番素直な方法は、定期的に軽いクエリを投げて成功するか見ること。例：stop_points を1行だけ select する等の軽い読み取りを一定間隔で実行し、最後に成功した時刻を覚えておく。成功するたびに時刻を更新する。heartbeat＝心拍のように一定リズムで生存確認する仕組み。
  3. rclpy のタイマーで監視ループを回す
    ROS2 ノードの中で node.create_timer(周期, コールバック) を使い、一定周期で『最後に成功した時刻からどれだけ経ったか』を判定するコールバックを回す。ここで spin（ノードを動かし続ける処理）との関係を docs で確認する（研究『rclpy タイマー』）。判定で『一定時間（仮：数秒）成功していない＝切断』とみなす。
  4. 切断を検知したら“まず止める”を実装する（最優先）
    検知したら最初にやるのは停止。Nav2 を使っている場合は、走行中のゴールをキャンセルするのが第一手段（研究『Nav2 の停止』）。補助として速度ゼロ（cmd_vel に 0 を発行）も考えられるが、Nav2 の controller と競合しうるので最終手段（つまずきポイント参照）。この段階ではまず『キャンセル指示を発行できた』ことをログで確認できれば最小Done達成。
  5. 復帰の検知と、復帰後の状態の戻し方を決める
    監視ループが再び成功し始めたら『復帰』とみなしてログを出す。復帰したら本来は robot_status を idle 相当へ戻したいが、更新主体・タイミングは設計未確定（#8参照）。確定するまではローカルログで代替し、『復帰時に state を idle へ戻す“想定”』を方針メモに書いておく（実際の更新は設計確定後）。
  6. （後続）ホーム帰還処理の“置き場と方針メモ”を用意する
    復帰しないまま規定時間（仮：30秒）が過ぎたらホームへ帰す——という処理の関数の枠だけ作り、中身は『ホーム座標が未定義（#9のW5まで実座標なし）なので後で実装』とTODOコメントを残す。帰還の出し方（Nav2 にホーム座標のゴールを出す）は、自走とマップがそろってから埋める。

📚 進めながら調べる・学ぶこと

  • 接続監視（heartbeat / タイムアウト検知）の考え方
    『切れた』をどう判断するかが本Issueの起点。常時つなぎっぱなしを監視するより、一定間隔で軽い通信を試し、最後に成功した時刻からの経過時間で『切断』を判定する方式が単純で堅い。何秒で切断とみなすか（しきい値）の置き方も学ぶ。
    supabase-py（ロボ側クライアント）: https://github.com/supabase/supabase-py で select の呼び方と例外（失敗時に何が起きるか）を確認。heartbeat用の“軽いクエリ”は例として stop_points を1行 select（schema.sql の列: id,label,x,y,theta）。一般的なタイムアウト/再接続の設計は『heartbeat timeout failsafe』で検索。
  • rclpy のタイマーで監視ループを回す（create_timer と spin の関係）
    監視は『定期的に』判定する必要がある。while True のベタなループはROSのスピンと衝突しがち。rclpy では node.create_timer(period, callback) で周期コールバックを登録し、rclpy.spin(node) の中で回すのが定石。spin が回ってはじめてタイマーやコールバックが動く関係を理解する。
    ROS2 Humble: https://docs.ros.org/en/humble/ のチュートリアル内検索で 『Using timers』『rclpy create_timer』 を探す。あわせて『Writing a simple publisher (Python)』のタイマー例、rclpy.spin の説明を確認。
  • Nav2 のゴールキャンセル/停止（止め方の第一手段）
    『止める』の正解は、走行中のNav2のゴールをキャンセルすること。手段の名前を知らないとdocsで探せない。
    Nav2: https://docs.nav2.org/ のドキュメント内検索で Simple Commander API の cancelTask、および NavigateToPose アクションの cancel を探す（ページ名: 『Simple Commander API』『Navigating』）。低レベルでは rclpy の ActionClient の cancel_goal_async を https://docs.ros.org/en/humble/ で『rclpy ActionClient cancel』で検索。cmd_vel への速度ゼロ発行は補助／最終手段（次項）。
  • 速度ゼロ発行は最終手段（Nav2 controller との競合に注意）
    『手っ取り早く止めるなら cmd_vel に 0 を出せばいい』と思いがちだが、Nav2 の controller が同時に cmd_vel を出していると取り合いになり、かえって不安定になる。優先順位を間違えないため。
    Nav2: https://docs.nav2.org/ で controller / cmd_vel の章を確認し、誰が cmd_vel を出しているかを把握。考え方として『まずゴールキャンセルを正、cmd_vel直送は controller を止めた上での最終手段』。
  • 復帰の検知と、状態の戻し方（robot_status の扱いは未確定）
    切断と同じ仕組み（監視ループの成功）で『復帰』も判定できる。ただし復帰後に robot_status をどう更新するかは設計未確定。
    schema.sql（supabase/schema.sql）で robot_status は 常に1行を update する設計（id=1固定、insert ポリシー無し＝行追加はしない）。誰が・いつ・どのキーで更新するかは未確定（#8参照、PM確認）。supabase-py の update の書き方は https://github.com/supabase/supabase-py 。確定するまではローカルログで代替する方針。
  • ホーム位置への帰還（ホーム座標がまだ無い）
    『ホームへ帰る』には帰還先の座標が要る。だが現状ホームは存在しない。
    schema.sql の stop_points（id,label,x,y,theta）に home を表す行も label 規約も無い。さらに #9 によれば実座標は W5（実機マッピング）まで存在しない。ホームをどこにするか（stop_points に home 行を1つ用意する／地図（map frame）の原点を使う）は PM／設計で決める。Nav2 で座標をゴール化する基礎は #9（座標→ゴール変換）と https://docs.nav2.org/ を参照。
  • フェイルセーフ設計の考え方（安全第一）
    本Issueの背骨。『壊れたら危ない側に倒れない』ための原則（止まる方に倒す、検知漏れより誤検知で止める方がマシ、復帰は慎重に）を言語化しておくと実装判断がぶれない。
    一般原則は『fail-safe robot design』『emergency stop ROS2』で検索。ROS2 のライフサイクル/非常停止の考え方は https://docs.ros.org/en/humble/ で『lifecycle』『safety』を参照。実装より“判断基準”を持ち帰るのが目的。
  • シミュレータ（Gazebo）／実機での確認環境
    拡張Done（実走行を止める確認）には走行環境が要る。環境が無い段階で実走テストを試すと空回りする。
    Nav2 Getting Started: https://docs.nav2.org/getting_started/index.html を確認。Gazebo Classic / Ignition(gz) で手順が異なる版差があるので、自分の環境に合う版を選ぶ（#9 同様の注意）。Gazebo が無い場合は『ログだけで検知→停止指示の発行を確認する』を先にやる。

つまずきポイントと対処

安全上の注意（実機を動かす前に）

実機テストは必ず広い場所・低速で、すぐ電源を切れる/物理的に止められる体制で行う。最初は車輪を浮かせる（地面から離す）か、人がいない安全な場所で。『止まるはず』を過信せず、手元に物理的な非常停止手段（電源/スイッチ）を用意してから通信を切るテストをする。

症状・勘違い原因・対処
『止まらずに帰還を試みる』を先に作ってしまう順序が逆。まず確実に止まるを作るのが最優先。帰還は止まれることが確認できてから（しかも座標がそろってから）。
切断中に robot_status を更新しようとして失敗する切断中はそもそもネットが無いので DB に書けない（物理的事実）。これは正常。書けたら復帰、と割り切る。RLS が理由ではない点に注意（下の行）。
『RLSのせいでロボが書けない』と誤解するschema.sql の insert ポリシー無しはアプリ(publishable/anonキー)向けの話。robot_status はそもそも insert ではなく 1行を update する設計。ロボがどのキー（service_role か publishable か）で書くかは別Issue/設計で確認（#8）。書けない理由は『ネット断』であって RLS ではない。
ホームの stop_points が見つからず詰まる現状 stop_points に home 行は無いのが正しい。実座標は W5 まで無い（#9）。帰還は後続スコープ。今は関数の枠とTODOだけでよい。
速度ゼロ(cmd_vel=0)を出しても止まらない/暴れるNav2 の controller が同時に cmd_vel を出していて競合している可能性。ゴールキャンセルが第一手段。cmd_vel 直送は controller を止めた上での最終手段。
監視ループをどこに置けばいいか分からないwhile True で自前ループを回さず、rclpy の create_timer で周期コールバックにする。rclpy.spin が回ってはじめてタイマーが動く（研究『rclpy タイマー』）。
検知が遅くて不安数秒のしきい値で十分。検知が多少遅れても、止まりさえすれば安全。むしろしきい値を短くしすぎると、一瞬の通信ゆらぎで誤検知して頻繁に止まる。少し余裕を持たせる。
Gazebo が無くて確認できない実走確認は拡張Done。まずはログだけで『検知→停止指示の発行』を確認すれば最小Done。Gazebo は Nav2 Getting Started を見て後で。版差に注意。

動作確認のしかた

前提（Nav2自走・マップ・シミュレータ）が無くても確認できる最小確認から始める。実走確認は環境が揃ってからの拡張。

  1. 【最小】ノードを起動し、監視ループ（heartbeat）が一定周期で動いていることをログで確認する（成功のたびに『最後に成功した時刻』が更新される）。
  2. 【最小】わざと通信を止める（Wi-Fiを切る／Supabaseのキーを一時的に無効にする等）と、しきい値経過後に『切断を検知』のログが出ることを確認。
  3. 【最小】切断検知の直後に『停止指示を発行（ゴールキャンセル or 速度ゼロ）』のログ/出力が出ることを確認。実際の走行が無くても“発行できた”ことが見えればOK。
  4. 【最小】通信を戻すと『復帰を検知』のログが出ることを確認。復帰時の state 戻し（idle相当）は当面ローカルログで代替（#8で更新主体が確定するまで）。
  5. 【拡張・任意】Gazebo（Nav2 Getting Started: https://docs.nav2.org/getting_started/index.html）または実機で、走行中に通信を切ると走り続けず止まることを確認（実機は広い場所・低速・非常停止手段ありで）。
  6. 【拡張・任意・W5以降】ホーム座標が決まったら、復帰しないまま規定時間（仮30秒）経過でホーム帰還ゴールが出て、実際に戻ることを確認。
