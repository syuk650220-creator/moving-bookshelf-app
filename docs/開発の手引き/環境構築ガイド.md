# 環境構築ガイド（はじめての人向け・完全版）

このページは **「移動本棚ロボ アプリ（moving-bookshelf-app）を自分のパソコンで動かせるようにする」** ための手順書です。
プログラミングやコマンド操作がはじめての人でも、上から順番にそのままマネすれば動くように書いています。

- 所要時間：だいたい **30〜60分**（ダウンロード待ちを含む）
- ゴール：ブラウザで **`http://localhost:3000`** を開くと、アプリの画面が表示される状態
- 対象 OS：**Windows / Mac 両方**（手順が違うところは分けて書いています）

> ❓ わからない単語が出てきたら、まず末尾の「[用語集](#用語集よく出る言葉)」を見てください。
> ❓ 30分やって動かなかったら、抱え込まずチームに聞きましょう（[CONTRIBUTING.md](../../CONTRIBUTING.md) の「困ったら」参照）。

---

## 0. 全体像（これから何をするのか）

このアプリは、次の4つの部品が組み合わさって動きます。

```
   あなたのPC                              インターネットの向こう側
 ┌──────────────┐                        ┌────────────────┐
 │  ソースコード   │ ── npm install ──▶    │   Supabase       │
 │ (GitHubから取得)│                       │ （データの保管庫・  │
 │      ↓        │                        │   ログイン・DB）   │
 │   npm run dev  │ ◀── データの読み書き ──▶ │                  │
 │      ↓        │                        └────────────────┘
 │ localhost:3000 │
 │ （ブラウザで見る）│
 └──────────────┘
```

やることは、ざっくり次の5ステップだけです。

1. **道具をインストールする**（Node.js / Git / エディタ）
2. **ソースコードを自分のPCに持ってくる**（GitHubから clone）
3. **部品をそろえる**（`npm install`）
4. **接続情報を書いた設定ファイルを作る**（`.env.local`）
5. **起動して確認する**（`npm run dev`）

## 2. 道具をインストールする

必要な道具は次の3つです。**すでに入っている人は飛ばしてOK**（確認方法も書いています）。

| 道具 | 役割 | 必須？ |
| --- | --- | --- |
| **Node.js** | アプリを動かすエンジン | ✅ 必須 |
| **Git** | コードのダウンロード・共有 | ✅ 必須 |
| **Visual Studio Code (VS Code)** | コードを書く・見るエディタ | ✅ 必須 |
| GitHub Desktop | Git を画面操作でかんたんにする（任意） | ⬜ あると便利 |

### 2-1. Node.js（ノードジェイエス）

アプリを動かすための土台です。**このプロジェクトは Node.js v20.9.0 以上が必要**です（v20 か v22 の「LTS」版を推奨）。

- ダウンロード：https://nodejs.org/ja → **「LTS」** と書かれた方をダウンロードしてインストール
- インストール中の選択肢は、基本そのまま「次へ」でOK

**確認方法**（インストール後）：ターミナルを開いて、次を1行ずつ実行します（ターミナルの開き方は[2-5](#2-5-ターミナルの開き方)）。

```bash
node -v
```
→ `v20.x.x` や `v22.x.x` のように **v20以上** の数字が出ればOK。

```bash
npm -v
```
→ `10.x.x` などの数字が出ればOK（npm は Node.js に同梱されています）。

> ⚠️ `v18` 以下が出た人は、古いので新しい LTS を入れ直してください。

### 2-2. Git（ギット）

ソースコードをダウンロードしたり、変更を共有したりする道具です。

- **Windows**：https://git-scm.com/download/win → ダウンロードしてインストール（選択肢は基本そのままでOK）
- **Mac**：ターミナルで `git --version` と打つと、入っていない場合は自動でインストールを促されます。または https://git-scm.com/download/mac

**確認方法**：
```bash
git --version
```
→ `git version 2.x.x` のように出ればOK。

**初回だけ**、自分の名前とメールを設定します（コミットに記録される名前です）。
```bash
git config --global user.name "あなたの名前"
git config --global user.email "あなたのメール@example.com"
```

### 2-3. Visual Studio Code（VS Code）

コードを書いたり見たりするエディタです。

- ダウンロード：https://code.visualstudio.com/

おすすめ拡張機能（VS Code 左の四角アイコン＝Extensions から検索して入れる）：

- **ESLint** … コードの書き方ミスを教えてくれる
- **Tailwind CSS IntelliSense** … デザイン用クラスの補完
- **Japanese Language Pack**（任意）… VS Code を日本語表示にする

### 2-4. GitHub Desktop（任意・初心者におすすめ）

Git のコマンドが不安な人は、**画面のボタン操作**でクローン／コミット／プッシュができる GitHub Desktop が便利です。

- ダウンロード：https://desktop.github.com/

> このガイドでは「コマンドでやる方法」を基本に書きますが、GitHub Desktop を使う場合の補足も各所に入れています。

### 2-5. ターミナルの開き方

「ターミナル」とは、文字でコマンドを打って操作する黒い（白い）画面のことです。

- **Windows**：スタートメニューで「**PowerShell**」と検索して開く。または VS Code 上部メニュー → `Terminal` → `New Terminal`
- **Mac**：「**ターミナル(Terminal)**」アプリを開く。または VS Code 上部メニュー → `Terminal` → `New Terminal`

> 💡 **一番ラクなのは VS Code 内のターミナル**です。フォルダを開いた状態で使えば、置き場所を間違えにくいのでおすすめです。

## 3. ソースコードを自分のPCに持ってくる（clone）

GitHub にあるコードを、自分のPCにコピーしてくる作業です。これを「**クローン（clone）**」と言います。

### 3-1. 置き場所を決める

まず、コードを置くフォルダに移動します。ここでは例として「デスクトップ」に置きます。

```bash
cd ~/Desktop
```
> `cd` は「フォルダを移動する」コマンド（change directory）。Windows の PowerShell でも `cd ~/Desktop` でデスクトップに移動できます。

### 3-2. クローンする

GitHub のリポジトリページ（PMから共有されるURL）を開き、緑色の **「Code」ボタン → HTTPS → URLをコピー** します。そして：

```bash
git clone <コピーしたURL>
```

例：
```bash
git clone https://github.com/＜ユーザー名＞/moving-bookshelf-app.git
```

実行すると `moving-bookshelf-app` というフォルダができます。その中に入ります。

```bash
cd moving-bookshelf-app
```

> 🖱️ **GitHub Desktop の場合**：`File → Clone repository` から対象リポジトリを選んでボタンを押すだけ。クローン後、`Repository → Open in Visual Studio Code` でエディタが開けます。

### 3-3. VS Code で開く

```bash
code .
```
（最後の `.` は「今いるフォルダ」という意味です。`code` コマンドが効かない場合は、VS Code を開いて `File → Open Folder` から `moving-bookshelf-app` フォルダを選んでください。）

## 4. 部品をそろえる（npm install）

アプリが使っている外部の部品（ライブラリ）をまとめてダウンロードします。
**`moving-bookshelf-app` フォルダの中にいること**を確認してから実行してください。

```bash
npm install
```

- 数分かかります。最後に `added XXX packages` のように出れば成功です。
- `node_modules` という大きなフォルダができますが、これは自動生成なので触らなくてOK（Gitにも含めない設定済み）。

> ⚠️ 黄色い `warn` がたくさん出ても、基本は無視してOK。赤い `error` で止まったときだけ[トラブルシューティング](#7-うまくいかないときトラブルシューティング)を見てください。

## 5. 設定ファイル `.env.local` を作る

アプリが Supabase（データの保管庫）につなぐための **接続情報** を書いたファイルを作ります。
このファイルは秘密情報を含むため **Git には含めません**（`.gitignore` 済み）。だから一人ひとり手で作る必要があります。

### 5-1. ファイルを作る

`moving-bookshelf-app` フォルダの **直下**（`package.json` と同じ場所）に、`.env.local` という名前のファイルを新規作成します。

- **VS Code でやる方法**：左のファイル一覧で `moving-bookshelf-app` を選び、新規ファイルボタン → 名前を **`.env.local`**（先頭のドットを忘れずに）にする
- **コマンドでやる方法（Mac/PowerShell共通）**：
  ```bash
  ni .env.local        # Windows PowerShell の場合
  touch .env.local     # Mac の場合
  ```

### 5-2. 中身を貼り付ける

作った `.env.local` に、下の内容を **そのままコピペ** して保存してください。

```env
NEXT_PUBLIC_SUPABASE_URL=https://zljswppciglhvwjyquow.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=sb_publishable_Ebl1Rfth5d2_Kikh6nK2wA_PEM75NnJ
```

> 📌 この2つの名前（`NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`）は、コード [lib/supabaseClient.ts](../../lib/supabaseClient.ts) が読みに行く名前と一致している必要があります。名前を変えると動かなくなるので、そのまま貼ってください。

### 5-3. 🔐 セキュリティの超重要ルール

- ここに載せている `sb_publishable_...` は **公開してよい種類のキー（publishable key）** です。ブラウザにも送られる前提なので、チーム内資料に書いても問題ありません。
- ただし、Supabase の **`service_role`（サービスロールキー）は絶対に共有・コピペ・コミットしないこと。** これが漏れると DB を誰でも全操作できてしまいます。
- `.env.local` ファイル自体を Slack 等に貼ったり、Git に commit したりしないでください（`.gitignore` 済みなので普通は commit されません）。

> 🔎 自分で値を確認したい場合：Supabase ダッシュボード → 対象プロジェクト → `Project Settings → API` に URL とキーがあります（PMから閲覧権限をもらってください）。

## 6. 起動して確認する（npm run dev）

いよいよ起動です。`moving-bookshelf-app` フォルダの中で実行します。

```bash
npm run dev
```

成功すると、こんな表示が出ます（数字や文言は多少違ってOK）：

```
  ▲ Next.js 16.2.7
  - Local:   http://localhost:3000
  ✓ Ready in X.Xs
```

ブラウザ（Chrome など）で **http://localhost:3000** を開いてください。
**「本一覧（ホーム） / S-1」** という見出しのページが表示されたら **成功です！🎉**

### 終了のしかた

ターミナルで **`Ctrl + C`**（Mac も `Ctrl + C`）を押すと、サーバーが止まります。
また開発するときは、もう一度 `npm run dev` を実行すればOKです。

> 💡 コードを保存すると、ブラウザが自動で更新されます（ホットリロード）。いちいち再起動は不要です。

## 7. （参考）データベースについて

このアプリのデータ（本・貸出履歴・ロボの状態など）は **Supabase** に入っています。

- テーブルの定義は [supabase/schema.sql](../../supabase/schema.sql) にあります。
- **通常メンバーは、自分でDBを作る必要はありません。** PMがすでに用意した共有のSupabaseにつなぎます（ステップ5の接続情報がそれです）。
- もし自分専用のSupabaseで試したい人だけ、`schema.sql` の中身を Supabase の `SQL Editor` に貼って実行するとテーブル一式が作られます。
- ⚠️ このSQLには **削除（DELETE）のポリシーがわざと入っていません**（全削除事故を防ぐため）。仕様です。

## 8. うまくいかないとき（トラブルシューティング）

> エラーが出たら、**メッセージの全文をコピー** して読む or チームに貼るのが解決の近道です。

### `node` や `npm` が「コマンドが見つかりません」と出る
- Node.js のインストールが終わっていないか、**ターミナルを開き直していない**だけのことが多いです。ターミナルを一度閉じて開き直してから、もう一度 `node -v`。

### `git clone` で `Permission denied` / `Repository not found`
- このリポジトリへの **招待が届いていない**可能性大。PMにGitHubユーザー名を伝えて招待してもらってください。
- GitHubにログインしているかも確認。初回はユーザー名・パスワード（またはトークン）を求められることがあります。

### `npm install` が赤い error で止まる
- まず `node -v` で **v20以上** か確認（古いと失敗します）。
- ネットワークが不安定なことも。もう一度 `npm install` を試す。
- それでもダメなら、`node_modules` フォルダと `package-lock.json` は消さずに、エラー全文をチームに共有。

### `npm run dev` が起動するが、画面が真っ白／エラーになる
- `.env.local` の **置き場所**（`package.json` と同じ階層か）と **ファイル名**（先頭のドット `.env.local`）を確認。
- 値の貼り付けミス（余計なスペース・改行・引用符）がないか確認。
- `.env.local` を編集したら、**いったん `Ctrl + C` で止めて `npm run dev` を再実行**してください（環境変数は再起動しないと反映されません）。

### `supabaseKey is required` のようなエラーが出る
- `.env.local` のキー名が違っている可能性。[ステップ5-2](#5-2-中身を貼り付ける)の2行をそのまま貼り直してください（キー名は `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`）。

### `Port 3000 is already in use`（3000番が使用中）
- すでに別の `npm run dev` が動いています。古いターミナルで `Ctrl + C` するか、PCを再起動。
- 急ぎなら別ポートで起動：`npm run dev -- -p 3001` → http://localhost:3001

### `.env.local` ファイルが作れない／見えない
- ファイル名が `.env.local` になっているか（`env.local` や `.env.local.txt` はNG）。
- Windows のエクスプローラーで作る場合、拡張子が隠れて `.txt` が付くことがあります。VS Code で作るのが確実です。

## 9. 環境ができたら：日々の開発の流れ

環境構築は最初の1回だけ。ここから先、毎日の作業は次のサイクルです（詳細は [CONTRIBUTING.md](../../CONTRIBUTING.md)）。

```bash
git switch main && git pull        # ① 最新の本流を取り込む
git switch -c feature/◯◯           # ② 作業用のブランチを作る
# （コードを編集）
npm run dev                        # ③ ローカルで動作確認
git add .                          # ④ 変更をステージ
git commit -m "やったことを1行で"    # ⑤ 記録
git push -u origin feature/◯◯      # ⑥ GitHubへ送る
# → GitHubでPull Request → 水谷がレビュー → マージ → 自動デプロイ
```

**ルール（最重要）**
- `main` に直接 push しない。必ずブランチを切って **Pull Request** を出す。
- 1 PR = 1 Issue を目安に小さく。
- マージは PM（水谷）の承認後。

## 用語集（よく出る言葉）

| 言葉 | かんたんな意味 |
| --- | --- |
| **リポジトリ (repository / repo)** | ソースコードの保管場所。GitHub上にある“本棚”のようなもの |
| **クローン (clone)** | GitHubのコードを自分のPCにコピーしてくること |
| **ターミナル / コマンドライン** | 文字でコマンドを打ってPCを操作する画面 |
| **Node.js** | JavaScript / Next.js のアプリを動かすための土台（エンジン） |
| **npm** | Node.js の部品（ライブラリ）を管理する道具。`npm install` で部品をそろえる |
| **依存パッケージ / ライブラリ** | アプリが使う外部の部品。`node_modules` に入る |
| **環境変数 / `.env.local`** | URLやキーなどの“接続情報・設定”を書いておくファイル |
| **localhost:3000** | 自分のPCの中だけで動いている、開発用のアプリのアドレス |
| **Supabase** | データベース・ログイン機能をまとめて提供するクラウドサービス |
| **ブランチ (branch)** | 作業を枝分かれさせる仕組み。本流(`main`)を壊さず安全に開発できる |
| **コミット (commit)** | 変更内容を記録すること（セーブのようなもの） |
| **プッシュ (push)** | 自分の記録をGitHub側に送ること |
| **Pull Request (PR)** | 「この変更を本流に入れてください」とお願いする仕組み。レビューを受ける |
| **ホットリロード** | コードを保存すると、ブラウザが自動で更新される機能 |
