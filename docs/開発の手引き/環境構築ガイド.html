プログラミングがはじめての人でも、上から順番にマネすれば動くように書いています。

⏱ 所要時間 30〜60分 🎯 ゴール：localhost:3000 が開ける 💻 Windows / Mac 両対応

❓ わからない単語が出てきたら、まず末尾の「用語集」を見てください。

❓ 30分やって動かなかったら、抱え込まずチームに聞きましょう（CONTRIBUTING.md の「困ったら」参照）。

0. 全体像（これから何をするのか）

このアプリは、次の部品が組み合わさって動きます。

   あなたのPC                          インターネットの向こう側
 ┌────────────────┐                   ┌──────────────────┐
 │  ソースコード      │ ── npm install ──▶ │   Supabase         │
 │ (GitHubから取得)  │                    │ （データの保管庫・    │
 │       ↓          │                   │   ログイン・DB）      │
 │   npm run dev    │ ◀─ データ読み書き ─▶ │                    │
 │       ↓          │                    └──────────────────┘
 │ localhost:3000   │
 │ （ブラウザで見る）   │
 └────────────────┘

やることは、ざっくり次の5ステップだけです。

  1. 道具をインストールする（Node.js / Git / エディタ）
  2. ソースコードを自分のPCに持ってくる（GitHubから clone）
  3. 部品をそろえる（npm install）
  4. 接続情報を書いた設定ファイルを作る（.env.local）
  5. 起動して確認する（npm run dev）

1. 事前に用意するもの（アカウント）

必要なもの用途取得方法
GitHub アカウントソースコードの取得・PRgithub.com で無料登録
このリポジトリへの招待コードにアクセスする権限PM（水谷）に GitHub のユーザー名を伝えて招待してもらう
Supabase の接続情報DB・ログイン機能に接続この資料のステップ5に値を載せています

💡 GitHub に登録したら、自分のユーザー名（@のあとの文字列）を PM に伝えてください。招待されないとコードを取得できません。

2. 道具をインストールする

必要な道具は次の3つです。すでに入っている人は飛ばしてOK（確認方法も書いています）。

道具役割必須？
Node.jsアプリを動かすエンジン✅ 必須
Gitコードのダウンロード・共有✅ 必須
Visual Studio Codeコードを書く・見るエディタ✅ 必須
GitHub DesktopGit を画面操作でかんたんに⬜ あると便利

2-1. Node.js（ノードジェイエス）

アプリを動かすための土台です。このプロジェクトは Node.js v20.9.0 以上が必要です（v20 か v22 の「LTS」版を推奨）。

  • ダウンロード：nodejs.org/ja → 「LTS」と書かれた方をインストール
  • インストール中の選択肢は、基本そのまま「次へ」でOK

確認方法（ターミナルの開き方は 2-5）：

node -v

→ v20.x.x や v22.x.x のように v20以上の数字が出ればOK。

npm -v

→ 10.x.x などの数字が出ればOK（npm は Node.js に同梱）。

⚠️ v18 以下が出た人は、古いので新しい LTS を入れ直してください。

2-2. Git（ギット）

  • Windows：git-scm.com/download/win からインストール（選択肢は基本そのまま）
  • Mac：ターミナルで git --version と打つと、未インストール時は自動で案内が出ます

確認方法：

git --version

→ git version 2.x.x のように出ればOK。

初回だけ、自分の名前とメールを設定します（コミットに記録される名前です）。

git config --global user.name "あなたの名前"
git config --global user.email "あなたのメール@example.com"

2-3. Visual Studio Code（VS Code）

コードを書いたり見たりするエディタです。ダウンロード：code.visualstudio.com

おすすめ拡張機能（左の四角アイコン＝Extensions から検索）：

  • ESLint … コードの書き方ミスを教えてくれる
  • Tailwind CSS IntelliSense … デザイン用クラスの補完
  • Japanese Language Pack（任意）… VS Code を日本語表示にする

2-4. GitHub Desktop（任意・初心者におすすめ）

Git のコマンドが不安な人は、画面のボタン操作でクローン／コミット／プッシュできる GitHub Desktop が便利です。ダウンロード：desktop.github.com

このガイドは「コマンドでやる方法」を基本に書きますが、GitHub Desktop を使う場合の補足も各所に入れています。

2-5. ターミナルの開き方

「ターミナル」とは、文字でコマンドを打って操作する画面のことです。

  • Windows：スタートメニューで「PowerShell」と検索して開く。または VS Code 上部 → Terminal → New Terminal
  • Mac：「ターミナル(Terminal)」アプリを開く。または VS Code 上部 → Terminal → New Terminal

💡 一番ラクなのは VS Code 内のターミナルです。フォルダを開いた状態で使えば、置き場所を間違えにくくおすすめ。

3. ソースコードを自分のPCに持ってくる（clone）

GitHub にあるコードを、自分のPCにコピーしてくる作業です。これを「クローン（clone）」と言います。

3-1. 置き場所を決める

コードを置くフォルダに移動します（例：デスクトップ）。

cd ~/Desktop

cd は「フォルダを移動する」コマンド（change directory）。Windows の PowerShell でも cd ~/Desktop でデスクトップに移動できます。

3-2. クローンする

GitHub のリポジトリページで、緑の「Code」ボタン → HTTPS → URLをコピー。そして：

git clone <コピーしたURL>

例：

git clone https://github.com/＜ユーザー名＞/moving-bookshelf-app.git

実行すると moving-bookshelf-app フォルダができます。その中に入ります。

cd moving-bookshelf-app

🖱️ GitHub Desktop の場合：File → Clone repository から選んでボタンを押すだけ。クローン後 Repository → Open in Visual Studio Code でエディタが開けます。

3-3. VS Code で開く

code .

（最後の . は「今いるフォルダ」の意味。code が効かない場合は VS Code で File → Open Folder から moving-bookshelf-app を選択。）

4. 部品をそろえる（npm install）

アプリが使う外部の部品（ライブラリ）をまとめてダウンロードします。moving-bookshelf-app フォルダの中にいることを確認してから実行してください。

npm install
  • 数分かかります。最後に added XXX packages のように出れば成功。
  • node_modules という大きなフォルダができますが、自動生成なので触らなくてOK（Gitにも含めない設定済み）。

⚠️ 黄色い warn が大量に出ても基本は無視でOK。赤い error で止まったときだけトラブルシューティングを見てください。

5. 設定ファイル .env.local を作る

アプリが Supabase（データの保管庫）につなぐための接続情報を書いたファイルを作ります。このファイルは秘密情報を含むためGit には含めません（.gitignore 済み）。だから一人ひとり手で作ります。

5-1. ファイルを作る

moving-bookshelf-app フォルダの直下（package.json と同じ場所）に、.env.local という名前のファイルを新規作成します。

  • VS Code でやる方法：ファイル一覧で moving-bookshelf-app を選び、新規ファイル → 名前を .env.local（先頭のドットを忘れずに）
  • コマンドでやる方法： 
    ni .env.local      # Windows PowerShell の場合
touch .env.local   # Mac の場合

5-2. 中身を貼り付ける

作った .env.local に、下の内容をそのままコピペして保存してください。

NEXT_PUBLIC_SUPABASE_URL=https://zljswppciglhvwjyquow.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=sb_publishable_Ebl1Rfth5d2_Kikh6nK2wA_PEM75NnJ

📌 この2つの名前（NEXT_PUBLIC_SUPABASE_URL と NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY）は、コード lib/supabaseClient.ts が読みに行く名前と一致している必要があります。名前を変えると動かなくなるので、そのまま貼ってください。

5-3. 🔐 セキュリティの超重要ルール

ここだけは必ず守る
  • 載せている sb_publishable_... は公開してよい種類のキー（publishable key）。ブラウザにも送られる前提なので、チーム内資料に書いてもOK。
  • ただし Supabase の service_role（サービスロールキー）は絶対に共有・コピペ・コミットしないこと。漏れると DB を誰でも全操作できてしまいます。
  • .env.local ファイル自体を Slack 等に貼ったり、Git に commit したりしない（.gitignore 済みなので通常 commit されません）。

🔎 自分で値を確認したい場合：Supabase ダッシュボード → 対象プロジェクト → Project Settings に URL（Project ID）とキーがあります（PMから閲覧権限をもらってください）。

6. 起動して確認する（npm run dev）

いよいよ起動です。moving-bookshelf-app フォルダの中で実行します。

npm run dev

成功すると、こんな表示が出ます（数字や文言は多少違ってOK）：

  ▲ Next.js 16.2.7
  - Local:   http://localhost:3000
  ✓ Ready in X.Xs

ブラウザ（Chrome など）で http://localhost:3000 を開いてください。
「本一覧（ホーム） / S-1」 という見出しのページが表示されたら…

成功です！🎉

終了のしかた

ターミナルで Ctrl + C（Mac も Ctrl + C）を押すとサーバーが止まります。また開発するときは、もう一度 npm run dev を実行すればOK。

💡 コードを保存すると、ブラウザが自動で更新されます（ホットリロード）。いちいち再起動は不要です。

7. （参考）データベースについて

このアプリのデータ（本・貸出履歴・ロボの状態など）は Supabase に入っています。

  • テーブルの定義は supabase/schema.sql にあります。
  • 通常メンバーは、自分でDBを作る必要はありません。 PMがすでに用意した共有のSupabaseにつなぎます（ステップ5の接続情報がそれです）。
  • 自分専用のSupabaseで試したい人だけ、schema.sql の中身を Supabase の SQL Editor に貼って実行するとテーブル一式が作られます。

⚠️ このSQLには削除（DELETE）のポリシーがわざと入っていません（全削除事故を防ぐため）。仕様です。

8. うまくいかないとき（トラブルシューティング）

エラーが出たら、メッセージの全文をコピーして読む or チームに貼るのが解決の近道です。

node や npm が「コマンドが見つかりません」と出る

  • Node.js のインストールが終わっていないか、ターミナルを開き直していないだけのことが多い。一度閉じて開き直してから node -v。

git clone で Permission denied / Repository not found

  • このリポジトリへの招待が届いていない可能性大。PMにGitHubユーザー名を伝えて招待してもらう。
  • GitHubにログインしているかも確認。初回はユーザー名・パスワード（またはトークン）を求められることがあります。

npm install が赤い error で止まる

  • まず node -v で v20以上か確認（古いと失敗）。
  • ネットワークが不安定なことも。もう一度 npm install。
  • それでもダメなら、node_modules と package-lock.json は消さずにエラー全文をチームに共有。

npm run dev は起動するが、画面が真っ白／エラーになる

  • .env.local の置き場所（package.json と同じ階層か）とファイル名（先頭のドット）を確認。
  • 値の貼り付けミス（余計なスペース・改行・引用符）がないか確認。
  • .env.local を編集したら、いったん Ctrl + C で止めて npm run dev を再実行（環境変数は再起動しないと反映されません）。

supabaseKey is required のようなエラーが出る

  • .env.local のキー名が違っている可能性。ステップ5-2の2行をそのまま貼り直してください（キー名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY）。

Port 3000 is already in use（3000番が使用中）

  • すでに別の npm run dev が動いています。古いターミナルで Ctrl + C するか、PCを再起動。
  • 急ぎなら別ポートで起動：npm run dev -- -p 3001 → http://localhost:3001

.env.local ファイルが作れない／見えない

  • ファイル名が .env.local になっているか（env.local や .env.local.txt はNG）。
  • Windows のエクスプローラーで作ると拡張子が隠れて .txt が付くことがあります。VS Code で作るのが確実。

9. 環境ができたら：日々の開発の流れ

環境構築は最初の1回だけ。ここから先、毎日の作業は次のサイクルです（詳細は CONTRIBUTING.md）。

git switch main && git pull        # ① 最新の本流を取り込む
git switch -c feature/◯◯           # ② 作業用のブランチを作る
# （コードを編集）
npm run dev                        # ③ ローカルで動作確認
git add .                          # ④ 変更をステージ
git commit -m "やったことを1行で"    # ⑤ 記録
git push -u origin feature/◯◯      # ⑥ GitHubへ送る
# → GitHubでPull Request → 水谷がレビュー → マージ → 自動デプロイ
ルール（最重要）
  • main に直接 push しない。必ずブランチを切って Pull Request を出す。
  • 1 PR = 1 Issue を目安に小さく。
  • マージは PM（水谷）の承認後。
