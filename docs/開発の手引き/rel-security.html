このIssueは何？

公開（本番デプロイ）の直前に、DB の安全装置と秘密情報の扱いを最終チェックするタスク。コードを書くより「現状を点検して、危ない穴がないか確認する」のが主目的。

なぜ必要か。このアプリは誰でもアクセスできる Web に出る。もし RLS（行レベルセキュリティ＝テーブルごとに誰が何をできるか決める仕組み）が緩いと、知らない人にデータを全削除されたり書き換えられたりする。逆にキツすぎると正規の操作が動かない。また service_role（管理者キー）が外に漏れると RLS を無視して何でもできてしまう。公開前に一度きちんと棚卸しするのがこのIssue。

前提の整理

RLS の方針はすでに supabase/schema.sql に書かれている（select/insert/update は許可・delete はポリシー無しで全員不可）。このタスクは「新しく作る」のではなく「設計どおりに効いているか・漏れがないか確認する」点検作業です。

完了の定義（Done）

  • pg_policies を見て、各テーブルのポリシーが下の「期待ポリシー早見表」と一致している（足りない・余分が無い）
  • delete 系ポリシーがどのテーブルにも1件も無いことを確認した
  • 代表として books と robot_calls の2テーブルで、publishable キー側から delete しても行が消えないことを確認した
  • アプリが実際に行う操作（books の追加/更新・robot_calls の insert・robot_status の購読）が今まで通り動く
  • service_role キーがソース・ドキュメント・コミット履歴のどこにも露出していない
  • .env.local が git に追跡されていない（git ls-files に出ない）
  • .gitignore の !.env.local.example 行を削除し、.env* が確実に全部無視される状態にした
  • （任意）scripts/check-supabase.mjs の旧キー名 NEXT_PUBLIC_SUPABASE_ANON_KEY を正キー名に直した

進め方ステップ

大事な前提

「全テーブルに select/insert/update が揃っているはず」という思い込みは間違い。この設計は意図的にテーブルごとに権限を絞っている。まず下の早見表を頭に入れてから点検すること。

テーブルselectinsertupdatedelete
books○ あるのが正○ あるのが正○ あるのが正✗ 無いのが正
loans○ あるのが正○ あるのが正○ あるのが正✗ 無いのが正
robot_calls○ あるのが正○ あるのが正○ あるのが正✗ 無いのが正
robot_status○ あるのが正✗ 無いのが正○ あるのが正✗ 無いのが正
stop_points○ あるのが正✗ 無いのが正✗ 無いのが正✗ 無いのが正
users✗ 無いのが正(現状)✗ 無いのが正(現状)✗ 無いのが正(現状)✗ 無いのが正
早見表の読み方

○=この操作のポリシーがあるべき／✗=無いのが正しい。users はポリシーが1つも無い（RLS有効だが誰も読み書きできない）のが現状の設計。stop_points と robot_status に insert ポリシーが無いのは AGENTS.md の『アプリからは robot_status/stop_points に insert 不可』と整合しています。

  1. 1. schema.sql で設計の意図を読む
    supabase/schema.sql の RLS 以降を上から読み、どのテーブルに sel_/ins_/upd_ ポリシーが書かれているかを上の早見表と突き合わせる。delete のポリシーが1つも書かれていないことも確認。
  2. 2. 本番DBの pg_policies で実状を一覧する
    Supabase の SQL Editor を開き、pg_policies から全ポリシーを一覧表示して、schema.sql どおりに本番にも反映されているか照合する（クエリは下のcodeブロック参照）。schema.sql を一度も流していなければここで差が出る。
  3. 3. delete が本当に効かないか試す（代表2テーブル）
    アプリと同じ立場＝publishable キー側から delete を投げて、行が消えないことを確認する。具体手段は『動作確認のしかた』に。代表として books と robot_calls で試せば十分（全テーブル同方針のため）。
  4. 4. 普段の操作が動くか確認する
    アプリが実際に行う操作（books 追加=insert／タグ更新=books update／ロボ呼出=robot_calls insert／robot_status は購読=select）が成功するか確認。stop_points・robot_status への insert は失敗するのが正常なので、混同しないこと。
  5. 5. service_role キーの露出を調べる
    作業ツリーとコミット履歴の両方を全文検索して、管理者キーが紛れ込んでいないか確認する（調べ方は『進めながら調べる・学ぶこと』）。
  6. 6. .env と .gitignore を点検する
    .env.local が git に追跡されていないことを確認し、.gitignore の !.env.local.example 行（廃止済みファイルの例外指定）を削除する。
  7. 7. （任意）旧キー名の残骸を直す
    scripts/check-supabase.mjs は今も旧名 NEXT_PUBLIC_SUPABASE_ANON_KEY を参照している。正キー名 NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY に直すと、キー名の最終統一が完了する。

📚 進めながら調べる・学ぶこと

  • pg_policies でポリシー一覧を出す方法
    本番DBに実際どんなRLSポリシーが効いているかは、コードを見るだけでは分からない。本番で select * from pg_policies 相当を実行し、早見表と照合するのが点検の核心。
    Supabase 管理画面の SQL Editor。RLSの考え方は https://supabase.com/docs/guides/database/postgres/row-level-security 、PostgreSQL公式の pg_policies ビューは https://www.postgresql.org/docs/current/view-pg-policies.html
  • publishable キーと service_role キーの違い
    publishable（旧anon）はブラウザに置いてよい『RLSに従うキー』、service_role はRLSを無視できる『管理者キー＝絶対に公開してはいけない』。この違いを理解しないと事故の重大度が判断できない。
    Supabase の API Keys 解説 https://supabase.com/docs/guides/api/api-keys 。本プロジェクトの正キー名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY（AGENTS.md 参照）。
  • 管理画面 SQL Editor とアプリで権限が違う理由（ロール）
    SQL Editor からは delete できるのにアプリからはできない。これはロール（role＝DB上の権限の種類）が違うため。SQL Editor は管理者ロール、アプリは anon/authenticated ロールで動く。この仕組みを知らないと『SQL Editorで消せた＝RLSが壊れている』と誤解する。
    RLSとロールの関係 https://supabase.com/docs/guides/database/postgres/row-level-security 。PostgreSQLのロール https://www.postgresql.org/docs/current/user-manag.html
  • RLS違反時に delete がどう返るか
    DELETEポリシーが無いテーブルへ delete を投げても、多くの場合エラーは出ず『0行削除（=何も消えない）』になる。『エラーが出ない＝許可されている』ではない点を知らないと検証結果を誤読する。
    Supabase JS の delete 仕様 https://supabase.com/docs/reference/javascript/delete 。返り値（data/error/count）の見方を確認する。
  • service_role キーをコミット履歴から探す方法
    今のファイルに無くても、過去に一度コミットして後で消したケースがある。その場合 GitHub 上に履歴として残り続けるので、作業ツリーだけでなく履歴も調べる必要がある。
    作業ツリーは git grep（node_modules を自動除外）、履歴は git log -p -S で文字列が出入りしたコミットを追える。Git公式 https://git-scm.com/docs/git-log と https://git-scm.com/docs/git-grep
  • .gitignore の否定パターン（!）の意味
    !.env.local.example は『この1ファイルだけは無視しない（=コミット対象に戻す）』という例外指定。例外ファイルは廃止済みなので、この行が残ると意図がぼやける。否定パターンの挙動を理解して安全に消す。
    Git公式 gitignore 仕様 https://git-scm.com/docs/gitignore （Patterns / ! prefix の節）

つまずきポイントと対処

「ポリシーが足りない＝壊れている」ではない

users にポリシーが1つも無い、stop_points/robot_status に insert ポリシーが無いのは設計どおり。pg_policies を見て早見表と一致していればOK。揃っていないこと自体は異常ではありません。

SQL Editor では delete できてしまう

Supabase の SQL Editor は管理者ロールで動くため、RLS を無視して delete できます。これは正常。『SQL Editor で消せた＝RLSが壊れている』は誤解。RLS の検証は必ず publishable キー側（アプリと同じ立場）から行うこと。

insert が通らない＝壊れた、ではない

robot_status・stop_points への insert はポリシーが無いので publishable キーからは失敗します。これが正常。むしろこれらに insert できてしまったら逆に危険なので、その場合は余分なポリシーが入っていないか疑うこと。

delete してもエラーが出ないことがある

DELETEポリシーが無いテーブルへ delete を投げると、多くの場合エラー無しで『0行削除』になります。エラーが出ないこと＝許可されている、ではありません。判定は『行が実際に消えていないか』で行うこと。

.gitignore の !.env.local.example は現に残っている

現ブランチでは .env.local.example ファイル自体は削除済みですが、.gitignore の !.env.local.example 行（35行目あたり）は今も残存しています。この最終点検でこの1行を削除し、.env* を確実に全部無視させてください。

git grep と通常 grep で node_modules の扱いが違う

git grep は追跡対象ファイルだけ見るので node_modules を自動で除外します。一方ふつうの grep / Select-String は node_modules まで舐めて誤検知だらけになりがち。キー検索は git grep を使うのが楽です。

消せないテーブルの後始末

検証用にダミー行を入れると、delete できない仕様なので消せません。books なら status を 'available' に戻す、robot_calls なら status を 'canceled' にするなど、無害な状態に戻して放置する運用にしてください。

動作確認のしかた

① ポリシー一覧の照合（pg_policies）。 Supabase の SQL Editor で次を実行し、結果を上の早見表と突き合わせる。

select tablename, cmd, policyname
from pg_policies
where schemaname = 'public'
order by tablename, cmd;
  • books/loans/robot_calls に SELECT・INSERT・UPDATE の3つがある
  • robot_status は SELECT・UPDATE のみ（INSERT が無いのが正）
  • stop_points は SELECT のみ（INSERT・UPDATE が無いのが正）
  • users は1行も出てこない（ポリシー無しが現状）
  • DELETE（cmd が d）の行がどのテーブルにも無い

② delete が効かないことの確認（代表2テーブル）。 publishable キー側から delete を投げる。一番手軽なのはブラウザの devtools コンソール。本番サイトを開き、コンソールで supabase クライアントを使って supabase.from('books').delete().eq('id', 'ダミーID').select() 相当を実行する（クライアントの呼び方は lib/supabaseClient.ts の supabase を import する設計。調べ方は Supabase JS の delete ドキュメント）。

  • 返り値の data が空配列（消えた行が0）／または error が返る → どちらでも『消えていない』ならOK
  • 実行後にそのIDの行を select し直して、まだ存在することを確認する
  • books と robot_calls の2つで試して、両方とも消えなければ全テーブルOKとみなす（全テーブル同方針のため）

③ 普段の操作が動く確認。 アプリが実際に行う操作だけを対象にする（users/stop_points への書き込みは対象外＝アプリは書かない設計）。

操作対象テーブル期待結果
本を追加books（insert）成功する
タグを更新books（update）成功する
ロボを呼ぶrobot_calls（insert）成功する
ロボ状態を見るrobot_status（select/購読）成功する（insert はしない）
停車点を追加stop_points（insert）失敗するのが正常（アプリからは入れない）

④ キー・env の確認。 下のコマンドで service_role キーと .env の追跡状況を点検する。

# 作業ツリーに管理者キーが紛れていないか（node_modules は自動除外）
git grep -i service_role
git grep -i NEXT_PUBLIC_SUPABASE_ANON_KEY

# 履歴に過去混入していないか（文字列の出入りを追う）
git log -p -S service_role

# .env.local が追跡されていないこと（何も出なければOK）
git ls-files | grep -i env

用語ミニ辞典

言葉かんたんな意味
RLS行レベルセキュリティ。テーブルごとに『誰が・何を(select/insert/update/delete)できるか』を決める安全装置。
ポリシー(policy)RLSの中の1つ1つのルール。例『books は誰でも select してよい』。
pg_policies今どんなポリシーが効いているかを一覧できる、PostgreSQL の確認用ビュー。
publishable キーブラウザに置いてよいキー(旧 anon)。RLSに必ず従う。本プロジェクトの正式名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY。
service_role キーRLSを無視して何でもできる管理者キー。絶対に公開・コミットしてはいけない。
ロール / roleDB上の権限の種類。anon=未ログイン相当、postgres=管理者。SQL Editor は管理者、アプリは anon で動く。
anon未ログインの匿名ユーザーに相当するロール。publishable キーはこのロールで動く。
RESTAPIを URL で叩く仕組み。Supabase はテーブル操作を REST でも提供している。
.gitignore の ! 否定パターン。『このファイルだけは無視しない』という例外指定。
