▶ 進める前に（全Issue共通）

作業は main から ブランチを切って 進めます。main へ直接 push は禁止。完成したら Pull Request → PM（水谷）レビュー → マージ。

git switch main && git pull        # 最新を取り込む
git switch -c feature/◯◯           # このIssue用のブランチを作る（ここから作業）
# （資料を見ながら実装）→ npm run dev で確認
git add . && git commit -m "やったことを1行で"
git push -u origin feature/◯◯      # → GitHub で Pull Request

コマンドの全手順・GitHub Desktop での方法は 環境構築ガイドの「日々の開発の流れ」へ。物理作業中心のIssueでも、robot/ のコードや記録を変更するときは同じ流れで。

このIssueは何？

ゴール: Raspberry Pi 5（以下 Pi）に OS（Ubuntu）と ROS2 Humble を入れ、自分の PC から SSH で入って ROS2 のサンプルノード（talker と listener）が通信できる状態にする。

なぜ必要か: ロボ側のソフト（Nav2 で自走、NeoPixel で光る、Supabase の robot_calls を購読して動く）は、すべてこの Pi の上で ROS2 を使って動きます。つまりこの Issue は ロボ開発すべての土台。ここが立ち上がらないと、後続のロボ系 Issue（地図づくり・自律走行・LED・アプリ⇔ロボのブリッジ）が一歩も進めません。

この先のワークフロー: 以降ロボ側のコードは、各自の PC で書いて GitHub に push → Pi 側で git pull して実行、という流れになります。だから Pi には「SSH で入って操作できる」ことが何より大事。モニタとキーボードを常時つなぐ運用はしません（ヘッドレス＝画面なし運用）。

前提のすり合わせ（PMに要確認）

stack は ROS2 Humble ですが、Humble が公式に対応する Ubuntu は 22.04（Jammy） です。Ubuntu 24.04 に正式対応するのは ROS2 Jazzy。資料・後続Issueはすべて Humble 前提で書かれているので、『24.04＋Humble』『22.04＋Humble』『24.04＋Jazzy』のどれで行くかは着手前に PM と確定してください。詳しくは下の『進めながら調べる・学ぶこと』を参照。

完了の定義（Done）

  • OS を書き込んだ microSD で Pi が起動する
  • 起動直後に uname -m が aarch64 を返す（＝64bit OS で書き込めている確認）
  • vcgencmd get_throttled が throttled=0x0（電源警告なし）になっている
  • 自分の PC から ssh ユーザー名@PiのIP で Pi にログインできる
  • ROS2 Humble が apt で入っている（ros2 --help が動く）
  • demo ノードを追加した（sudo apt install ros-humble-demo-nodes-cpp。ros-base には含まれないので別途必要）
  • ターミナル2枚で talker と listener を動かし、メッセージのやり取りが流れる
  • Pi の固定的なアクセス方法（IP もしくはホスト名）をチームで共有できている

進め方ステップ

  1. 1. 必要な物をそろえる
    Raspberry Pi 5 本体 / microSD（32GB 以上推奨・できれば高速なもの）/ microSD を書き込む PC とカードリーダー（USB のSDカードリーダー。PC内蔵スロットがあればそれでも可）/ 公式 USB-C PD 電源（純正『27W USB-C 電源（5.1V/5A）』が安全。型番例: Raspberry Pi 27W USB-C Power Supply）/ ヒートシンクまたは公式アクティブクーラー（Pi5は発熱が大きい）。初回トラブル対応用にモニタとキーボードもあると安心。
  2. 2. モニタを使う場合の注意（買う前に確認）
    Pi5 の映像出力は micro-HDMI で、普通の HDMI と形が違います。モニタにつなぐには『micro-HDMI → HDMI 変換ケーブル/アダプタ』が必要。普通の HDMI ケーブルを買って『映らない！』は定番のつまずき。ヘッドレス運用なら無くても進められますが、SSH が通らない時の最終手段として1本あると詰まりません。
  3. 3. Raspberry Pi Imager を PC に入れる
    公式サイトから Raspberry Pi Imager をダウンロードしてインストール。これが microSD に OS を書き込むアプリです。
  4. 4. Imager で Ubuntu Server (64bit) を書き込む
    デバイス＝Raspberry Pi 5、OS＝『Other general-purpose OS → Ubuntu → Ubuntu Server (64bit)』を選ぶ。32bit / Desktop 版を選ばないこと（後述）。どのバージョンを選ぶかは PM 確認事項（上のcalloutと『学ぶこと』参照）。
  5. 5. 書き込み前に『ヘッドレス設定』を仕込む
    Imager の設定（歯車 or 『次へ→設定を編集』）で、ホスト名・ユーザー名/パスワード・Wi-Fi の SSID とパスワード・SSH 有効化を事前に入れておく。これをやっておくと、画面をつながずに最初から SSH で入れます。最重要ステップ。
  6. 6. SD を Pi に挿して電源 ON
    書き込んだ microSD を Pi に挿し、ヒートシンク/クーラーを付けて、公式 PD 電源を接続。初回起動は少し時間がかかります。電源ランプの稲妻マークが点滅していないか（＝電源不足の警告）も見ておく。
  7. 7. Pi の IP を調べて SSH ログイン
    同じ Wi-Fi にいる PC のターミナルから ssh 設定したユーザー名@ホスト名.local を試す。.local が引けないのは普通に起きます（次のステップ参照）。引けない場合はルーターの管理画面で Pi の IP を探すか、一度モニタをつないで ip a / hostname -I で実 IP を確認し、ssh ユーザー名@実IP で直打ちする。
  8. 8. ログイン直後のチェック
    uname -m で aarch64（＝64bit）か確認。vcgencmd get_throttled で電源警告が出ていないか確認（0x0 ならOK）。sudo apt update && sudo apt upgrade で OS を最新化（最初は時間がかかる）。
  9. 9. ROS2 Humble を apt でインストール
    公式のインストールページ（Ubuntu deb）を開き、そこに書いてある手順を上から順にそのまま実行します。リポジトリ（ソフトの配布元）の登録方法は近年更新されているので（後述）、古いブログのコピペではなく公式ページの最新コマンドを使うこと。パッケージは ros-humble-ros-base を入れる（理由は次ステップ）。
  10. 10. ros-base を選ぶ（desktop ではない）
    今回はヘッドレス（画面なしの Server）構成なので、GUI ツール（RViz 等）を含む ros-humble-desktop は不要で重いだけ。軽い ros-humble-ros-base を選ぶ。後で Nav2 の可視化などが必要になれば、その時に追加で入れられます（名前が『desktop＝全部入りで安心』に見えても、画面の無い Pi では逆に無駄になる）。ただし次ステップの talker/listener（demo_nodes_cpp）は ros-base には含まれないので、動作確認の前に sudo apt install ros-humble-demo-nodes-cpp を別途入れておくこと（desktop を入れれば最初から含まれるが、ヘッドレスでは demo パッケージだけ足すのが軽い）。
  11. 11. 環境設定（source）を通す
    ROS2 のコマンドを使うには、毎回のターミナルで環境設定ファイルを読み込む（source する）必要があります。まずは手で source して動作確認 → 通ったら毎回自動で読まれるようにする方法（.bashrc への追記）を調べて設定する。やり方は『学ぶこと』参照。
  12. 12. talker / listener で動作確認
    前提: ros-base には demo ノードが含まれないので、ステップ10の sudo apt install ros-humble-demo-nodes-cpp を済ませておくこと（これが無いと Package 'demo_nodes_cpp' not found で詰まる）。ターミナルを2枚開き、片方で talker、もう片方で listener を起動して、メッセージが流れることを確認する。具体的なコマンドは公式チュートリアル（Understanding nodes）で確認すること。これが本Issueの本命の合格確認です。
  13. 13. アクセス方法をチームに共有
    Pi の IP（または安定して引けるホスト名）と、SSH のユーザー名をチームの連絡手段で共有。これで誰でも同じ Pi に入れるようになります。

📚 進めながら調べる・学ぶこと

答えのコマンドを丸暗記するのではなく、『なぜそれをやるのか』を理解しながら進めると、後続のロボ系 Issue で詰まったとき自分で直せるようになります。下の各テーマを『何を/なぜ/どこで』で押さえていきましょう。

  • Raspberry Pi Imager での Ubuntu 書き込み＆ヘッドレス設定（Wi-Fi/SSH の事前仕込み）
    ここを書き込み前に設定しておくと、モニタ無しで最初から SSH で入れる。後で設定し直すのは面倒なので最初の一発勝負が大事。
    Raspberry Pi Imager 配布元 https://www.raspberrypi.com/software/ と、Imager の『設定を編集（歯車）』画面。OS書き込みとヘッドレス設定の公式手順は https://www.raspberrypi.com/documentation/computers/getting-started.html
  • ROS2 Humble の apt インストール手順（リポジトリ登録の現行方式）
    本Issue最重要の手順。現在の公式手順は ros2-apt-source パッケージを dpkg で入れる新方式に変わっており、手動で GPG 鍵を curl する古いブログをコピペすると apt update でエラーになって詰まる。リポジトリ（＝ソフトの配布元）と GPG 鍵（＝配布元が本物か確認する署名）の登録は、必ず公式ページの最新コマンドを一行ずつ使うこと。
    ROS2 Humble の deb インストール公式ページ（Humble は綴りが『Debians』）: https://docs.ros.org/en/humble/Installation/Ubuntu-Install-Debians.html ／ 開けない/拒否ページが出るときは index.ros.org か GitHub https://github.com/ros2/ros2_documentation を見る。
  • ros-base と desktop の違い（どちらを入れるか）
    ヘッドレスの Pi では GUI ツール（RViz 等）入りの desktop は重く依存も増える。軽い ros-base を選ぶ判断の根拠を理解しておくと、後で必要なものだけ足せる。
    上記 Ubuntu-Install-Debians.html の『Install ROS 2 packages』節（ros-humble-desktop と ros-humble-ros-base の説明がある）。
  • ROS2 の環境設定 source と .bashrc 自動化
    source していないと ros2 コマンドが『command not found』になる初心者あるある。毎回打つのは面倒なので .bashrc に追記して自動化する流れまで理解する。
    公式チュートリアル『Configuring environment』: https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Configuring-ROS2-Environment.html
  • talker / listener（デモノード）の動かし方
    本Issueの本命の動作確認。ros2 run demo_nodes_cpp talker と listener でノード間のメッセージ通信を体験する＝ROS2 が正しく動いている証拠。ノード/トピックという ROS2 の基本概念もここで掴める。注意: demo_nodes_cpp は ros-base には含まれず desktop variant 側のパッケージ（REP 2001 の variant 定義）。ros-base で進めた場合は sudo apt install ros-humble-demo-nodes-cpp を先に入れないと Package 'demo_nodes_cpp' not found になる。
    公式チュートリアル『Understanding nodes』（talker/listener＝demo_nodes_cpp を実際に扱う唯一の正参照）: https://docs.ros.org/en/humble/Tutorials/Beginner-CLI-Tools/Understanding-ROS2-Nodes/Understanding-ROS2-Nodes.html ／ 一括起動したいだけなら talker_listener.launch.py の存在に触れる程度でOK。Turtlesim（亀の描画）は別概念なので混同しない。提供パッケージ名は apt search ros-humble-demo で調べられる。
  • ネットワーク / IP の確認と SSH 接続（.local が効かない問題）
    チームに Windows 利用者が多く、ホスト名.local（mDNS）は Windows では Bonjour 未導入だと引けず、学内 Wi-Fi で遮断されることもある。『SSH で入れない』の隠れ原因はこれが多い。実 IP を調べる手段を知っておく。
    Pi 側で ip a / hostname -I、ルーターの管理画面の端末一覧。Windows の SSH/.local については OpenSSH クライアントの導入と、必要なら Bonjour（Apple）導入を確認。SSH 一般は https://www.raspberrypi.com/documentation/computers/remote-access.html
  • Pi 5 の電源要件と under-voltage（throttled）確認
    Pi5 は電源不足だと不安定になり、原因が『電源』なのに『SSH/ROS が壊れた？』と誤診して時間を溶かす最頻出ハマり。vcgencmd get_throttled で機械的に切り分けられると速い。
    Raspberry Pi 公式 電源ドキュメント https://www.raspberrypi.com/documentation/computers/raspberry-pi.html#power-supply 。コマンドは vcgencmd get_throttled（0x0＝正常）。
  • OS バージョンと ROS2 ディストロの対応（PM相談の材料）
    Humble の Tier1 対応 OS は Ubuntu 22.04、24.04 の Tier1 は Jazzy。もし『24.04＋Jazzy』に倒すなら後続Issue全部の Humble 記述が連鎖して変わる。また 24.04+Jazzy(arm64) には apt の依存問題（libzstd 絡み、ros2/ros2 #1789）が報告されており、ハマりどころも違う。事実を押さえて PM 相談すると判断が早い。
    ROS2 ディストロ一覧 https://docs.ros.org/en/rolling/Releases.html （各ディストロの対応 Ubuntu バージョン）／ Jazzy インストール https://docs.ros.org/en/jazzy/Installation.html

つまずきポイントと対処

ハード・安全注意（電源）

必ず 公式 USB-C PD 電源（5.1V/5A・27W） を使う。非PDのスマホ充電器などだと USB 給電が 600mA 程度に制限され、Pi5 が不安定になったり起動しない・稲妻マークが出る。原因不明の不安定はまず電源を疑うこと。書き込み済み SD の抜き差しは必ず電源を切ってから。

ハード・安全注意（発熱）

Pi5 は高負荷で熱くなる。ヒートシンクか公式アクティブクーラーを必ず付ける。長時間 apt upgrade などを回すと触れないほど熱くなることがあるので、放熱を確保した状態で作業する。

症状ありがちな原因対処
SSH で入れない.local（mDNS）が引けない（Windows で多い）ルーター管理画面か、モニタをつないで ip a/hostname -I で実 IP を調べ直打ち。DHCP（IP を自動割り振りする仕組み）で再起動するとIPが変わることがある点も注意
apt update でエラー古いブログの手動 GPG 鍵登録をコピペした公式ページ（Debians.html）の最新の ros2-apt-source 方式で登録し直す
ros2: command not found環境設定を source していないターミナルで source する → 通ったら .bashrc に追記して自動化
動作が不安定・突然落ちる電源不足（under-voltage）公式 27W PD 電源に替える。vcgencmd get_throttled で確認
32bit を選んでしまったImager で 64bit Server 以外を選んだuname -m が aarch64 でなければ書き込みからやり直し（ROS2 は 64bit 前提）
Package 'demo_nodes_cpp' not foundros-base には demo ノードが含まれない（demo は desktop variant 側）sudo apt install ros-humble-demo-nodes-cpp を入れる。apt search ros-humble-demo で名前を確認できる
公式 docs が開けない/拒否ページdocs.ros.org の bot 対策（Anubis）ブラウザを変える・少し待つ・index.ros.org や GitHub(ros2_documentation) のミラーを見る

動作確認のしかた

下の順に確認すれば、土台が正しく立ち上がったと判断できます。

  1. 起動: Pi の電源を入れて、稲妻マーク（電源警告）が出ていないことを目視。
  2. アーキ確認: uname -m が aarch64（64bit）と出る。
  3. 電源確認: vcgencmd get_throttled が throttled=0x0（under-voltage 警告なし）。0x0 以外なら電源を疑う。
  4. SSH: 自分の PC から ssh ユーザー名@PiのIP でログインできる。
  5. ROS2 導入: ros2 --help がヘルプを表示する（コマンドが認識されている）。
  6. demo パッケージ: ros2 run demo_nodes_cpp talker を打って Package 'demo_nodes_cpp' not found が出る場合は sudo apt install ros-humble-demo-nodes-cpp を入れる（ros-base には含まれないため）。
  7. 本命: ターミナル2枚で talker / listener を起動し、メッセージのやり取りが流れる（具体コマンドは公式 Understanding nodes 参照）。listener 側に talker からの文字列が連続表示されれば合格。
# ログイン直後のチェック（確認系コマンドのみ）
uname -m              # aarch64 と出れば 64bit でOK
vcgencmd get_throttled # throttled=0x0 なら電源警告なし
hostname -I           # この Pi の IP を確認（チーム共有用）

# ROS2 を使う準備：まず手で source して動作確認 →
# 通ったら .bashrc に追記して次回以降自動化（やり方は research の Configuring environment 参照）
# source の対象ファイルや書式は公式チュートリアルどおりに。
ros2 --help           # コマンドが認識されているか確認

# talker/listener（demo_nodes_cpp）は ros-base に含まれないので先に追加が必要
sudo apt install ros-humble-demo-nodes-cpp

# talker / listener の実行コマンドは丸写しせず、
# 公式『Understanding nodes』を見て自分で打ってみること（ここが最大の学びどころ）

listener 側に talker のメッセージが流れ続けたら、ROS2 のノード間通信が成立＝この Issue の Done 達成です。スクショを残してチームに共有しておくと後続Issueの担当が安心します。

🤖 AIコーチの使い方

AI には『答えのコマンドを書かせる』のではなく、『どこを見ればいいか・なぜそうするか』を引き出すコーチとして使います。下のプロンプトをコピペして使ってください。

あなたは ROS2 初心者の学習コーチです。私は Raspberry Pi 5 に Ubuntu Server と ROS2 Humble を入れ、SSH で入って talker/listener を動かそうとしています。

お願い:
- いきなり正解コマンドを全部書かないでください。まず「今やろうとしている目的」と「次に確認すべきこと」を質問してください。
- 私が詰まったら、原因の切り分け方（何を確認すれば原因が分かるか）と、公式ドキュメントのどのページを見ればいいか（URL/章名）を教えてください。
- ROS2 Humble の apt 手順は、手動でGPG鍵をcurlする古い方式ではなく、現行の公式ページ(ros2-apt-source方式)に沿っているか毎回確認してください。
- talker/listener は demo_nodes_cpp のノードで、Turtlesim とは別物です。混同しないでください。
- demo_nodes_cpp は ros-base には含まれず、別途 `ros-humble-demo-nodes-cpp` の追加が必要な点を見落とさないでください。

今の状況: （ここに今やっていること・出たエラー・打ったコマンドを書く）

AI が古い手順（手動GPG鍵登録など）を出すことがあります。必ず公式ページ（Debians.html）の最新コマンドと突き合わせてから実行してください。docs.ros.org が拒否ページになる時はミラー（index.ros.org / GitHub の ros2_documentation）を案内するよう AI に頼むと良いです。
