このIssueは何？

ゴール: ゼミ発表会に向けて、(1) 短く確実に見せられるデモ、(2) 流れの分かるスライド、(3) 通しリハ を準備する。本番で何が起きても発表が成立するように、失敗時のバックアップ（録画）まで用意する。

なぜ必要? 3ヶ月の成果は、当日の数分でどれだけ伝わるかで評価が決まる。デモは生き物で、当日に限って Wi-Fi が落ちたり本棚が思った所に止まらなかったりする。だからこそ『短く確実に見せる順番』『失敗時の代替』『時間内に収める段取り』を事前に固めておくことが、発表の成否を分ける。

発表の主役は“伝わること”

技術的にすごいことより、『何の課題を、どう解決し、実際に動くところ』が審査員・先生に伝わるかが大事。デモが全部完璧に動かなくても、課題と解決の筋が通っていて、動く部分（または録画）を見せきれれば発表は成立する。

完了の定義（Done）

  • デモシナリオ（見せる順番）が1枚の紙にまとまっている
  • 通しリハで本番手順を2回連続で成功させた（『成功』の定義は下記）
  • 成功した本番フローの画面録画（バックアップ動画）を撮ってある
  • スライドができている（課題→解決→デモ→学び の流れ・持ち時間内に収まる）
  • 役割分担が決まっている（話す人 / PC操作 / ロボ操作 / 緊急停止係 / タイムキーパー）
  • 機材チェック（PC・投影端子・電源・走行スペース）が済んでいる
『デモ成功』の合格基準（これがないと“何回成功したか”が判断できない）

デモ成功＝シナリオの最後（例: ロボ到着→LED点灯→借りる登録の完了）まで、途中で止まらず到達できること。

代替の合格ライン: ロボ実機が当日不調でも、事前に撮った録画で全フロー（呼出→自走→到着→LED→借りる）を見せきれれば、発表としては成立とみなす。実機が動くに越したことはないが、録画があれば“失敗”ではない。

進め方ステップ

  1. 1. 持ち時間と発表条件を確認する（PMに聞く）
    PM(水谷)に最新の発表条件を確認し、聞いた答えをこのリストに書き込む。聞き漏らすと当日詰むので必ず全部確認する：□発表は何分か □そのうちデモは何分か □会場の Wi-Fi は使えるか（不安定なら録画前提で組む） □電源コンセントの口数（PC・ロボ充電に足りるか） □投影の接続端子は何か（HDMI / USB-C など。変換アダプタが要るか） □スクリーン解像度・縦横 □本棚を走らせる床スペースと段差・カーペットの有無。
  2. 2. デモシナリオ（見せる順番）を設計する
    『短く・確実に動く順』で並べる。例: 本一覧(S-1)を見せる→借りる/返す(S-2)→ロボ呼出(S-4)→ロボが自走→到着→LED点灯→現在地表示。1機能ずつ『今これをやると、こうなります』と一言添えられる粒度にする。動かない可能性がある機能は前半に置かない（最初でコケると立て直せない）。確実に動くものを先に、リスクの高いものを後ろに。
  3. 3. デモ用の“仕込み”をしてデモPCで通しテストする
    本番で慌てないよう、デモ前に本(books)を数冊登録しておく（仕込みは insert できるテーブル＝books/loans/robot_calls に限る。詳細は下のつまずき参照）。デモPCで実際に最後まで操作して、止まらず通るか確認する。ローカル起動には .env.local が必要なので、本番(Vercel)URLを使うか、ローカルなら事前に npm run dev が通ることを確かめる（下の研究セクション参照）。
  4. 4. バックアップ（録画）を用意する
    成功したデモを最初から最後まで画面録画しておく。撮り方の例: Windows は『Win + G』でゲームバーを開いて録画ボタン、スマホは標準の『画面収録（iOS）/ スクリーンレコード（Android）』でロボの動きを撮る（詳しい撮り方は下の研究セクション参照）。当日デモが失敗したらこの録画を流せば発表は成立する。
  5. 5. スライドを作る
    『課題 → 解決（アプリ＋ロボの仕組み）→ デモ → 学び・今後』の流れで作る。技術説明では『アプリがロボを直接動かす』と言わない（実際は疎結合。下の研究セクション参照）。ステップ4で撮った録画はスライドに挿入しておくと、デモ失敗時にスライド内から再生できて安心（挿入方法は下の研究セクション参照）。文字を詰め込みすぎず、1スライド1メッセージ。
  6. 6. 役割分担を決める
    5役を割り当てる：□話す人（スライド説明）□PC操作（アプリを触る）□ロボ操作（電源・配置）□緊急停止係（ロボ走行中にすぐ止められる人＝ハード制御担当）□タイムキーパー（残り時間を合図）。緊急停止係は安全に直結する必須役なので必ず当日アサインする（当日いない事故を防ぐ）。
  7. 7. 通しリハーサルをする
    本番と同じ手順・同じ役割で頭から通す。本番手順を2回連続で成功させるまで繰り返す（合格基準は Done 参照）。タイムキーパーが時間を計り、持ち時間をオーバーしていたら台本を削る。詰まった所・言葉に迷った所をメモして次の回で直す。
  8. 8. 機材チェックと当日の持ち物を確定する
    下のチェックリストで持ち物（PC・充電器・変換アダプタ・ロボ・モバイルバッテリー・録画ファイル）を確認。可能なら本番会場で投影端子と走行スペースを事前に下見する。

📚 進めながら調べる・学ぶこと

  • デモ環境の安定化（ネット依存を減らす・ローカル起動の前提）
    本番でWi-Fiが落ちる/遅いとアプリが止まる。安定して見せるには、事前にデモPCで動くことを確かめておく必要がある。ローカルで動かすには各自手作りの .env.local（Supabase接続情報）が必須で、これが無いと npm run dev しても画面が真っ白になり原因が分からず詰まる定番ポイント。
    本番(Vercel)URLを使うのが一番安定。ローカルで見せるなら 同フォルダ『環境構築ガイド.html』（.env.local の作り方・キー名は NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY・編集後は npm run dev の再起動が必要・真っ白になる時の対処）。デモ用PCで事前に npm run dev が通ることを必ず確認する。
  • 画面録画の撮り方（バックアップ動画）
    デモ失敗時の保険。アプリ画面はPCで、ロボの動きはスマホで撮ると確実。撮り方を知らないと当日まで保険が無いまま本番に臨むことになる。
    Web検索『Windows 画面録画 Win+G ゲームバー 使い方』『iPhone 画面収録 やり方』『Android 画面録画 やり方』。ロボの自走シーンはスマホの通常のカメラ動画でもOK。
  • 録画をスライドに『挿入する』方法
    撮った録画をスライドに貼っておけば、デモが失敗してもスライド内からすぐ再生できる。録画の“撮り方”は分かっても“スライドへの貼り方”でつまずく人が多い。
    Web検索『PowerPoint 動画 挿入 やり方』（挿入→ビデオ→このデバイス）。Googleスライドの場合『Googleスライド 動画 挿入 ドライブ』（先に Google ドライブに動画をアップしてから挿入）。各ツールのヘルプで確認する。
  • スライド構成（課題→解決→デモ→学び）の作り方
    技術自慢でなく『何の課題を、どう解決したか』が伝わる順番にする必要がある。構成がブレると審査員に成果が届かない。
    Web検索『発表 スライド 構成 課題 解決』『研究発表 スライド 作り方 初心者』。基本設計・要件一覧（同フォルダ）を見て『何を解決する課題だったか（F-◯/C-◯）』を言語化する。
  • アプリ⇔ロボの“疎結合”（技術説明を正確にするため）
    デモ説明で『呼出ボタンを押すとアプリがロボを動かす』と言うと誤り。実際はアプリが robot_calls に1行追加→ロボ側がそれを検知して動き robot_status を更新→アプリが購読して画面反映、という間接連携。スライドの技術パートを正確にするために必要。
    CLAUDE.md / AGENTS.md の『アプリ⇔ロボの疎結合』の項。Supabase Realtime の考え方は https://supabase.com/docs/guides/realtime 。
  • リハーサルでの時間管理
    持ち時間オーバーは減点・打ち切りの原因。事前に計って台本を削る練習が要る。本番でアドリブを足すと必ず伸びる。
    Web検索『プレゼン 時間管理 リハーサル コツ』。タイムキーパー役が残り時間を合図する方法（指で残り分数を出す等）を決めておく。
  • デモシナリオの設計（短く確実に見せる順番）
    限られた時間で全部は見せられない。確実に動くものを前に、リスクの高いもの（ロボ実機）を後ろに置くと、序盤でコケて立て直せない事故を防げる。
    Web検索『デモ シナリオ 作り方 順番』。本Issueの『動作確認のしかた』の表をシナリオ表の雛形に使ってよい。

つまずきポイントと対処

ロボ走行中の安全（緊急停止係は必須）

本棚ロボは重く、人や物にぶつかると危険。デモ中は『緊急停止係（ハード制御担当）』が常にロボを見て、いつでも止められる位置に立つ。観客とロボの間に十分な距離を取り、走行ライン上に人を立たせない。床の段差・カーペット・コードに乗り上げないか事前に確認する。

“仕込み”できるテーブルは insert 可能なものだけ

デモ前のデータ仕込みは books / loans / robot_calls に限る（RLSで insert 全員OK）。robot_status と stop_points はアプリから insert できない（ポリシー無し）。特に席データ(stop_points)はロボ呼出デモに必須なので、Supabase の SQL Editor 側で事前に投入してもらう（アプリからは入れられない）。空のままだと呼出が外部キー違反で必ず失敗する。

ローカル(npm run dev)は .env.local が無いと真っ白

デモをローカルで見せる場合、.env.local（各自手作り）が無い／キー名が違うと画面が真っ白になる。編集したら必ず npm run dev を再起動（環境変数は再起動しないと反映されない）。設定とよくあるエラーは『環境構築ガイド.html』参照。不安なら本番(Vercel)URLで見せるのが安全。

Wi-Fiは当日落ちる前提で組む

会場のWi-Fiは混んで遅い・落ちることがある。アプリはSupabase通信が必要なので、ネットが死ぬと操作が固まる。録画バックアップを必ず用意し、可能ならテザリング等の予備回線も考えておく。

本番で初めて触らない

当日ぶっつけは事故のもと。通しリハを2回連続成功させてから本番に臨む。話す人も声に出して練習する（黙読と本番は別物）。

動作確認のしかた

デモが『最後まで止まらず通るか』を、下の各ステップで確認する。アプリの状態は Supabase の Table Editor で行が増えたか目視できる。ロボの動き（robot_status の変化・LED）はロボ実機/シミュレータが稼働している時だけ確認できる。

デモの流れOKの目安
本一覧(S-1)を見せる本の一覧が表示され、画面が崩れずエラーも出ない
借りる/返す(S-2)借りると on_loan 表示・loans に行が増える / 返すと available に戻る
ロボ呼出(S-4)席(stop_point)を選んで呼ぶと robot_calls に status=queued の行が1件増える
ロボが自走→到着（実機/シミュレータ稼働時）robot_status.state が moving→arrived に変わる
到着LED点灯（実機稼働時）NeoPixel が点灯する
現在地表示アプリ画面にロボの現在状態（移動中/到着など）が反映される

最終OK判定: 上のフローを最後（借りる登録完了 or 到着LED点灯）まで止まらず到達でき、それを通しリハで2回連続成功できたら合格。実機が不調でも、録画で全フローを見せきれれば発表としては成立（Done の合格基準どおり）。

当日の持ち物チェック:

  • デモPC（充電満タン）と充電器
  • 投影用の変換アダプタ（HDMI / USB-C など会場の端子に合うもの）
  • ロボ本体（充電済み）とモバイルバッテリー/予備電源
  • バックアップ録画ファイル（PC内＋スライドにも挿入済み）
  • スライドデータ（PC内＋USB or クラウドにも控え）

🤖 AIコーチの使い方

AIには答え（完成スライドや完成台本）を書かせず、考え方とチェックの相棒として使う。プロンプト内の『動く機能は: ◯◯』を埋める前に、GitHub Projects の Done 列、または『全Issue一覧_完成までのロードマップ』で実際に動く機能を確認してから書く（自分たちの実装が今どの状態か思い込みで判断しない）。下をコピペして使う。

あなたはコーチです。答えそのもの（完成スライドや完成台本）を書かず、私が自分で考えられるよう質問と観点で導いてください。

状況: 学生プロジェクト『移動本棚ロボ アプリ』のゼミ発表会を準備中。持ち時間は◯分、デモは◯分。今あるアプリはNext.js+Supabase、ロボはRaspberry Pi+ROS2で、アプリとロボはSupabaseのテーブル(robot_calls/robot_status)経由の疎結合です。実際に動く機能は: ◯◯, ◯◯。会場のWi-Fiは不安定かもしれません。

困ったら・参照

  • デモで使う機能（MUST・実際にデモに乗る）: 本一覧 S-1 / 本登録 S-3 / 借りる・返す S-2 → これらは #1〜#9（本一覧/登録/貸出など）の実在Issue。ロボ呼出 S-4 / ロボ現在地表示 → こちらは番号未採番の新規Issue（#1〜#9 には含まれない）。『検索』はロードマップ上 COULD（余力があれば）なのでデモ前提に入れない
  • 今“動く機能”の確認: GitHub Projects の Done 列、または同フォルダ『全Issue一覧_完成までのロードマップ』。デモシナリオはここで動く機能だけで組む
  • ローカルでデモする時の設定: 同フォルダ『環境構築ガイド.html』（.env.local の作り方・キー名 NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY・真っ白対処・再起動が要る件）
  • 技術説明の正確さ: CLAUDE.md / AGENTS.md の『アプリ⇔ロボの疎結合』（呼出→robot_calls insert→ロボ側が検知→robot_status 更新→アプリが購読、の流れ）
  • 席データ(stop_points)の投入: Supabase の SQL Editor 側で事前投入が必要（アプリからは insert 不可）。PM/ハード担当に確認
  • 関連Issue（タスク名）: 『受入テスト（ゼミ生が実際に使う）』『結合テスト/通し確認』『公開・本番デプロイ（Vercel）』。発表準備はこれらの仕上げの後に位置する
  • デプロイURL: 本番(Vercel)URLが公開・稼働しているかは PM / Vercel ダッシュボードで確認してからデモに使う
