🌳 Git早見表（ブランチ → PR）

移動本棚ロボ アプリ｜チーム開発の進め方チートシート

最終更新: 2026年6月18日 ｜ ルール出典: CONTRIBUTING.md

この資料の使い方： 上半分は「初めてプロジェクトに参加する人」向けのセットアップ。下半分は「毎回の作業サイクル（ブランチ→PR）」です。困ったらここを見返してください。
大原則： main ブランチには直接 push しない！ 必ず「ブランチを作る → PR を出す → 承認後にマージ」の流れで進めます。（main は保護されています）

📖 まず言葉の意味（初心者向け）

リポジトリ (repository)
プロジェクト一式が入った「箱」。GitHub 上にチーム共有の本体があります。
クローン (clone)
GitHub の箱を、自分のパソコンに丸ごとコピーしてくること。最初の1回だけ。
ブランチ (branch / 枝)
main のコピーを作って、その上で安全に作業する「作業場」。失敗しても本流は無事。
コミット (commit)
変更内容に名前を付けて履歴に刻むこと。「セーブポイント」のイメージ。
プッシュ (push)
手元の変更を GitHub にアップロードすること。
プル (pull)
GitHub の最新を手元にダウンロードすること。
PR (プルリクエスト)
「この作業場のもの、本流に入れていい？」とお願いする仕組み。レビューしてから合流させる。
マージ (merge)
PR を承認して、本流（main）に取り込むこと。

🚀 はじめて参加する人のセットアップ 最初の1回だけ

自分以外のメンバーがこのプロジェクトを始めるときの手順です。上から順にやればOK。

1

事前準備（道具をそろえる）

下の3つをパソコンに入れておく（入っていなければ各公式サイトから）。

  • Git（バージョン管理ツール）
  • Node.js（アプリを動かす土台。LTS版でOK）
  • VS Code など好きなエディタ
2

リポジトリをクローン（コピーしてくる）

git clone https://github.com/syuk650220-creator/moving-bookshelf-app.git
cd moving-bookshelf-app

GitHub の本体を自分のパソコンに持ってきて、そのフォルダに移動します。

3

必要な部品をインストール

npm install

アプリの動作に必要なライブラリ（部品）を一括ダウンロード。少し時間がかかります。

4

接続設定ファイル .env.local を用意

# テンプレをコピーして…
cp .env.local.example .env.local

コピーしたら中身を編集して、Supabase の接続情報を書き込みます↓

NEXT_PUBLIC_SUPABASE_URL=https://zljswppciglhvwjyquow.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=（publishable key を貼る）
鍵の入手： publishable key（sb_publishable_…）は公開しても安全な鍵なので、PM（水谷さん）から共有してもらうか、Supabase ダッシュボードの Project Settings → API Keys から取得します。
絶対NG： secret key（sb_secret_…）は使わない・共有しない・GitHub に上げない。.env.local は最初から GitHub に上げない設定（.gitignore）になっています。
5

接続できているか確認

node scripts/check-supabase.mjs

6つのテーブルすべてに ✓ が出れば成功。アプリも起動してみましょう↓

npm run dev   # → ブラウザで http://localhost:3000 を開く（止めるのは Ctrl+C）

🔁 毎回の作業サイクル（ブランチ → PR） 作業のたびに

1

最新の本流に追いつく

git switch main
git pull

main に移動して、GitHub 上の最新を取り込む。作業前は毎回これ。

2

作業用のブランチを作る

git switch -c feature/book-list

feature/book-list という枝を作って移動。-c は「作る(create)」。
名前のルール: 機能なら feature/◯◯、不具合修正なら fix/◯◯。

3

編集して、動作を確認

npm run dev   # http://localhost:3000 で確認

コードを直したら、ブラウザで思った通り動くか見ます。

4

変更を記録（コミット）

git add .
git commit -m "本一覧の枠を追加"

add で記録対象を選び、commit でメッセージ付きで履歴に刻む。メッセージは日本語1行でOK。

5

GitHub へ送る（プッシュ）

git push -u origin feature/book-list

枝を GitHub にアップロード。-u はその枝の初回だけ（2回目以降は git push だけ）。

6

GitHub で PR を作る（ここはブラウザ操作）

  1. リポジトリ を開く
  2. 黄色い帯の 「Compare & pull request」 を押す
  3. タイトルと説明を書いて 「Create pull request」
  4. 水谷さん（PM）にレビュー依頼 → 承認されたら 「Squash and merge」
  5. main に入ると Vercel が自動でデプロイ（公開）

⚡ サイクルまとめ（これだけ覚える）

git switch main && git pull        # 最新に追いつく
git switch -c feature/◯◯           # 枝を作る
# （編集）→ npm run dev で確認
git add . && git commit -m "..."    # 記録
git push -u origin feature/◯◯      # 送る
# → GitHub で PR → レビュー → Squash and merge

📏 チームのルール（CONTRIBUTING.md より）

ルール内容
main直push禁止必ずブランチを切って PR を出す
ブランチ名feature/画面名 / fix/不具合名
コミット日本語OK・1行で要点（例「本一覧の枠を追加」）
PRの大きさ小さく。1 PR = 1 Issue が目安
承認者水谷（PM）がレビュー → 承認後マージ
マージ方法Squash and merge が基本

🆘 困ったときは

  • 30分悩んだら抱え込まず共有する
  • エラーは全文をコピーして貼る
  • コンフリクト（変更の衝突）は自己判断で消さず、相手と相談
